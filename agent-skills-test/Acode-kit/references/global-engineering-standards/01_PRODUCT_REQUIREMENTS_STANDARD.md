# 文件名
`01_PRODUCT_REQUIREMENTS_STANDARD.md`

# 文件定位
产品需求文档与需求治理标准。

# 适用范围
适用于所有新项目、版本迭代、功能增强、重构项目的需求定义、范围控制与验收约束。

# 与其他文件的关系
本文件先于 `02_UI_UX_DESIGN_SPEC.md`、`05_API_DESIGN_SPEC.md`、`06_DATABASE_DESIGN_SPEC.md` 使用，是设计与研发的输入文档标准。

# 编写目的
将模糊业务想法沉淀为结构化、可评审、可验收的需求文档，作为 UI、前后端、数据库与测试的统一输入。

## 1. 目标
1. 将模糊想法转为结构化需求。
2. 统一术语、边界、优先级、验收标准。
3. 减少需求歧义、反复返工、范围失控。
4. 为 AI 与工程师提供可执行输入。

## 1.1 PRD 的治理定位
1. 一旦项目建立 `PRD.md`，该文档即为项目最高级的需求界定文档。
2. 系统规划、业务逻辑、边界范围、版本优先级等高层需求事实，必须优先写回 PRD，而不是散落在会话记录、页面稿、接口稿或临时说明中。
3. 后续页面、接口、数据库、测试、路线图等文档，均不得脱离 PRD 自行扩展核心需求边界。
4. 若用户提出的新需求与现有 PRD 冲突，AI 必须先明确指出冲突点和影响范围，再由人工决定：
   - 替换覆盖原需求
   - 保留原需求，不新增
   - 暂缓评估
5. 未经人工明确决策，不得默认覆盖 PRD 已确认内容。

## 2. 标准需求文档结构
每份 PRD 至少包含以下章节：
1. 文档信息：版本、作者、评审人、状态、更新时间。
2. 项目背景：业务问题、当前痛点、触发原因。
3. 业务目标：量化目标、阶段目标、成功标准。
4. 用户角色：系统管理员、业务管理员、审核员、普通操作员、访客等。
5. 用户场景：谁在什么条件下做什么、为何做、失败时怎么办。
6. 核心流程：主流程、分支流程、异常流程。
7. 功能清单：模块、页面、能力、权限、边界。
8. 非功能需求：性能、安全、审计、可用性、兼容性、可维护性。
9. 约束条件：技术约束、时间约束、资源约束、外部依赖。
10. 里程碑：需求冻结、设计冻结、开发完成、联调、上线。
11. 风险与假设：未确认点、依赖方风险、数据风险。
12. 验收标准：功能验收、数据验收、性能验收、权限验收。

## 4. 需求分级规则
采用 Must / Should / Could 三层级：
1. Must：首期不做则目标无法成立，或影响核心流程闭环、合规、安全、上线门槛。
2. Should：首期建议做，能显著提升效率、体验、稳定性，但不阻断上线。
3. Could：有价值但可延后，通常用于增强体验、增强分析、增强自动化。

禁止使用“重要但不紧急”“最好有”“以后再说”等模糊表述，必须明确级别与原因。

## 4.1 MVP 切分规则
1. MVP 只保留能闭环验证业务目标的最小功能组合。
2. 一切不能直接支撑首期目标的增强项默认进入 `Should` 或 `Could`。
3. 首期必须优先保证登录、权限、核心业务链路、关键异常处理、基本审计、上线可用性。
4. 若一个功能无法在当前版本内形成完整闭环，应整体后移，而不是半做半留。

## 4.2 需求编号规则
建议统一采用：
```text
REQ-模块缩写-三位序号
PAGE-模块缩写-三位序号
API-模块缩写-三位序号
TC-模块缩写-三位序号
```

示例：
```text
REQ-USER-001 用户列表查询
REQ-USER-002 新增用户
REQ-ORDER-003 审核订单
```

## 5. 需求范围控制规则
1. 一个版本只服务一个明确业务目标。
2. PRD 中每个功能点都必须能映射到业务目标。
3. 无法说明业务价值的功能不得进入 Must。
4. 需求变更必须记录来源、原因、影响范围与版本归属。
5. 未进入本期范围的想法放入 Backlog，不得混入当前开发指令。
6. 随项目推进出现“有必要的补充、明确化、规划丰富”时，应更新 PRD 对应章节，而不是仅在会话中临时说明。
7. 如果新增想法尚未进入 PRD，不得直接进入设计、数据库、接口或代码实现。

## 5.1 本期不做清单
每份 PRD 必须显式列出：
1. 本期明确不做的功能。
2. 延后原因。
3. 预估放入的后续版本。
4. 是否影响当前架构预留。

模板：
```md
## 本期不做
- 用户批量导出：延期到 V1.1，原因是首期优先完成导入与权限。
- 审批撤回：延期到 V1.2，原因是业务规则尚未确认。
```

## 6. 术语统一规范
1. 一个业务对象只允许一个主名称。
2. 页面名称、菜单名称、接口名称、表名命名应尽量同源。
3. 状态词必须定义枚举含义，如“草稿、待审核、已驳回、已发布”。
4. 角色、权限、组织、租户、用户、账号等高频词必须单独列术语表。

## 7. AI 需求整理方法
### 7.1 输入来源
1. 口语化想法。
2. 聊天记录。
3. 会议纪要。
4. 旧系统截图。
5. 竞品页面。

### 7.2 AI 输出步骤
1. 抽取业务目标。
2. 抽取角色与场景。
3. 梳理主流程与异常流程。
4. 拆成功能模块与页面。
5. 补齐非功能需求与约束。
6. 标记待确认项，禁止 AI 自造关键业务规则。

### 7.3 AI 必须显式标记的待确认项
1. 审批流规则。
2. 金额计算规则。
3. 数据保留期限。
4. 外部系统交互方式。
5. 导入导出字段。
6. 权限颗粒度。

## 7.4 待确认假设清单
AI 或人工在 PRD 草案中必须单独列出“待确认假设”：
1. 当前默认认为成立、但尚未被业务方正式确认的规则。
2. 会直接影响页面、接口、表结构或上线范围的假设。
3. 若假设失效，需明确会影响哪些模块。

模板：
```md
## 待确认假设
- 假设 1：订单审核只有一级审核。
  影响范围：订单详情页、审核接口、审核日志表。
- 假设 2：用户状态只包含 ENABLED / DISABLED。
  影响范围：字典、查询条件、批量启停接口。
```

## 8. 防歧义规则
1. 每个功能项必须写“谁、何时、做什么、成功是什么、失败怎么办”。
2. 禁止使用“支持管理”“支持维护”“支持查看全部”等空泛描述，必须列出具体操作。
3. 对“列表、详情、新增、编辑、审核、导入、导出、删除、启停”类动作必须逐项说明。
4. 对“可配置”类需求必须明确配置项、作用范围、生效方式与默认值。

## 9. 非功能需求最小清单
1. 权限：角色权限、菜单权限、按钮权限、数据权限。
2. 性能：列表查询响应、导入导出耗时、峰值并发。
3. 安全：登录、鉴权、敏感信息、审计、接口幂等。
4. 运维：日志、健康检查、配置隔离、回滚。
5. 测试：核心流程、边界输入、异常处理、回归影响。

## 10. 验收标准写法
1. 用可验证语言，不用“基本可用”“体验良好”。
2. 明确输入、操作、期望结果、数据变化、权限限制。
3. 验收项必须覆盖正常、异常、权限、边界四类场景。

示例：
```md
- 当审核员拥有 `order:approve` 权限时，可在订单详情页执行“通过”操作。
- 执行成功后订单状态由 `PENDING` 变为 `APPROVED`，记录审核人、审核时间、审核意见。
- 无权限用户不得看到该按钮，直接调用接口返回 403。
```

## 11. 里程碑模板
```md
| 里程碑 | 输出物 | 负责人 | 完成标准 | 日期 |
| --- | --- | --- | --- | --- |
| 需求冻结 | PRD V1.0 | 产品 | Must 范围确认，术语统一 | YYYY-MM-DD |
| 设计冻结 | Pencil 设计稿 | UI | 全页面与状态齐全 | YYYY-MM-DD |
| 接口冻结 | OpenAPI 文档 | 后端 | 核心接口完成评审 | YYYY-MM-DD |
| 开发完成 | 提测包 | 研发 | 自测通过，文档齐全 | YYYY-MM-DD |
| 发布上线 | 发布记录 | 运维/研发 | 回滚预案就绪 | YYYY-MM-DD |
```

## 12. 需求评审检查清单
1. 背景与目标是否明确。
2. 角色与场景是否完整。
3. 核心流程是否闭环。
4. 页面与接口是否可映射。
5. 非功能需求是否遗漏。
6. Must/Should/Could 是否清晰。
7. 验收标准是否可测试。
8. 待确认项是否单独列出。

## 本文件使用建议
任何项目在产出 UI、API、数据库方案前，必须先按本模板形成结构化 PRD。

## AI 调用建议
AI 在接收到模糊业务诉求时，应先按本文件产出 PRD 草案，再推进设计、接口与代码生成。

## 后续可扩展点
可补充行业特定需求模板，如审批流产品、报表平台、主数据平台、运营平台模板。
