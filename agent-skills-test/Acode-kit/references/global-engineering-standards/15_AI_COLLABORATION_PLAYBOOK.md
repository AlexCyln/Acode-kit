# 文件名
`15_AI_COLLABORATION_PLAYBOOK.md`

# 文件定位
AI 参与需求、设计、开发、测试、部署全过程的操作手册。

# 适用范围
适用于 Codex、VSCode 内置 AI、IDE Agent、脚本式 AI 助手等在本母规范体系下的长期协作。

# 与其他文件的关系
本文件以 `00_GLOBAL_ENGINEERING_PRINCIPLES.md` 为总纲，串联全套文档，为 AI 使用顺序与决策顺序提供执行说明。

# 编写目的
让 AI 在需求、设计、开发、测试、部署全过程中按统一顺序读取规范、生成产物并自检，避免上下文漂移。

## 1. AI 的工作定位
1. AI 是工程协作者，不是技术栈决策替代者。
2. AI 负责加速结构化、模板化、重复性工作。
3. AI 必须在明确约束内工作，不能自创体系。

## 2. AI 在各阶段应做什么
### 2.1 需求阶段
1. 将模糊想法整理为 PRD 草案。
2. 抽取角色、场景、流程、功能清单、待确认项。
3. 识别范围失控与术语冲突。

### 2.2 UI/UX 阶段
1. 根据 PRD 与 Pencil 设计稿整理页面树、组件清单、状态清单。
2. 标记设计缺失项，如空态、错误态、权限态。

### 2.3 前端阶段
1. 生成目录骨架、路由、页面容器、业务组件、API 模块、store、composables。
2. 确保遵守 Vue 3 + TypeScript + Vite + Pinia + Vue Router + Element Plus。

### 2.4 后端阶段
1. 生成模块骨架、DTO、VO、Entity、Mapper、Service、Controller。
2. 根据 API 与数据库规范补充校验、异常、日志、审计、权限扩展点。

### 2.5 数据库阶段
1. 从业务对象生成数据模型、表结构、索引建议。
2. 标记逻辑删除、状态机、字典表、关联表、日志表策略。

### 2.6 测试阶段
1. 生成单元测试、接口测试、联调清单、Mock 数据、回归清单。
2. 将验收标准映射成测试用例。
3. 按务实分层 TDD 介入开发，而不是默认在实现完成后才补测试。

### 2.7 部署阶段
1. 生成 Dockerfile、compose 骨架、环境变量清单、发布与回滚说明。
2. 生成 CI/CD 流水线草案。

## 3. AI 生成代码前必须先读取哪些规范
1. 所有任务先读 `README.md` 与 `00_GLOBAL_ENGINEERING_PRINCIPLES.md`。
2. 涉及需求先读 `01_PRODUCT_REQUIREMENTS_STANDARD.md`。
3. 涉及页面先读 `02_UI_UX_DESIGN_SPEC.md` 和 `03_FRONTEND_ARCHITECTURE_SPEC.md`。
4. 涉及接口和后端先读 `04_BACKEND_ARCHITECTURE_SPEC.md` 与 `05_API_DESIGN_SPEC.md`。
5. 涉及数据和缓存先读 `06_DATABASE_DESIGN_SPEC.md`、`07_REDIS_CACHE_SPEC.md`、`20_DATA_MODELING_PLAYBOOK.md`。
6. 涉及代码生成先读 `08_CODE_STYLE_AND_NAMING_SPEC.md`。
7. 涉及发布与安全先读 `13` 至 `18` 号文档。

## 4. AI 遇到不确定技术选型时的决策顺序
1. 先查母规范是否已有明确默认决策。
2. 若母规范无定义，查项目级文档。
3. 若仍无定义，输出备选方案与影响，并请求人工确认。
4. 不得在关键栈上自作主张切换框架、数据库、协议风格。

## 5. AI 不得擅自更换技术栈
禁止擅自切换：
1. Vue 3 到 React。
2. Spring Boot 到 Node.js 主后端。
3. MySQL 到 PostgreSQL/MongoDB。
4. RESTful 到 GraphQL。
5. Docker compose 到其他编排方式作为默认交付。

## 6. AI 如何生成项目级文档
1. 先复制母规范目录。
2. 生成项目级 `PROJECT_OVERRIDES.md`，只写增量约束与显式覆盖。
3. 记录覆盖原因、覆盖范围、审批人、失效条件。

## 7. AI 输出不同产物时必须遵守的规则
1. 代码：遵守架构分层、命名、异常处理、权限、安全、日志规范。
2. SQL：遵守命名、审计字段、逻辑删除、索引规范。
3. API：遵守 URI、Method、统一返回、错误码、幂等规范。
4. 测试：遵守测试分层、提测门槛、验收标准映射。
5. Docker 文件：遵守镜像分层、配置注入、健康检查、日志规范。

## 7.1 AI 的 TDD 执行义务
1. AI 不得把“先实现、后补验证”当作默认模式。
2. 对后端业务逻辑、状态机、校验、查询过滤、映射转换、回归 bug 修复等任务，AI 必须优先选择“先写失败测试或失败断言，再实现”。
3. 对无法立即自动化的 bug 或页面交互任务，AI 必须先沉淀最小复现步骤、测试点或验收点，再进入实现。
4. 对数据库迁移任务，AI 必须先写迁移前后验证点，再提交迁移脚本和执行验证。
5. AI 必须在最终输出中明确说明本轮测试证据，不能只说“已修改代码”。
6. AI 在每轮测试、联调、smoke 或预验收后，必须主动检查并清理本轮临时拉起的前端、后端、脚本、容器或端口转发实例；若因明确原因需要保留，也必须在项目级文档中写明理由。

## 8. 减少上下文漂移的方法
1. 每次新任务先声明引用了哪些规范文件。
2. 输出前复核是否与固定技术栈冲突。
3. 对未确认事项显式列出，而不是隐式猜测。
4. 长周期项目使用统一 Prompt 模板与项目级覆盖文档。

## 8.1 Session Handoff 模板
单人 + AI 长周期协作时，每次任务结束应沉淀：
```md
## Session Handoff
- 本次完成：
- 当前状态：
- 下一步：
- 风险与阻塞：
- 待人工确认：
- 已更新文档：
```

补充要求：
1. `Session Handoff` 只保留当前有效交接，不得长期累计成项目全历史流水账。
2. 长历史进度、已关闭阶段和已结束分支任务，应转入项目级变更历史或归档目录。

## 8.2 AI 固定输出格式
AI 在复杂任务结束时，建议至少输出：
1. 已完成事项。
2. 下一步建议。
3. 风险与假设。
4. 待确认项。
5. 已修改文件或已更新文档。

## 8.3 AI 必须更新的文档
在以下情况必须同步更新文档：
1. 需求范围变化时，更新 PRD 或范围记录。
2. API 变化时，更新接口文档或变更记录。
3. 表结构变化时，更新数据模型或 SQL 脚本说明。
4. 上线前后，更新验收与发布记录。

## 8.3.1 项目文档分层要求
1. 项目文档应至少区分：当前总控文档、当前执行中的模块级正式审阅稿、当前执行中的核心执行说明、已审阅参考资料、详细实现文档、历史归档。
2. 当前执行资料与已审阅参考资料必须显式分开，避免误把历史稿件当作当前执行输入。
3. 历史交接、旧审阅稿、旧核心执行说明、被替换的长历史记录，应迁入 `archive/` 或等价归档区，而不是继续堆在主入口文档中。

## 8.3.2 高信噪比输出要求
1. 除正式代码与正式文档外，AI 与用户的过程性互动默认应保持高信噪比。
2. 输出优先采用结构化短结论，而不是重复背景、重复引用、重复解释。
3. 在不丢失核心信息的前提下，优先压缩为：当前判断、影响范围、下一步、待确认点。
4. 若信息已在正式文档中存在，过程性互动中只引用关键结论，不重复大段展开。

## 8.4 用户实机验证前的 AI 预验收义务
1. 当且仅当本轮任务已进入“确有必要要求用户实际登录系统进行验证”的阶段时，AI 才负有真实浏览器预验收义务，不得直接把首轮页面验证转交给用户。
2. 若当前环境已安装 Chrome DevTools MCP 或等价真实浏览器自动化入口，AI 必须先在真实浏览器中完成一轮自动预验收；若本轮并不需要用户实际登录验证，则必须继续使用构建、接口 smoke、日志与本地回归等原有验证手段，且不得调用该工作流。普通编码、局部修复、纯后端改动、文档改动、数据库预埋、接口 smoke、构建验证、日志排查等场景一律视为禁止调用场景。
3. 预验收至少覆盖：
   - 真实账号登录
   - 关键页面跳转
   - 关键按钮点击与主要流程操作
   - 表格、表单、弹窗、导航、分页、空态与错误态检查
   - 控制台报错、接口失败、视觉错位、控件不可用等异常捕获
4. 发现问题后，AI 应优先直接定位、修复、回归，而不是先让用户重复遇到同类问题。
5. 每轮预验收完成后，AI 必须输出执行简报，并同步记录到项目级测试、交接或质量文档。
6. 若当前会话缺少 Chrome DevTools MCP 或真实浏览器自动化入口，AI 必须明确记录该阻塞，并把“真实浏览器预验收待补执行”写入项目级文档。
7. 若预验收或其他验证动作中临时启动了前端 dev server、后端测试实例、Docker 容器或端口代理，AI 在完成验证后必须主动执行清理，避免留下历史残留实例。

## 9. 让 Codex / VSCode 长期稳定工作的建议
1. 将本目录放入仓库根目录并长期维护。
2. 在 `AGENTS.md`、IDE Workspace 指令、系统 Prompt 中引用母规范路径。
3. 新成员与新 Agent 先读 `README.md`、`00`、`15`、`21`。
4. Review 时检查 AI 产物是否引用了正确规范。

## 10. AI 输出模板
```md
### 本次任务引用规范
- 00_GLOBAL_ENGINEERING_PRINCIPLES.md
- 03_FRONTEND_ARCHITECTURE_SPEC.md
- 08_CODE_STYLE_AND_NAMING_SPEC.md

### 待人工确认
- 是否需要新增审核撤回流程
- 是否要求导出异步化
```

## 本文件使用建议
将本文件作为所有 AI 工具的入口工作流文档，要求 AI 在开始编码前先声明所引用的规范范围。

## AI 调用建议
AI 每次输出代码、方案、SQL、测试或部署脚本时，都应在内部工作流中先按本文件完成“读规范 -> 判断边界 -> 生成 -> 自检”的顺序。
