# 文件名
`17_OBSERVABILITY_SPEC.md`

# 文件定位
日志、指标、链路追踪、告警与运行观测规范。

# 适用范围
适用于前端、后端、Python 服务、Redis、MySQL 及部署平台的观测设计。

# 与其他文件的关系
本文件与 `12_DEBUG_AND_TROUBLESHOOTING_SPEC.md`、`13_DEPLOYMENT_AND_DEVOPS_SPEC.md` 共同构成运行期治理规范。

# 编写目的
建立日志、指标、链路与告警的统一观测语言，让系统问题可发现、可定位、可量化、可复盘。

## 1. 观测目标
1. 快速发现故障。
2. 快速定位问题。
3. 快速评估影响范围。
4. 为容量规划和优化提供数据。

## 2. 三大观测支柱
1. 日志：记录发生了什么。
2. 指标：量化系统状态。
3. Trace：串联请求链路。

## 3. 日志规范
1. 日志级别至少区分 `INFO`、`WARN`、`ERROR`。
2. 关键业务操作写业务日志。
3. 所有请求日志应带 `traceId`。
4. 错误日志应包含异常分类、关键上下文、资源 ID。
5. 禁止记录敏感明文。

## 4. 指标规范
必须具备以下基础指标：
1. 请求量、成功率、错误率、响应时间。
2. JVM、CPU、内存、线程、磁盘。
3. 数据库连接池、慢查询数量。
4. Redis 命中率、连接数、慢命令。
5. 异步任务积压量、失败数。

## 5. Trace 规范
1. 网关、后端、Python 服务之间统一传递 `traceId`。
2. 核心跨服务调用必须可串联。
3. 数据库、缓存、外部 API 耗时应可观测。

## 6. 告警规范
1. 告警应按严重程度分级。
2. 生产不可用、错误率突增、数据库异常、缓存雪崩需立即告警。
3. 告警必须绑定负责人和值班规则。
4. 避免噪音告警，需设置聚合与抑制策略。

## 7. 前端观测建议
1. 记录 JS 异常、接口失败、页面白屏、资源加载失败。
2. 记录关键页面加载耗时和关键交互失败率。
3. 必要时上报用户标识、浏览器信息、版本号，但不得泄露隐私数据。

## 8. 后端观测建议
1. 提供健康检查与运行指标端点。
2. 关键任务、导入导出、审批流应有业务指标。
3. 慢查询与异常接口需进入重点看板。

## 9. 仪表盘最小集合
1. 全局服务状态看板。
2. API 质量看板。
3. 数据库与 Redis 运行看板。
4. 任务执行看板。
5. 登录与权限异常看板。

## 10. 观测落地清单
1. 是否生成 `traceId`。
2. 是否有统一日志字段。
3. 是否暴露关键指标。
4. 是否配置核心告警。
5. 是否保留发布版本与环境维度。

## 本文件使用建议
系统上线前应至少完成本文件定义的日志、指标、告警最小闭环，避免“可运行但不可运维”的交付。

## AI 调用建议
AI 在生成日志、监控、健康检查、追踪上下文代码时，应按本文件补齐 traceId、业务指标和告警所需字段。
