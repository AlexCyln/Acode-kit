# 文件名
`30_DEVELOPMENT_DOCUMENTATION_GOVERNANCE_SPEC.md`

# 文件定位
开发过程文档治理专项规范。

# 适用范围
适用于使用本套母规范推进的单人 + AI 或小团队软件项目，覆盖需求落地后的设计、开发、联调、测试、上线准备阶段。

# 与其他文件的关系
- 以 `00_GLOBAL_ENGINEERING_PRINCIPLES.md` 为总纲。
- 与 `05_API_DESIGN_SPEC.md`、`06_DATABASE_DESIGN_SPEC.md`、`11_TESTING_AND_QA_SPEC.md`、`15_AI_COLLABORATION_PLAYBOOK.md`、`27_PROJECT_EXECUTION_FLOW_SPEC.md` 联动使用。
- 本文件关注“开发过程文档必须存在并持续更新”的治理要求，不替代任何专题文档本身。

# 编写目的
确保接口文档、数据库文档、功能函数文档、测试文档在项目推进过程中持续存在、持续更新、可查可交接，避免只依赖临时会话说明或零散提交信息。

## 1. 基本原则
1. 文档不是收尾动作，而是开发过程的一部分。
2. 任何已实现的关键能力，都应能在对应详细文档中找到口径、边界与验证结论。
3. AI 和研发在开始实现前，必须先检查目标子任务对应的详细文档是否已存在。
4. 若缺少对应文档入口，先补 `md`，再进入实现。
5. 对于新的大模块，若仍存在较大范围不确定性，必须先补模块级正式审阅稿，经过确认后再进入实现。
6. 对于模块下的子模块 / 系统 / 页面，在执行前必须先补轻量“核心执行说明”，经过确认后再进入实现。
7. 对于需要页面承载的节点，模块级 UI 设计审阅稿、页面核心执行说明与前端复刻/还原记录都属于开发过程文档治理范围。

## 2. 必须维护的开发过程文档
每个项目在 `docs/project/` 下，至少应持续维护以下四类文档：

### 2.1 接口文档
- 适用触发条件：
  - 出现真实接口
  - 已定义前后端联调契约
  - 接口字段、权限、缓存、状态口径发生变化
- 最低要求：
  - 路径、方法、用途
  - 请求头、参数、请求体
  - 响应结构与示例
  - 权限要求
  - 错误口径
  - 缓存 / 幂等 / 联调说明

### 2.2 数据库文档
- 适用触发条件：
  - 新增建表设计
  - 变更字段、索引、约束
  - 执行增量迁移
- 最低要求：
  - 表关系
  - 字段、类型、默认值、中文说明
  - 状态值 / 字典关系
  - 执行状态：待审阅 / 待执行 / 已执行落库

### 2.3 功能函数文档
- 适用触发条件：
  - 新增关键业务模块
  - 新增公共服务、状态机、校验逻辑、核心函数
  - 关键实现规则发生变化
- 最低要求：
  - 模块职责
  - 关键入口文件
  - 核心函数 / 服务
  - 依赖的表、缓存、接口
  - 关键规则与边界
  - 已知限制

### 2.4 测试文档
- 适用触发条件：
  - 执行构建验证
  - 执行接口 smoke
  - 执行写路径联调
  - 执行关键手工验收
- 最低要求：
  - 时间
  - 范围
  - 执行方式 / 命令
  - 结果
  - 异常
  - 结论

### 2.5 模块级正式审阅稿
- 适用触发条件：
  - 新的大模块正式启动
  - 模块实现范式、边界、输入输出、验证口径需要先冻结
- 类型收敛：
  - 需求变更审阅稿
  - 数据库表设计审阅稿
  - 接口设计审阅稿
  - UI 设计审阅稿
  - 测试用例审阅稿
- 治理规则：
  - 正式审阅稿只在模块级产出，不在每个子模块 / 系统 / 页面重复铺开
  - 五类正式审阅稿按需存在，不强制模块机械凑满
  - 历史页面级 / 子模块级审阅稿默认作为历史参考保留，不要求批量重写

### 2.6 核心执行说明
- 适用触发条件：
  - 模块下的子模块 / 系统 / 页面准备开始执行
  - 本次执行需要在实现前冻结范围和边界
- 最低要求：
  - 本次目标
  - 输入依据
  - 实施范围
  - 明确不做
  - 执行步骤
  - 待确认点

### 2.7 UI 设计与复刻记录
- 适用触发条件：
  - 页面/模块需要先做 UI 设计
  - 页面/模块已进入 Pencil 设计或前端复刻
- 最低要求：
  - 模块级 UI 设计审阅稿
  - 当前页面核心执行说明
  - Pencil 设计确认结论
  - 前端一比一复刻范围与验收结论

## 3. 文档存在性检查规则
1. 每次开始新子任务前，先检查是否存在对应文档。
2. 若不存在，则至少新增一个可持续维护的 `md` 入口文档。
3. 新增文档后，应立即纳入项目文档索引或目录总表。
4. 不允许出现“代码已经实现，但无法定位详细文档”的情况。

## 3.1 文档分层与归档规则
1. 项目级文档应至少分为：
   - 当前有效总控文档
   - 当前执行中的模块级正式审阅稿
   - 当前执行中的核心执行说明
   - 已审阅通过但当前不执行的参考资料
   - 接口 / 数据库 / 功能函数 / 测试等详细实现文档
   - 历史归档
2. 当前执行中的模块级正式审阅稿、核心执行说明必须与已审阅参考资料显式分开登记。
3. 已结束且当前不再主导执行的审阅稿、核心执行说明、旧交接、被替换的长历史记录，应迁入归档区。
4. 总控文档只保留当前有效信息，不得长期混入完整历史流水。

## 4. 重要节点的强制更新规则
以下节点发生时，必须同步更新对应详细文档：
1. 接口首次真实可用。
2. 表结构首次落库或发生增量迁移。
3. 核心函数 / 状态机 / 校验规则发生变化。
4. 关键 smoke / 联调 / 构建验证执行完成。
5. 影响后续查阅、联调、排障、交接的重要问题被修复。
6. 模块级正式审阅稿确认通过、页面核心执行说明确认通过、Pencil 设计确认通过、前端一比一复刻确认通过。

## 5. 项目建议文档清单
项目级除总控文档外，建议至少有：
1. `DEVELOPMENT_DOCUMENTATION_INDEX.md`：开发文档索引
2. `API_DETAIL_CATALOG.md`：接口详细文档
3. `FUNCTION_MODULE_RECORD.md`：功能函数与模块记录
4. `TEST_EXECUTION_RECORD.md`：测试与验证记录

## 6. AI 执行要求
1. AI 在每轮任务结束前，必须检查本轮是否影响接口文档、数据库文档、功能函数文档、测试文档。
2. 若影响而未更新，应优先补文档，再结束任务。
3. `DECISION_LOG`、`TRACEABILITY_MATRIX`、`SESSION_HANDOFF` 只能作为总控文档，不能替代详细过程文档。

## 7. 验收标准
当项目进入联调、提测、上线准备阶段时，应满足：
1. 已实现接口可在详细接口文档中查到。
2. 已执行数据库变更可在数据库文档中查到。
3. 关键业务模块和核心函数可在功能函数文档中查到。
4. 已执行验证动作可在测试文档中查到。

## 8. 反例
1. 只有总控文档，没有详细接口文档。
2. 只有 SQL，没有结构化数据库说明文档。
3. 关键状态机只存在代码里，没有任何文字说明。
4. 测试只说“已验证”，但没有记录范围、方式和结论。

## 9. 使用建议
将本文件作为项目进入“真实开发”后的文档治理基线。任何长期推进、多人接手或多轮 AI 会话项目，都应强制启用本规范。
