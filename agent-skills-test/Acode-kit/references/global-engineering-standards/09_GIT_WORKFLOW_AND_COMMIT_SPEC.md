# 文件名
`09_GIT_WORKFLOW_AND_COMMIT_SPEC.md`

# 文件定位
Git 分支协作、Commit、PR 与版本管理规范。

# 适用范围
适用于单仓、多仓、前后端分仓、单仓多应用等团队协作场景。

# 与其他文件的关系
本文件与 `10_CODE_REVIEW_SPEC.md`、`14_CICD_SPEC.md` 联动，约束代码进入主分支与环境流水线的方式。

# 编写目的
统一分支协作、提交信息、PR 与版本管理规则，确保研发过程可追踪、可审查、可发布、可回滚。

## 1. 分支模型
1. `main` 或 `master`：生产稳定分支，仅接受受控合并。
2. `develop`：日常集成分支。
3. `feature/*`：功能开发分支。
4. `release/*`：发布准备分支。
5. `hotfix/*`：生产紧急修复分支。

## 2. 分支命名模板
1. `feature/user-management`
2. `feature/order-approve-flow`
3. `release/v1.4.0`
4. `hotfix/login-token-expire`

规则：
1. 统一小写英文与中划线。
2. 分支名体现业务主题，不写个人昵称。
3. 需求编号可前置，如 `feature/PROJ-128-user-import`。

## 3. 提交流程
1. 从 `develop` 或指定基线拉出 `feature/*`。
2. 本地自测通过后提交 PR。
3. PR 通过 Review 与必要流水线后合并。
4. 发版时从 `develop` 切 `release/*`，仅允许修复发布问题。
5. 线上紧急修复走 `hotfix/*`，完成后回合并到 `main` 与 `develop`。

## 4. Commit Message 规范
推荐 Conventional Commits：
1. `feat: 新增用户批量导入功能`
2. `fix: 修复订单审核重复提交问题`
3. `refactor: 拆分用户详情页查询逻辑`
4. `docs: 更新全局部署规范`
5. `test: 补充登录接口单元测试`
6. `chore: 调整 Docker 构建参数`

规则：
1. 第一行说明变更类型与核心内容。
2. 一个 Commit 聚焦一个明确目的。
3. 禁止 `update`, `modify`, `test123` 这类无语义提交信息。

## 5. PR 标题规范
1. `[Feature] 用户管理批量导入`
2. `[Fix] 订单审核重复提交`
3. `[Refactor] 拆分权限模块缓存逻辑`

PR 描述至少包含：
1. 变更背景。
2. 变更范围。
3. 风险点。
4. 测试说明。
5. 回滚影响。

## 6. 合并策略规范
1. 默认使用 `Squash and Merge` 或 `Rebase and Merge`，保持主线清晰。
2. 是否保留历史分支提交由项目级统一，不允许个人随意切换。
3. 禁止把调试提交、格式化噪音提交直接合入主线。

## 7. 提交粒度要求
1. 一个 Commit 只做一个逻辑变更。
2. 代码重构与功能变更尽量分开提交。
3. 大范围重命名、格式化、依赖升级要独立提交。

## 8. 禁止直接改主分支
1. 禁止直接向 `main/master` 推送。
2. `develop` 也不允许绕过 PR 与 Review 直接强推，除非组织级紧急授权。
3. 热修复完成后必须补充 Review 与回归记录。

## 9. 版本号策略
采用语义化版本：
1. `MAJOR`：破坏性变更。
2. `MINOR`：向后兼容的新功能。
3. `PATCH`：向后兼容的问题修复。

示例：
1. `v1.0.0`：首个正式版。
2. `v1.3.0`：新增模块。
3. `v1.3.2`：修复线上 Bug。

## 10. 发布标签建议
1. 生产发布打 Tag，如 `v1.4.0`。
2. 预发布可打 `v1.4.0-rc.1`。
3. Tag 必须对应发布记录与回滚依据。

## 11. Git 操作禁令
1. 未沟通情况下强制推送共享分支。
2. 将未评审 SQL、配置、密钥文件直接合入主线。
3. 大量无关文件混入同一 PR。

## 12. 自检清单
1. 分支是否从正确基线切出。
2. Commit 是否语义清晰。
3. PR 是否说明风险与测试结果。
4. 是否含敏感文件与无关改动。

## 本文件使用建议
仓库应通过保护分支、PR 模板、Commit Lint、Tag 规范来制度化执行本文件。

## AI 调用建议
AI 生成变更说明、Commit Message、PR 描述、发布说明时，必须遵循本文件中的分支、提交与版本语义。
