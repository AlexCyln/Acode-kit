# 文件定位
B2B SaaS Admin UI 规范包。

## 适用范围
- B 端管理台
- 运营后台
- 控制台系统

## 典型技术组合位置
- 与 React + shadcn 或类似 B 端组件体系配合
- 适用于高信息密度、高操作频率、强权限控制的后台界面

## 推荐目录结构
- 统一的导航框架
- 列表页、详情页、表单页、配置页采用稳定页型
- 页面区块、工具栏、筛选区、操作区、日志区有统一骨架

## 分层与职责边界
- 视觉规范服务于效率和可解释性，不服务于炫技装饰
- UI 包定义页型、信息密度、交互模式，不替代前端框架和组件库规范
- 页面级特殊风格必须能说明业务理由

## 编码复杂度与可维护性要求
- 信息密度高但层级必须清晰
- 表格、筛选器、标签、状态、危险操作模式统一
- 同类页面的筛选区、操作区、详情区、日志区保持一致结构
- 不允许因单页面需求把整体后台模式打散

## 测试策略与质量门禁
- 列表筛选、批量操作、关键表单、危险操作确认要覆盖
- 列表页、详情页、编辑页至少各有一条关键路径 smoke
- 审核、导出、权限态、空态、错误态必须纳入验收

## 数据与 Migration 协同要求
- 页面字段、筛选项、状态标签必须与真实 API / schema / 字典一致
- 新增字段和状态值要同步进入列表、详情和编辑页的统一模式中

## API / 安全 / 权限协同要求
- 列表、详情、按钮、批量操作都必须受统一权限控制
- 危险操作必须有二次确认、审计留痕和失败反馈
- 导出、删除、审核类动作不得仅做前端限制

## 日志与可观测性协同要求
- 关键后台操作应具备审计入口和排障上下文
- 危险操作、批量操作、导入导出失败要可定位

## 多租户适配要求
- 多租户后台必须清楚区分平台视角和租户视角
- 平台级菜单、租户级菜单、组织级数据范围不允许混乱呈现

## 文档要求
- 模块文档说明页面骨架、关键状态、危险操作点、权限点和日志入口

## 常见误用
- 用营销站思路做管理台
- 列表、详情、编辑风格割裂
- 过度装饰导致信息层级混乱

## 禁止事项
- 在审批前随意增加视觉装饰导致信息噪音
- 用单页“特殊做法”破坏整个后台的一致交互模型

## 追加增强：B 端后台页型细则

### 列表页
- 筛选区字段顺序按业务决策频率排序
- 批量操作、导出、危险操作必须有明确风险提示

### 详情页
- 头部摘要必须优先展示主键、状态、关键指标、责任人、时间信息
- 历史记录、审计记录、操作日志应有稳定落点

### 编辑页
- 长表单优先分组或分步，不允许无层级长表单堆叠
- 必填、默认值、帮助说明、错误提示样式要统一
