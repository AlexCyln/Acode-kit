# 文件定位
Vercel Web 交付规范包。

## 适用范围
- Next.js / 前端 Web 项目
- 以 Vercel 为主的前端或前后端一体化 Web 交付
- 需要预览环境、环境变量治理和快速回滚的项目

## 典型技术组合位置
- Next.js 或前端 Web 应用部署到 Vercel
- 配合 Git 集成、Preview Deployment、环境变量和监控工具使用

## 推荐目录结构
- `deploy/vercel/` 放 Vercel 配置、发布约束和运行说明
- 环境变量、预览环境说明、回滚说明有固定文档落点

## 分层与职责边界
- 应用负责业务能力
- Vercel 交付层负责构建、环境、域名、预览环境和发布约束
- 不把环境差异和生产密钥硬编码进应用代码

## 编码复杂度与可维护性要求
- 构建脚本、环境变量和域名策略项目级统一
- Preview、Production、必要时 Development 环境职责清晰
- 一次只处理一个明确交付目标，不把前端发布和后端迁移混成黑盒脚本

## 测试策略与质量门禁
- 发布前至少完成构建、环境变量校验和最小 smoke
- 高风险 UI 或路由改动应先在 Preview 环境验收
- 回滚前后要有可验证的访问与关键路径检查

## 数据与 Migration 协同要求
- 若项目同时含服务端 schema 变更，必须说明与 Vercel 发布的顺序关系
- 不允许默认假设“前端先发不会影响后端 contract”

## API / 安全 / 权限协同要求
- 环境变量、密钥、第三方 token 不入仓库
- 域名、回调地址、OAuth 配置需与环境显式绑定

## 日志与可观测性协同要求
- 发布后要能关联版本、commit、环境、错误率和关键访问路径
- 前端异常、路由错误和首屏失败要可追踪

## 多租户适配要求
- 多租户 Web 项目需说明租户域名、子路径或 header 策略
- Preview 环境不能误连生产租户数据

## 文档要求
- 记录环境变量来源、域名映射、发布策略、回滚方式和预览验收要求

## 常见误用
- 把环境变量与本地调试配置混写
- Preview 通过即默认 Production 无风险
- 回滚只看代码不看环境和域名配置

## 禁止事项
- 未声明环境变量边界就直接上线
- 在无回滚说明和无预览验收的情况下发布高风险变更

## 追加增强：Vercel 交付细则

### 预览环境细则
- 关键页面、关键路由和登录链路要在 Preview 环境先验收
- 外部回调、白名单和 OAuth 场景要明确 Preview 策略

### 回滚细则
- 回滚前确认配置、域名和第三方回调未受影响
- 回滚后要重新执行最小 smoke

### Review 附加清单
- 是否明确 Preview / Production 差异
- 是否记录环境变量来源和回滚路径
- 是否能从监控中定位版本与错误
