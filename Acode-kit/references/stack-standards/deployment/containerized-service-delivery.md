# 文件定位
容器化服务交付规范包。

## 适用范围
- Docker / Compose / Kubernetes 为主的服务交付项目
- 需要标准化镜像、配置、迁移、回滚和观测的企业级系统

## 典型技术组合位置
- 应用服务容器化交付
- 配合 CI/CD、制品仓库、配置中心、日志与指标体系

## 推荐目录结构
- `deploy/` 或等价目录承载部署脚本、环境模板、清单和回滚说明
- 应用镜像、迁移任务、作业任务、环境配置边界清晰

## 分层与职责边界
- 应用负责业务能力
- 部署编排负责环境组装、配置注入、制品发布和 rollout
- Migration 执行链路必须显式，不得混入人工步骤

## 编码复杂度与可维护性要求
- 部署脚本命名和职责清晰
- 一个脚本只处理一个明确目标，如构建、部署、回滚、迁移、健康检查
- 不把大量环境差异硬编码在同一脚本里

## 测试策略与质量门禁
- 发布前至少完成构建、镜像、配置校验、健康检查和最小 smoke
- 高风险变更要有预发验证和回滚演练
- Migration 执行前后要有验证点

## 数据与 Migration 协同要求
- Migration 步骤必须显式进入发布链路
- 结构变更、回填、应用发布顺序要可说明、可审计、可回退

## API / 安全 / 权限协同要求
- 环境配置中的凭据、密钥、Token 不入仓库
- 发布、回滚、环境切换必须有权限边界和审计记录

## 日志与可观测性协同要求
- 发布后必须能看见版本、环境、实例、错误率、健康检查和核心日志
- 回滚和故障定位必须能关联到构建号、镜像 Tag 和部署批次

## 多租户适配要求
- 多租户项目发布时要确认租户级配置、隔离参数和迁移策略未被破坏

## 文档要求
- 部署文档必须记录制品版本、环境变量来源、迁移执行方式、回滚路径和验证清单

## 常见误用
- 生产变更靠人工 SSH 执行散落命令
- 先手工改库再补发布记录
- 发布成功但无健康检查与观测确认

## 禁止事项
- 绕过正式发布链路直接改正式环境
- 在无回滚路径、无验证点、无审计记录的情况下发布高风险变更

## 追加增强：企业级交付细则

### 发布链路细则
- 制品版本、配置版本、Migration 版本、观测版本必须可关联
- 高风险发布建议分“迁移 → 后端 → 辅助服务 → 前端 → 验证”逐步推进

### 回滚细则
- 回滚前先确认数据是否可逆、配置是否兼容、观测告警是否足以支撑判断
- 回滚后必须重新执行最小健康检查和关键链路 smoke

### 评审附加清单
- 是否明确生产审批节点
- 是否记录发布负责人、回滚负责人、观察期指标
