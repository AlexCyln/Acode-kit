# 文件定位
React + shadcn/ui 前端实现规范包。

## 适用范围
- Web 管理台
- B 端运营后台
- 需要组件化、类型化和一致交互的 React 项目

## 典型技术组合位置
- React 负责页面与状态
- shadcn/ui 负责标准交互组件
- 配合 Tailwind 或项目既定样式体系
- API 层、权限层、日志/埋点层、表单层边界清晰

## 推荐目录结构
- `app/` 或 `src/` 作为应用根
- `components/` 放页面组合组件，不重复造基础组件
- `features/` 或 `modules/` 按业务模块分组
- `lib/` 放工具、API 客户端、schema、权限辅助
- `hooks/`、`services/`、`schemas/`、`guards/`、`telemetry/` 职责清楚

## 分层与职责边界
- 页面组件优先负责编排
- hook 负责查询、表单、权限、数据装配等可复用逻辑
- service / api 负责请求和协议适配
- 通用组件与业务组件显式分层
- 不允许把数据请求、复杂权限判断、埋点和页面模板混在同一组件

## 编码复杂度与可维护性要求
- 声明使用 shadcn 后，不自造 Button/Input/Card/Dialog/Table 等基础组件
- 页面实现必须一比一对应批准设计
- 页面内如出现 2 层以上连续条件嵌套，应优先考虑 early return、配置映射或子组件拆分
- 列定义、筛选映射、表单 schema、按钮权限、状态展示逻辑应模块化管理

## 测试策略与质量门禁
- 关键页面用 Vitest + RTL 做交互测试
- 表单校验、过滤器、关键按钮和错误态需要覆盖
- 真实浏览器预验收按模块风险分级触发
- 页面级 smoke 至少覆盖加载、查询、关键动作和失败反馈

## 数据与 Migration 协同要求
- 若后端 schema、字典、状态值发生变更，前端 schema、mock、筛选项和状态展示必须同步更新
- 不允许页面依赖过时字段或私有映射继续“假通过”

## API / 安全 / 权限协同要求
- 页面级数据请求通过统一 API 层，不在组件树中散落直接 fetch
- 前端只做展示拦截和体验提示，后端仍是最终鉴权
- 权限码、菜单、按钮、路由命名统一
- 敏感信息展示默认脱敏，导出行为必须提示权限和风险

## 日志与可观测性协同要求
- 关键页面错误、接口失败、空白页和核心操作失败要可记录
- 前端日志、埋点、错误上报要带版本、页面、用户和租户上下文

## 多租户适配要求
- 多租户项目页面必须考虑租户上下文切换、权限差异和数据隔离
- 不允许在页面层随意伪造租户上下文

## 文档要求
- 在模块文档中记录组件组合方式
- 在测试文档中记录关键交互覆盖点
- 记录已声明的 UI 组件约束、表单 schema、权限点与关键页面状态

## 常见误用
- 用自定义基础组件替代现成 shadcn 组件
- 把业务逻辑塞进通用组件
- 把权限、状态、请求、埋点散落在页面模板中

## 禁止事项
- 跳过批准设计直接自由发挥 UI
- 在声明 shadcn 后回退到裸 HTML/CSS 基础组件体系

## 追加增强：企业级前端细则

### 页面骨架细则
- 列表页应优先形成“筛选区 + 工具栏 + 表格 + 分页”稳定骨架
- 详情页应优先形成“头部摘要 + 分区内容 + 操作区 + 日志区”稳定骨架

### 状态与表单细则
- 查询参数、表单 schema、错误反馈、提交态、禁用态要显式建模
- 重复提交、权限受限、网络失败、旧数据兼容都必须有明确处理方式

### Review 附加清单
- 是否仍有页面组件承担过多逻辑
- 是否存在基础组件重复造轮子
- 是否把权限、埋点、请求、页面结构揉在一起
