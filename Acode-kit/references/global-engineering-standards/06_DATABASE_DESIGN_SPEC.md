# 文件名
`06_DATABASE_DESIGN_SPEC.md`

# 文件定位
数据库建模、建表与变更治理规范。

# 适用范围
适用于所有业务主库、基础配置库、日志库、任务库等表结构设计场景。具体数据库类型由 `PROJECT_OVERRIDES.md` 确定。

# 与其他文件的关系
本文件依赖 `01_PRODUCT_REQUIREMENTS_STANDARD.md` 与 `20_DATA_MODELING_PLAYBOOK.md` 的业务输入，并与 `05_API_DESIGN_SPEC.md`、`07_REDIS_CACHE_SPEC.md` 保持字段语义一致。

# 编写目的
统一数据库表结构设计、命名、审计字段、索引和逻辑删除策略，让数据库设计在不同项目间保持稳定可继承。

## 1. 建模原则
1. 先有业务对象与状态机，再有表结构。
2. 一张表只服务一个主要实体或清晰的关系语义。
3. 优先第三范式，必要冗余必须说明原因与同步策略。
4. 表结构设计必须服务查询、审计、权限、扩展。

## 2. 命名规范
1. 表名使用小写下划线，如 `sys_user`、`biz_order`。
2. 主表以领域前缀区分，如 `sys_`、`biz_`、`log_`、`cfg_`。
3. 字段名使用小写下划线，如 `created_time`、`tenant_id`。
4. 普通索引命名为 `idx_表名缩写_字段名`。
5. 唯一索引命名为 `uk_表名缩写_字段名`。

## 3. 主键策略
1. 默认使用整型作为主键类型（如 `BIGINT`、`INTEGER`、`SERIAL` 等，按已声明数据库选择适当类型）。
2. 单库单体项目可使用自增主键。
3. 需要跨服务写入、离线导入、分布式扩展时，使用应用侧统一 ID 生成方案，如雪花 ID。
4. 主键字段统一命名为 `id`。

## 4. 通用审计字段
除极少数纯中间表外，建议默认包含：
1. `created_by`
2. `created_time`
3. `updated_by`
4. `updated_time`
5. `is_deleted`
6. `version`，适用于乐观锁场景时启用
7. `remark`，仅用于简要备注，不承载长文本业务内容

示例：
```sql
created_by   BIGINT       NOT NULL DEFAULT 0 COMMENT '创建人ID',
created_time DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
updated_by   BIGINT       NOT NULL DEFAULT 0 COMMENT '更新人ID',
updated_time DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
is_deleted   TINYINT(1)   NOT NULL DEFAULT 0 COMMENT '逻辑删除标记',
version      INT          NOT NULL DEFAULT 0 COMMENT '乐观锁版本号'
```

说明：以上 SQL 语法为参考示例，具体类型和语法按已声明数据库调整。

## 5. 逻辑删除策略
默认支持逻辑删除，但需区分适用范围。

适合逻辑删除的表：
1. 用户、角色、菜单、字典等基础资料表。
2. 业务主数据表。
3. 可恢复的配置类表。

不适合逻辑删除的表：
1. 操作日志、审计日志、行为流水。
2. 大批量明细流水表。
3. 纯关联中间表且数据可重建。
4. 高吞吐临时任务表。

规则：
1. 使用逻辑删除的表必须在查询层默认过滤 `is_deleted = 0`。
2. 逻辑删除不等于审计留痕，删除动作仍需审计日志。

## 6. 外键策略
1. 默认不在生产业务库中强依赖物理外键。
2. 关联关系通过应用层约束、索引、代码校验保证。
3. 关联字段必须建立相应索引。
4. 需要强一致性校验时，在服务层和导入流程中增加显式校验。

## 7. 冗余字段策略
1. 高频展示字段允许冗余，如用户名称、部门名称、状态快照。
2. 冗余字段必须注明来源字段和同步策略。
3. 涉及金额、状态、主键标识的关键字段不建议无约束冗余。

## 8. 状态字段设计
1. 状态字段统一使用稳定编码，如 `status`、`audit_status`、`task_status`。
2. 状态值必须配套枚举文档。
3. 禁止同一字段混杂流程状态、启停状态、删除状态。

## 9. 金额字段设计
1. 涉及真实结算、账务、收费的金额默认使用最小货币单位整数存储，如分。
2. 字段命名建议为 `amount_cent`、`paid_amount_cent`。
3. 仅展示型或比例类小数可使用明确精度的十进制类型（如 `DECIMAL(18,4)` 或等效类型）。
4. 禁止使用浮点类型存储金额。

## 10. 时间字段设计
1. 时间统一使用日期时间类型（如 `DATETIME`、`TIMESTAMP` 等）。
2. 字段命名采用 `*_time`，如 `login_time`、`expire_time`。
3. 日期字段可单独使用日期类型，但必须有明确业务语义。
4. 时区策略在应用层统一，不在数据库层混乱处理。

## 11. 枚举字段设计
1. 枚举字段使用稳定编码字符串或整型编码，但全项目需统一。
2. 当状态值会频繁新增扩展时，优先字符串编码，便于可读性与兼容性。
3. 若使用字典表，仍需在应用层限制合法值。
4. 状态、类型、来源、角色、场景等稳定编码字段，必须先判断是否应进入数据字典治理，不得默认散落在业务表注释和页面硬编码中。
5. 国际标准、国家标准、行业标准类编码体系，优先识别为标准参考字典，不与普通业务字典混表。

## 12. 索引设计规范
1. 所有高频筛选、排序、关联字段必须评估索引。
2. 组合索引遵循最左前缀原则。
3. 索引为查询服务，不为"感觉可能有用"而滥建。
4. 大文本、低区分度字段慎建索引。
5. 建表时必须写出预期查询场景。

## 13. 大字段使用规范
1. 长文本字段使用大文本类型（如 `TEXT`、`LONGTEXT` 或等效类型），如备注历史、原始报文。
2. 大字段不应进入高频列表查询。
3. 附件、图片、二进制内容默认不存数据库，存对象存储或文件服务。

## 14. 典型表设计建议
### 14.1 关联表
1. 命名为 `主对象_从对象_rel` 或 `rel_业务1_业务2`。
2. 字段至少包含双方 ID、创建时间。
3. 高频关系表需建立唯一索引防重。

### 14.2 字典表
1. 包含字典类型、字典编码、字典标签、排序、状态。
2. 保留可扩展字段但不得泛滥。

### 14.3 标准参考字典表
1. 用于承载国家码、币种码、行业分类等标准数据。
2. 必须保留标准编码、名称、状态，以及必要的层级或版本字段。
3. 不允许租户自由新增或修改标准项。

### 14.4 配置表
1. 区分系统级配置与业务级配置。
2. 配置项需有 key、value、描述、作用域、生效状态。

### 14.5 日志表
1. 区分操作日志、登录日志、任务日志、接口日志。
2. 日志表优先写入效率与检索字段，不使用逻辑删除。

### 14.6 流水表
1. 状态不可回写覆盖历史，应保留关键快照。
2. 高并发流水需考虑分表或冷热分层，但项目级另行定义。

## 15. 建表 SQL 模板
```sql
CREATE TABLE `biz_order` (
  `id` BIGINT NOT NULL COMMENT '主键ID',
  `order_no` VARCHAR(64) NOT NULL COMMENT '订单号',
  `customer_id` BIGINT NOT NULL COMMENT '客户ID',
  `status` VARCHAR(32) NOT NULL COMMENT '订单状态',
  `amount_cent` BIGINT NOT NULL DEFAULT 0 COMMENT '订单金额(分)',
  `created_by` BIGINT NOT NULL DEFAULT 0 COMMENT '创建人ID',
  `created_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `updated_by` BIGINT NOT NULL DEFAULT 0 COMMENT '更新人ID',
  `updated_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  `is_deleted` TINYINT(1) NOT NULL DEFAULT 0 COMMENT '逻辑删除标记',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_biz_order_order_no` (`order_no`),
  KEY `idx_biz_order_customer_id` (`customer_id`),
  KEY `idx_biz_order_status_created_time` (`status`, `created_time`)
) COMMENT='订单表';
```

说明：以上为参考示例 SQL。引擎、字符集等配置按项目已声明数据库选择调整。

## 16. 禁止事项
1. 使用保留字做表名或字段名。
2. 大量 `NULL` 语义不清的字段堆在同一表。
3. 将不同生命周期的业务对象粗暴合表。
4. 未评估查询场景就上生产表结构。
5. 把 JSON 字段当万能扩展机制滥用。

## 17. 评审检查清单
1. 表是否对应清晰业务实体。
2. 字段命名是否统一。
3. 是否包含审计字段。
4. 主键、唯一约束、索引是否合理。
5. 逻辑删除是否适用。
6. 金额、时间、状态设计是否符合规范。
7. 是否遗漏应做业务字典或标准参考字典的字段分层。

## 18. 设计稿输出格式要求
数据库设计稿在正式落库前，必须先提供可审阅的结构化文档，不得只给 SQL 或只给字段名列表。

强制要求：
1. 每一批设计稿必须明确写出：版本、状态、更新时间、适用范围、是否已落库。
2. 每张表必须使用统一字段明细表输出，至少包含：`字段`、`类型`、`非空`、`默认值`、`说明`。
3. 说明列必须使用中文，写清业务含义、枚举值范围、唯一性或租户隔离语义。
4. 设计稿必须单独写明：表范围、设计说明、关联关系、索引与约束建议、待审阅点。
5. 已执行落库的设计稿，文档状态必须及时从"待执行/待审阅"更新为"已执行落库"。
6. 后续增量批次的设计稿格式必须与首批保持一致，禁止因赶进度退化成非结构化草稿。
