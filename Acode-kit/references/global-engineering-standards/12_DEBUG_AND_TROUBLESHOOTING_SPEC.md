# 文件名
`12_DEBUG_AND_TROUBLESHOOTING_SPEC.md`

# 文件定位
问题排查、缺陷定位与修复验证规范。

# 适用范围
适用于开发环境、测试环境、预发环境、生产环境的问题定位与缺陷闭环处理。

# 与其他文件的关系
本文件与 `11_TESTING_AND_QA_SPEC.md`、`17_OBSERVABILITY_SPEC.md`、`16_SECURITY_SPEC.md` 配合，用于标准化故障处理流程。

# 编写目的
建立统一的问题定位、根因分析、修复验证与缺陷闭环机制，减少盲修、漏修与重复故障。

## 1. Bug 定位总流程
1. 复现问题。
2. 确认影响范围。
3. 收集日志、请求、数据、环境信息。
4. 归类到前端、后端、SQL、缓存、配置、第三方依赖。
5. 做最小复现与根因分析。
6. 修复并回归验证。
7. 记录结论与预防措施。

## 2. 前端问题排查方法
1. 确认设计期望与实际表现差异。
2. 检查浏览器控制台错误、网络请求、路由参数、权限状态。
3. 检查请求参数是否正确、接口返回是否符合契约。
4. 检查组件状态、表单模型、缓存状态是否一致。
5. 优先排查最近改动与公共组件变更。
6. 若页面文字异常，必须同时检查浏览器本地缓存、接口响应正文和数据库原始值，区分“渲染问题”和“坏数据残留”。

## 3. 后端问题排查方法
1. 根据 `traceId` 关联请求链路。
2. 查看统一异常日志、业务日志、审计日志。
3. 检查参数校验、权限判断、事务边界、下游调用。
4. 对比预期状态流转与实际数据。
5. 对热点问题优先查看 Redis、数据库锁、重试任务。
6. 若接口缓慢或登录失败，必须优先核对数据库连接配置和连接池日志，避免把配置问题误判为业务问题。

## 4. SQL 问题排查方法
1. 还原完整 SQL 与入参。
2. 查看执行计划。
3. 检查索引命中、分页方式、排序字段、逻辑删除条件。
4. 区分数据问题、SQL 写法问题、连接池问题。
5. 数据修复必须有脚本、备份、审批与回滚方案。

## 5. 缓存问题排查方法
1. 确认缓存 Key 是否正确。
2. 确认 TTL、是否被提前删除、是否存在多实例不一致。
3. 检查数据库数据与缓存数据是否一致。
4. 检查是否发生穿透、击穿、雪崩或锁竞争。

## 6. 接口问题排查方法
1. 核对 URI、Method、Header、Token、Body。
2. 核对返回码、错误码、字段结构。
3. 核对接口版本与环境配置。
4. 复用接口测试样例做最小调用。
5. 对浏览器入口接口，必须补查 `OPTIONS` 预检请求，不得只验证主请求。

## 7. 日志定位方法
1. 所有排查优先找 `traceId`。
2. 按时间窗口、用户、资源 ID 缩小范围。
3. 区分业务日志与系统日志。
4. 生产环境禁止随意打开过量 debug 日志，必要时走临时变更流程。

## 8. 常见问题记录模板
```md

- 回归结果：
- 预防措施：
```

## 9. 修复后验证流程
1. 验证问题本身已消除。
2. 验证关联功能无回归。
3. 验证日志、审计、监控未受破坏。
4. 验证测试环境与生产环境配置差异不会导致二次失败。
5. 对编码或配置类问题，必须补“旧缓存场景”和“真实浏览器场景”的回归证据。

## 10. 缺陷闭环流程
1. 缺陷录入。
2. 优先级判定。
3. 指派负责人。
4. 复现与定位。
5. 修复与自测。
6. 回归与关闭。
7. 复盘与知识库沉淀。

## 11. 禁止事项
1. 未复现就盲改代码。
2. 在生产环境直接手工改数据且无记录。
3. 只修表象，不分析根因。
4. 缺陷关闭但无回归证据。

## 本文件使用建议
将本文件作为缺陷处理工单模板、生产事故处理 SOP 和值班排障手册的基础版本。

## AI 调用建议
AI 在协助定位问题时，应先按本文件收集环境、日志、复现步骤与数据证据，再输出根因分析和修复建议。

## 追加增强：企业级排障深化要求

### 12. 排障证据链要求
1. 排障必须形成“复现步骤 + 日志 + traceId + 数据证据 + 环境信息”的证据链。
2. 没有证据链的修复结论，不得视为闭环。

### 13. 高风险排障场景
1. 涉及 Migration、权限、租户隔离、回调、导出、缓存一致性的问题必须同时检查上下游影响。
2. 生产环境问题优先确认影响范围和可回退性，再进入修复操作。

### 14. 复盘附加清单
1. 是否明确根因，而非只修表象。
2. 是否补了回归点和预防措施。
3. 是否更新了相关规范、Runbook 或监控规则。
