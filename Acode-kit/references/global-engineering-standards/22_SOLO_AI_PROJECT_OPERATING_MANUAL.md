# 文件名
`22_SOLO_AI_PROJECT_OPERATING_MANUAL.md`

# 文件定位
单人 + AI 项目从需求到上线的最小运行手册。

# 适用范围
适用于一个人借助 AI 完成需求整理、设计落地、前后端开发、测试、部署、上线与迭代的完整流程。

# 与其他文件的关系
本文件以 `00_GLOBAL_ENGINEERING_PRINCIPLES.md`、`15_AI_COLLABORATION_PLAYBOOK.md` 为总纲，串联需求、设计、开发、测试、部署阶段的最小执行闭环。

# 编写目的
把多人项目中的复杂流程压缩成适合单人 + AI 的最小高质量执行方式，保证节奏稳定、阶段清晰、输出可追踪。

## 1. 适用原则
1. 只保留真正影响交付质量的最小流程。
2. 每个阶段必须有明确输入、输出和退出条件。
3. AI 负责加速，单人负责确认关键边界。

## 2. 单人项目最小阶段
1. 需求整理。
2. 页面与信息结构确认。
3. 数据模型与 API 设计。
4. 前后端实现。
5. 自测与联调。
6. 部署与上线。
7. 发布后观察与迭代。

## 3. 每阶段最低产出物
### 3.1 需求整理
1. PRD 草案。
2. 本期范围。
3. 本期不做清单。
4. 待确认假设。

### 3.2 页面与信息结构确认
1. 设计稿（使用已声明设计工具）或线框。
2. 页面清单。
3. 页面状态清单。

### 3.3 数据模型与 API 设计
1. 主要表结构草案。
2. API 清单。
3. 权限点清单。
4. 缓存使用点清单（若使用缓存）。

### 3.4 前后端实现
1. 目录结构。
2. 核心页面、接口、表、缓存实现。
3. 关键日志与异常处理。

### 3.5 自测与联调
1. Must 功能验收映射。
2. 冒烟测试结果。
3. 问题清单。

### 3.6 部署与上线
1. 部署文件（容器化配置或平台部署配置）。
2. 环境变量清单。
3. 回滚方案。
4. 上线检查清单。

## 4. 阶段进入条件
### 4.1 Definition of Ready
进入开发前至少满足：
1. 需求目标明确。
2. 本期不做清单明确。
3. 核心页面或流程已确认。
4. 核心表和 API 有草案。
5. 待确认假设已单独列出。

### 4.2 Definition of Done
一个需求项完成至少满足：
1. 代码实现完成。
2. 核心正常流程通过。
3. 关键异常流程可控。
4. 对应文档已更新。
5. 已纳入追踪矩阵。

## 5. 每次 AI 开工前的固定动作
1. 读取 `README.md` 与 `00_GLOBAL_ENGINEERING_PRINCIPLES.md`。
2. 按任务阶段读取对应专题文档。
3. 声明本次引用了哪些规范。
4. 列出待确认项。
5. 再开始生成或修改内容。

## 6. 每次任务结束的固定动作
1. 更新 Session Handoff。
2. 更新范围记录或决策记录。
3. 更新追踪矩阵。
4. 若涉及上线，更新上线检查清单。
5. 若当前控制文档开始堆积长历史，应主动做“当前有效信息收敛 + 历史归档”，不要让主文档持续膨胀。

## 7. 推荐工作节奏
1. 先定范围，再做首条核心路径。
2. 先打通骨架，再补完整状态与异常。
3. 先完成 Must，再考虑增强项。
4. 每次只推进一个清晰的小闭环。

## 8. 单人 + AI 常见风险与防法
1. 风险：需求边做边变。
   处理：强制维护范围记录与本期不做清单。
2. 风险：AI 输出太快导致架构漂移。
   处理：每次任务先声明引用规范。
3. 风险：做完功能没法上线。
   处理：在开发中提前维护部署与验收清单。
4. 风险：隔天接着做时忘了上下文。
   处理：每次结束必须写 Session Handoff。
5. 风险：主文档被旧历史淹没，导致 AI 反复读取过时内容。
   处理：把当前有效决策、当前交接和历史归档拆开维护。

## 9. 最小项目运行模板
```md
## 当前阶段
- 阶段：前后端实现
- 当前目标：完成订单审核主路径

## 本轮 Must
- REQ-ORDER-003 审核订单
- REQ-ORDER-004 审核记录查询

## 本轮不做
- 批量审核
- 审核撤回

## 待确认
- 是否需要二级审核
```

## 本文件使用建议
把本文件作为单人 + AI 项目的日常操作手册，每个迭代或每次开发会话都按其中的最小流程执行。

## AI 调用建议
AI 在复杂任务开始前和结束后，都应按本文件执行“读取规范、声明范围、更新 handoff、更新追踪”的固定动作。

## 后续可扩展点
可增加按日迭代模板、周复盘模板、单人项目知识库模板。

## 追加增强：企业级单人 + AI 运行要求

### 10. 最小治理不是最低标准
1. 单人 + AI 项目可以压缩流程，但不能压掉质量门禁。
2. 需求、设计、接口、数据库、测试、上线与回滚仍然必须有最小事实文档和验证证据。

### 11. 风险前置要求
1. 涉及数据库 migration、权限、安全、租户隔离、外部回调、正式上线时，必须前置到阶段文档中处理，不得留到最后补救。
2. 单人项目更要依赖文档和门禁，而不是依赖“自己记得住”。

### 12. 评审附加清单
1. 当前阶段输入、输出、退出条件是否清楚。
2. 是否存在跨阶段跳步实现。
3. 是否有足够 handoff、路线图和上线记录支撑中断后恢复。
