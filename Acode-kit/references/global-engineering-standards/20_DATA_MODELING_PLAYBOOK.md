# 文件名
`20_DATA_MODELING_PLAYBOOK.md`

# 文件定位
从业务对象到数据表、状态、字典、日志、流水的建模操作手册。

# 适用范围
适用于各类项目的业务建模、数据库拆解、状态设计与关联关系梳理。

# 与其他文件的关系
本文件是 `06_DATABASE_DESIGN_SPEC.md` 的方法论补充，也为 `05_API_DESIGN_SPEC.md` 和 `07_REDIS_CACHE_SPEC.md` 提供建模输入。

# 编写目的
提供从业务对象到数据结构的操作型建模方法，帮助团队和 AI 在落表之前先把实体、状态、关系拆清楚。

## 1. 建模步骤
1. 识别业务对象。
2. 识别角色与动作。
3. 识别状态与状态流转。
4. 识别对象间关系。
5. 识别审计、日志、配置、字典需求。
6. 输出表结构草案与接口字段草案。
7. 对状态、类型、来源、角色、场景、标准编码字段执行字典分层判断。

## 2. 业务对象识别法
1. 看名词：用户、订单、客户、项目、任务、审批单。
2. 看关键编号：单号、编码、账号、流水号。
3. 看生命周期：创建、编辑、提交、审核、完成、关闭。

## 3. 状态机设计法
1. 一张主业务表至少明确主状态字段。
2. 状态流转必须可解释、可审计、可拒绝非法跳转。
3. 启停状态与流程状态分开建模。

示例：
```md
订单主状态：DRAFT -> PENDING_APPROVAL -> APPROVED -> COMPLETED
订单启停状态：ENABLED / DISABLED
```

## 4. 关系设计法
1. 一对一：拆表价值不高时可合表，否则独立表。
2. 一对多：主表 + 明细表。
3. 多对多：关系表。
4. 字典引用：字典编码或稳定枚举值。
5. 标准参考字典引用：标准编码或标准编码 + 名称快照。

## 5. 常见表类型拆分
1. 主数据表：承载业务对象当前主状态。
2. 明细表：承载组成项。
3. 关系表：承载多对多关系。
4. 操作日志表：承载行为轨迹。
5. 状态历史表：承载流程历史。
6. 配置表：承载规则、模板、阈值。
7. 字典表：承载稳定映射关系。
8. 标准参考字典表：承载国家、币种、行业分类等标准数据。

## 6. 从 PRD 到表结构的映射模板
```md
业务对象：
- 用户
- 角色
- 菜单
- 订单

关键动作：
- 创建订单
- 提交审核
- 审核通过/驳回

核心表：
- biz_order
- biz_order_item
- biz_order_audit_log
```

## 7. 建模常见误区
1. 一个状态字段承载所有语义。
2. 把所有扩展信息塞 JSON。
3. 把快照需求、日志需求遗漏到后期补丁。
4. 用数据库字段名直接驱动页面文案。
5. 把标准参考字典和普通业务字典混为一谈。

## 8. 与缓存联动考虑
1. 热点主数据是否需要缓存。
2. 字典与配置是否需要缓存。
3. 状态变更后缓存如何失效。
4. 幂等控制是否需要缓存辅助。

## 9. 建模输出物清单
1. 业务对象清单。
2. 状态机说明。
3. 表结构草案。
4. 索引建议。
5. API 字段清单。
6. 审计与日志策略。
7. 字典识别清单与字典分层结论。

## 本文件使用建议
在开始写 SQL 之前先用本文件把业务对象、状态、关系、日志和配置拆清楚，再进入表结构设计。

## AI 调用建议
AI 在收到业务场景时，可先按本文件输出数据建模草案，再生成具体 SQL、DTO、VO 和缓存策略。

## 追加增强：企业级建模深化要求

### 10. 建模粒度控制
1. 建模不是只列实体名，而要冻结对象、动作、状态、关系、日志、配置和字典边界。
2. 涉及审批流、计费、订单、任务、导入导出等复杂模块时，必须单独建状态机和历史轨迹模型。

### 11. 多租户与作用域建模
1. 平台级对象、租户级对象、组织级对象、用户级对象必须分开建模。
2. 配置、字典、模板、权限、日志都要明确作用域，而不是只在业务主表补 `tenant_id`。

### 12. 评审附加清单
1. 是否遗漏日志、审计、历史状态、字典和配置模型。
2. 是否把多个生命周期不同的对象粗暴合并。
3. 是否提前考虑了 API、缓存、权限、租户和报表消费方式。
