# 文件名
`23_SCOPE_CONTROL_AND_DECISION_LOG_SPEC.md`

# 文件定位
范围控制、关键决策、假设与变更记录规范。

# 适用范围
适用于单人 + AI 项目在需求变化、技术调整、版本切分、关键假设确认过程中的记录与控制。

# 与其他文件的关系
本文件与 `01_PRODUCT_REQUIREMENTS_STANDARD.md`、`15_AI_COLLABORATION_PLAYBOOK.md`、`22_SOLO_AI_PROJECT_OPERATING_MANUAL.md` 联动，用于防止项目范围漂移。

# 编写目的
用最轻量的方式记录“本期做什么、为什么这样做、什么不做、改过什么”，避免单人 + AI 项目在快速推进中失控。

## 1. 必须记录的事项
1. 本期目标。
2. 本期范围。
3. 本期不做。
4. 关键技术决策。
5. 关键业务假设。
6. 范围变更记录。

## 2. 适用原则
1. 记录要轻，不写长篇报告。
2. 只记录会影响后续实现和上线判断的事项。
3. 每次发生范围或关键方案变化时立即更新。

## 3. 关键决策记录模板
```md
## DEC-001
- 标题：订单金额统一使用分存储
- 日期：YYYY-MM-DD
- 背景：
- 决策：
- 原因：
- 影响范围：
- 替代方案：
- 是否已确认：是/否
```

## 4. 范围变更记录模板
```md
## CR-001
- 变更内容：
- 来源：
- 变更原因：
- 影响页面：
- 影响接口：
- 影响表结构：
- 影响测试：
- 是否进入本期：
- 处理结论：
```

## 5. 假设记录模板
```md
## ASM-001
- 假设内容：
- 当前默认处理：
- 若假设不成立的影响：
- 最晚确认时间：
```

## 6. 单人项目范围控制规则
1. 任何新增需求都必须先判断是否进入当前版本。
2. 进入当前版本前必须评估对页面、API、数据库、测试、上线时间的影响。
3. 若影响核心闭环且未确认，不应立即编码。
4. 临时想法先记录，不直接转化为开发动作。

## 7. 何时必须更新本文件
1. 新增 Must 功能。
2. 删除已有 Must 功能。
3. 改动核心业务规则。
4. 改动数据模型或接口契约。
5. 修改上线范围。

## 8. 最小维护形式
建议一个项目只维护一个持续追加的记录文件，按 `DEC`、`CR`、`ASM` 三类编号追加，不拆成多个散乱文件。

## 本文件使用建议
每次 AI 或人工提出新想法、要改规则、要扩范围时，先写一条记录，再决定是否实施。

## AI 调用建议
AI 在发现需求变化、关键技术选项变化、重要假设时，不应直接执行，而应先按本文件格式产出记录并请求确认。

## 后续可扩展点
可增加版本范围模板、轻量 ADR 模板、风险台账模板。
