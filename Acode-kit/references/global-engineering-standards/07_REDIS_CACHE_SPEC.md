# 文件名
`07_REDIS_CACHE_SPEC.md`

# 文件定位
缓存与会话存储规范。

# 适用范围
适用于登录态、验证码、权限缓存、热点数据、分布式锁、幂等控制、限流等缓存场景。缓存技术由 `PROJECT_OVERRIDES.md` 确定，本文件以 Redis 作为参考实现，原则适用于所有缓存方案。

# 与其他文件的关系
本文件依赖 `04_BACKEND_ARCHITECTURE_SPEC.md` 与 `16_SECURITY_SPEC.md`，并与 `05_API_DESIGN_SPEC.md` 的 Token、幂等规则保持一致。

# 编写目的
统一缓存的使用边界、Key 规范、TTL 策略与缓存一致性策略，避免缓存系统失控或与主数据职责混淆。

## 1. 缓存的标准使用场景
1. 登录态会话缓存。
2. Token 黑名单或续期控制。
3. 验证码与短信码。
4. 限流计数。
5. 热点数据缓存。
6. 字典缓存与配置缓存。
7. 分布式锁。
8. 幂等控制。
9. 异步任务进度缓存。

## 2. Key 命名规范
统一格式：
```text
{project}:{module}:{biz}:{identifier}
```

示例：
```text
saas:auth:login:user:1001
saas:auth:captcha:login:13800000000
saas:sys:dict:country-code
saas:order:idempotent:create:request-uuid
saas:lock:order:approve:12345
```

规则：
1. 全部使用小写与冒号分隔。
2. Key 中必须体现模块和业务语义。
3. 避免只写 UUID 造成无法排查。

## 3. TTL 设计规范
1. 验证码：2 至 5 分钟。
2. 登录会话：与 Token 生命周期一致或略长。
3. 黑名单：至少覆盖 Token 剩余有效期。
4. 限流窗口：按规则设定，如 1 秒、1 分钟、1 小时。
5. 热点缓存：根据业务新鲜度设置，一般 5 分钟到 1 小时。
6. 分布式锁：必须短 TTL，并配合续期或幂等设计。

## 4. 登录态与 Token 规范
1. Token 本身可无状态，但登录态控制、强退、单点登录、黑名单建议使用缓存存储。
2. 用户登出时应清理登录态缓存，并在必要时加入黑名单。
3. Token 续期策略必须统一，不得有的模块续期、有的模块不续期。

## 5. 验证码规范
1. Key 必须包含用途，如登录、注册、重置密码。
2. 同一手机号或邮箱应有发送频率限制。
3. 使用后应立即失效。
4. 错误次数过多要触发额外风控。

## 6. 限流规范
1. 限流对象可按 IP、用户 ID、接口、租户维度。
2. 限流 Key 命名需体现动作和窗口。
3. 命中限流时返回明确错误码与提示。

## 7. 热点数据缓存规范
1. 字典、地区、配置、低频变化列表可优先缓存。
2. 缓存值结构应与业务对象解耦，避免直接缓存数据库实体。
3. 必须定义更新策略和失效策略。

## 8. 分布式锁规范
1. 锁 Key 需精确到资源粒度。
2. 锁必须设置过期时间。
3. 使用安全释放机制，避免误删他人锁。
4. 锁只用于保护短事务，不用于掩盖架构问题。

## 9. 幂等控制规范
1. 创建类接口、任务触发类接口、支付确认类接口优先使用缓存做短期幂等控制。
2. 幂等 Key 由业务唯一标识或 `Idempotency-Key` 构成。
3. 缓存只做短期去重，强一致唯一性仍需数据库兜底。

## 10. 缓存一致性策略
1. 优先使用"更新数据库后删除缓存"策略。
2. 对高一致性场景，可使用延迟双删。
3. 多级缓存场景必须明确主次关系与刷新顺序。
4. 读多写少场景可采用 Cache Aside。

## 11. 缓存穿透、击穿、雪崩处理
1. 穿透：对不存在对象缓存空值，或使用布隆过滤器。
2. 击穿：热点 Key 设置互斥重建或逻辑过期。
3. 雪崩：TTL 增加随机扰动，避免同一时间大面积失效。

## 12. 删除缓存与更新缓存规范
1. 默认删除缓存，不直接覆盖缓存，除非有明确版本控制与序列保证。
2. 删除缓存失败时需记录日志并允许补偿。
3. 批量变更时要考虑批量 Key 删除成本，必要时采用版本前缀策略。

## 13. 数据结构使用建议
若使用 Redis 或类似支持多数据结构的缓存系统：
1. `String`：验证码、Token、JSON 缓存。
2. `Hash`：对象级缓存、会话属性。
3. `Set`：去重、黑名单、权限集合。
4. `ZSet`：排行榜、延时任务、按时间排序队列。

若使用简单键值缓存（如 Memcached、内存缓存），以字符串序列化为主，结合业务场景选择适当的序列化格式。

## 14. 禁止事项
1. 不设 TTL 存放临时业务数据。
2. 用缓存持久保存唯一主业务数据。
3. 使用无业务语义的混乱 Key。
4. 锁 Key、验证码 Key、幂等 Key 混用命名。
5. 缓存数据库整表全量结果且无失效策略。

## 15. 评审检查清单
1. 是否说明使用缓存的业务目标。
2. Key 命名是否可定位。
3. TTL 是否合理。
4. 是否考虑一致性与异常处理。
5. 是否防止穿透、击穿、雪崩。

## 本文件使用建议
新增缓存、锁、限流、会话设计时必须先评估是否符合本文件的场景与命名、TTL、失效策略。

## AI 调用建议
AI 在生成缓存 key、登录态方案、幂等实现、缓存更新逻辑时，必须按本文件生成可审计、可排查、可失效的方案。
