# 文件名
`18_ENVIRONMENT_CONFIG_SPEC.md`

# 文件定位
多环境配置分层、配置注入与敏感配置治理规范。

# 适用范围
适用于本地、测试、预发、生产环境的前端、后端、辅助服务和部署编排配置管理。

# 与其他文件的关系
本文件为 `13_DEPLOYMENT_AND_DEVOPS_SPEC.md` 与 `14_CICD_SPEC.md` 提供配置治理基础，并受 `16_SECURITY_SPEC.md` 约束。

# 编写目的
统一多环境配置分层、敏感配置隔离、注入方式与变更流程，避免环境漂移和密钥泄露。

## 1. 配置治理原则
1. 配置与代码分离。
2. 环境差异通过配置表达，不通过分支长期分叉表达。
3. 敏感配置与普通配置分层管理。
4. 配置变更必须可追踪、可回滚。

## 2. 环境分层
1. `local`：开发者本地环境。
2. `dev`：共享开发环境。
3. `test`：测试联调环境。
4. `staging`：预发布环境。
5. `prod`：生产环境。

## 3. 配置分类
1. 基础配置：服务名、端口、日志级别。
2. 环境配置：域名、数据库地址、缓存地址（若使用）、对象存储地址。
3. 敏感配置：密码、密钥、证书、第三方凭证。
4. 功能开关：灰度开关、调试开关、实验特性开关。
5. 编码配置：数据库存储字符集、客户端导入字符集、驱动连接字符集、HTTP 响应字符集。

## 4. 命名规范
1. 环境变量使用 `UPPER_SNAKE_CASE`。
2. 前端公开变量使用已声明前端框架的公共变量前缀（如 `VITE_`、`NEXT_PUBLIC_`、`REACT_APP_` 等）。
3. 同一语义跨服务命名保持一致，如 `DB_HOST`、`CACHE_HOST`、`AUTH_SECRET`。

## 5. 注入方式
1. 本地环境可用 `.env.local`、IDE Run Config。
2. 服务器与容器环境通过 Secret、环境变量、配置中心注入。
3. 严禁把生产 `.env` 文件提交到仓库。

## 6. 配置文件组织建议
```text
deploy/
  env/
    .env.example
    .env.dev
    .env.test
```

规则：
1. 仓库内只保存示例配置和非敏感模板。
2. 实际敏感值通过部署系统注入。

## 7. 配置变更流程
1. 提出变更申请。
2. 评估影响范围。
3. 在测试或预发验证。
4. 审批后执行生产变更。
5. 留存变更记录与回滚参数。

## 8. 数据库与字符集配置专项规则
1. 数据库存储字符集、命令行客户端字符集、驱动连接字符集参数必须分开治理，禁止混为一个概念。
2. 数据库 seed、初始化脚本、批量导入脚本必须显式指定客户端字符集。
3. 数据库连接 URL 或驱动配置中的字符集参数必须使用当前驱动实际支持的取值，不得机械复制数据库排序规则或表字符集名称。
4. 任何字符集相关配置改动后，必须验证：
   - 应用能正常启动建连
   - 中文数据入库正确
   - 接口返回值正确
   - 浏览器页面最终显示正确
5. 当前后端分离开发场景下，前端公开地址、后端 API 地址、CORS 放行范围必须作为环境配置显式治理，不得依赖默认值碰运气。

## 9. 功能开关规范
1. 所有开关必须有名称、默认值、作用范围、负责人、失效时间。
2. 临时开关不得长期沉积。
3. 高风险功能优先通过开关控制上线。

## 10. 配置自检清单
1. 是否区分敏感与非敏感配置。
2. 是否有示例配置模板。
3. 是否有生效环境说明。
4. 是否能回滚。
5. 是否区分数据库字符集、导入字符集、驱动字符集。
6. 是否对本地浏览器跨域预检做过验证。

## 本文件使用建议
新建服务、增加第三方依赖、切换环境或发版前配置变更时，都应先对照本文件治理配置项。

## AI 调用建议
AI 在生成 `.env.example`、配置类、compose 环境变量、部署说明时，应按本文件区分敏感配置与普通配置，不得输出真实密钥。
