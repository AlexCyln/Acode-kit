# 文件名
`25_ACCEPTANCE_AND_GO_LIVE_CHECKLIST.md`

# 文件定位
单人项目验收、上线前检查、发布后观察的清单式规范。

# 适用范围
适用于单人 + AI 项目从“准备上线”到“上线完成并稳定运行”的最后一道检查环节。

# 与其他文件的关系
本文件依赖 `11_TESTING_AND_QA_SPEC.md`、`13_DEPLOYMENT_AND_DEVOPS_SPEC.md`、`17_OBSERVABILITY_SPEC.md`，是上线前的最终收口文档。

# 编写目的
把复杂的验收和上线流程压缩成可执行清单，防止单人项目在最后阶段因为遗漏小项导致上线失败或回滚。

## 1. 使用原则
1. 上线前逐项检查，不靠记忆。
2. 所有 Must 项必须有明确状态。
3. 检查失败时优先修复或延期，不带模糊状态上线。

## 2. 上线前验收清单
1. PRD 中 Must 功能全部完成。
2. 追踪矩阵中 Must 项全部有验收状态。
3. 冒烟测试通过。
4. 核心异常流程已验证。
5. 权限与菜单已核对。
6. 环境变量已确认。
7. 数据库迁移脚本已确认。
8. 回滚方案已准备。

## 3. 技术检查清单
1. 前端构建成功。
2. 后端构建成功。
3. Python 服务依赖和启动命令已验证。
4. Docker 镜像可构建。
5. compose 配置可启动。
6. 健康检查接口可访问。
7. 日志输出路径正常。

## 4. 数据检查清单
1. 初始化字典和配置已准备。
2. 测试数据已清理或隔离。
3. 生产必要基础数据已导入。
4. Redis 关键 key 前缀已确认。
5. 数据备份已完成。

## 5. 发布执行清单
1. 记录当前线上版本。
2. 执行数据库迁移。
3. 发布后端。
4. 发布 Python 服务。
5. 发布前端。
6. 验证健康检查。
7. 执行最小冒烟。

## 6. 发布后观察清单
1. 登录是否正常。
2. 主路径是否正常。
3. 核心接口错误率是否异常。
4. 日志是否连续报错。
5. Redis、数据库是否出现异常连接或慢查询。
6. 第三方回调或任务是否正常。

## 7. 回滚判断清单
触发以下任一项应立即评估回滚：
1. 登录失败或核心入口不可用。
2. 核心数据写入错误。
3. 核心业务主链路无法完成。
4. 错误率短时间显著升高。
5. 第三方集成关键路径失效且无绕行方案。

## 8. 最小记录模板
```md
## 上线记录
- 版本号：
- 发布时间：
- 发布内容：
- 数据库脚本：
- 回滚目标版本：
- 冒烟结果：
- 观察期结果：
- 是否完成：
```

## 本文件使用建议
每次上线都按此清单逐项勾选，并保留上线记录；不要依赖临场记忆。

## AI 调用建议
AI 在协助上线准备、发布说明、冒烟检查时，应按本文件输出逐项清单，而不是泛泛描述“检查一下环境和日志”。

## 后续可扩展点
可增加按环境的上线模板、停机/不停机发布模板、生产事故后检查模板。

## 追加增强：企业级上线验收要求

### 9. 高风险上线附加检查
1. 涉及 Migration、权限模型、租户模式、缓存策略、回调链路变更的上线，必须增加专项检查项。
2. 生产发布前应确认版本、配置、Migration、观测、回滚方案一致。

### 10. 观察期治理
1. 上线完成不等于交付完成，观察期内必须持续关注错误率、慢查询、任务积压、权限异常和第三方回调情况。
2. 若观察期指标异常，即使主路径暂时可用，也应评估回滚或止损。

### 11. 评审附加清单
1. 是否有明确上线负责人、回滚负责人、观察期负责人。
2. 是否有核心链路 smoke 与失败链路 smoke。
3. 是否能说明上线后 30 分钟和 24 小时内的重点观察项。
