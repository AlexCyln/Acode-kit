# 文件名
`30_DEVELOPMENT_DOCUMENTATION_GOVERNANCE_SPEC.md`

# 文件定位
开发过程文档治理专项规范。

# 适用范围
适用于使用本套母规范推进的单人 + AI 或小团队软件项目，覆盖需求落地后的设计、开发、联调、测试、上线准备阶段。

# 与其他文件的关系
- 以 `00_GLOBAL_ENGINEERING_PRINCIPLES.md` 为总纲。
- 与 `05_API_DESIGN_SPEC.md`、`06_DATABASE_DESIGN_SPEC.md`、`11_TESTING_AND_QA_SPEC.md`、`15_AI_COLLABORATION_PLAYBOOK.md`、`27_PROJECT_EXECUTION_FLOW_SPEC.md` 联动使用。
- 本文件关注“开发过程文档必须存在并持续更新”的治理要求，不替代任何专题文档本身。

# 编写目的
确保接口文档、数据库文档、功能函数文档、测试文档在项目推进过程中持续存在、持续更新、可查可交接，避免只依赖临时会话说明或零散提交信息。

## 1. 基本原则
1. 文档不是收尾动作，而是开发过程的一部分。
2. 任何已实现的关键能力，都应能在对应详细文档中找到口径、边界与验证结论。
3. AI 和研发在开始实现前，必须先检查目标子任务对应的详细文档是否已存在。
4. 若缺少对应文档入口，先补 `md`，再进入实现。
5. 对于新的大模块，若仍存在较大范围不确定性，必须先补模块级正式审阅稿，经过确认后再进入实现。
6. 对于模块下的子模块 / 系统 / 页面，在执行前必须先补轻量“核心执行说明”，经过确认后再进入实现。
7. 对于需要页面承载的节点，模块级 UI 设计审阅稿、页面核心执行说明与前端复刻/还原记录都属于开发过程文档治理范围。
8. 启动阶段形成的每一份项目级文档，都必须先以稳定文件名固化为文件，不得只存在于对话或临时草稿里。
9. `Step 4` 的职责是将已冻结启动文件归位到正式目录，而不是重新生成一份内容更弱的正式文档。

## 2. 必须维护的开发过程文档
每个项目在 `docs/project/` 下，至少应持续维护以下四类文档：

### 2.1 接口文档
- 适用触发条件：
  - 出现真实接口
  - 已定义前后端联调契约
  - 接口字段、权限、缓存、状态口径发生变化
- 最低要求：
  - 路径、方法、用途
  - 请求头、参数、请求体
  - 响应结构与示例
  - 权限要求
  - 错误口径
  - 缓存 / 幂等 / 联调说明

### 2.2 数据库文档
- 适用触发条件：
  - 新增建表设计
  - 变更字段、索引、约束
  - 执行增量迁移
- 最低要求：
  - 表关系
  - 字段、类型、默认值、中文说明
  - 状态值 / 字典关系
  - 执行状态：待审阅 / 待执行 / 已执行落库

### 2.3 功能函数文档
- 适用触发条件：
  - 新增关键业务模块
  - 新增公共服务、状态机、校验逻辑、核心函数
  - 关键实现规则发生变化
- 最低要求：
  - 模块职责
  - 关键入口文件
  - 核心函数 / 服务
  - 依赖的表、缓存、接口
  - 关键规则与边界
  - 已知限制

### 2.4 测试文档
- 适用触发条件：
  - 执行构建验证
  - 执行接口 smoke
  - 执行写路径联调
  - 执行关键手工验收
- 最低要求：
  - 时间
  - 范围
  - 执行方式 / 命令
  - 结果
  - 异常
  - 结论

### 2.5 模块级正式审阅稿
- 适用触发条件：
  - 新的大模块正式启动
  - 模块实现范式、边界、输入输出、验证口径需要先冻结
- 类型收敛：
  - 需求变更审阅稿
  - 数据库表设计审阅稿
  - 接口设计审阅稿
  - UI 设计审阅稿
  - 测试用例审阅稿
- 治理规则：
  - 正式审阅稿只在模块级产出，不在每个子模块 / 系统 / 页面重复铺开
  - 五类正式审阅稿按需存在，不强制模块机械凑满
  - 历史页面级 / 子模块级审阅稿默认作为历史参考保留，不要求批量重写

### 2.6 核心执行说明
- 适用触发条件：
  - 模块下的子模块 / 系统 / 页面准备开始执行
  - 本次执行需要在实现前冻结范围和边界
- 最低要求：
  - 本次目标
  - 输入依据
  - 实施范围
  - 明确不做
  - 执行步骤
  - 待确认点

### 2.7 UI 设计与复刻记录
- 适用触发条件：
  - 页面/模块需要先做 UI 设计
  - 页面/模块已进入 Pencil 设计或前端复刻
- 最低要求：
  - 模块级 UI 设计审阅稿
  - 当前页面核心执行说明
  - Pencil 设计确认结论
  - 前端一比一复刻范围与验收结论

## 3. 文档存在性检查规则
1. 每次开始新子任务前，先检查是否存在对应文档。
2. 若不存在，则至少新增一个可持续维护的 `md` 入口文档。
3. 新增文档后，应立即纳入项目文档索引或目录总表。
4. 不允许出现“代码已经实现，但无法定位详细文档”的情况。

## 3.3 启动期文件固化规则
1. 从 `Step 2` 开始，所有启动期产物必须先写入 `.acode-kit-startup/`。
2. 启动期文件名必须稳定、可预测、可被脚本引用，不得使用 `draft-final-v2`、`new-prd` 等临时命名。
3. 最低要求的启动期文件包括：
   - `PROJECT_SKELETON.approved.md`
   - `PROJECT_OVERVIEW.seed.md`
   - `PRD.approved.md`
   - `PROGRESS_PLAN.approved.md`
   - `TRACEABILITY_MATRIX.seed.md`
   - `DECISION_LOG.seed.md`
   - `STACK_AND_DIRECTORY_INPUTS.approved.md`
4. Gate 审批通过后，应更新对应启动期文件的状态与版本，而不是另起一份无关联文件。
5. `Step 4` 创建正式目录时，只允许移动、重命名、补元信息、建立索引引用；不得凭记忆重写正文。

## 3.1 文档分层与归档规则
1. 项目级文档应至少分为：
   - 当前有效总控文档
   - 当前执行中的模块级正式审阅稿
   - 当前执行中的核心执行说明
   - 已审阅通过但当前不执行的参考资料
   - 接口 / 数据库 / 功能函数 / 测试等详细实现文档
   - 历史归档
2. 当前执行中的模块级正式审阅稿、核心执行说明必须与已审阅参考资料显式分开登记。
3. 已结束且当前不再主导执行的审阅稿、核心执行说明、旧交接、被替换的长历史记录，应迁入归档区。
4. 总控文档只保留当前有效信息，不得长期混入完整历史流水。

## 3.2 更新节奏要求
1. 每次 gate、stage、模块 step 或重要验收节点确认后，必须立即更新对应总控文档，不得拖到任务结束再统一回填。
2. `PROJECT_OVERVIEW.md` 记录当前阶段、当前目标、当前进度和工具状态。
3. `TRACEABILITY_MATRIX.md` 记录模块状态、slice 状态、实施进度和验收状态。
4. `SESSION_HANDOFF.md` 记录当前节点、实施信息、下一步和阻塞项。
5. `DECISION_LOG.md` 记录决策、变更和假设。
6. 若接口、数据库、功能函数或测试事实发生变化，必须同步更新对应详细文档入口，再允许推进后续步骤。

## 4. 重要节点的强制更新规则
以下节点发生时，必须同步更新对应详细文档：
1. 接口首次真实可用。
2. 表结构首次落库或发生增量迁移。
3. 核心函数 / 状态机 / 校验规则发生变化。
4. 关键 smoke / 联调 / 构建验证执行完成。
5. 影响后续查阅、联调、排障、交接的重要问题被修复。
6. 模块级正式审阅稿确认通过、页面核心执行说明确认通过、Pencil 设计确认通过、前端一比一复刻确认通过。

## 5. 项目建议文档清单
项目级除总控文档外，建议至少有：
1. `DEVELOPMENT_DOCUMENTATION_INDEX.md`：开发文档索引
2. `API_DETAIL_CATALOG.md`：接口详细文档
3. `FUNCTION_MODULE_RECORD.md`：功能函数与模块记录
4. `TEST_EXECUTION_RECORD.md`：测试与验证记录

## 5.1 启动期文件与正式文件映射
1. `.acode-kit-startup/PROJECT_SKELETON.approved.md` -> `docs/project/PROJECT_SKELETON.md`
2. `.acode-kit-startup/PROJECT_OVERVIEW.seed.md` -> `docs/project/PROJECT_OVERVIEW.md`
3. `.acode-kit-startup/PRD.approved.md` -> `docs/project/PRD.md`
4. `.acode-kit-startup/TRACEABILITY_MATRIX.seed.md` -> `docs/project/TRACEABILITY_MATRIX.md`
5. `.acode-kit-startup/DECISION_LOG.seed.md` -> `docs/project/DECISION_LOG.md`
6. `.acode-kit-startup/STACK_AND_DIRECTORY_INPUTS.approved.md` -> `docs/project/DIRECTORY_PLAN.md` 的主输入之一
7. 进度计划可作为 `PROJECT_OVERVIEW.md`、`TRACEABILITY_MATRIX.md`、`SESSION_HANDOFF.md` 的承接内容来源，或在项目确有需要时落为单独计划文档。

## 6. AI 执行要求
1. AI 在每轮任务结束前，必须检查本轮是否影响接口文档、数据库文档、功能函数文档、测试文档。
2. 若影响而未更新，应优先补文档，再结束任务。
3. `DECISION_LOG`、`TRACEABILITY_MATRIX`、`SESSION_HANDOFF` 只能作为总控文档，不能替代详细过程文档。
4. 在启动阶段，AI 必须优先更新 `.acode-kit-startup/` 中对应文件，而不是把关键信息仅留在会话总结里。
5. 在 `Step 4`，AI 必须说明每个正式文件来自哪一份启动冻结稿，确保来源可追溯。
6. 在 `Step 4`，AI 必须先输出 `DIRECTORY_PLAN.md`，再创建目录和初始化依赖。

## 7. 验收标准
当项目进入联调、提测、上线准备阶段时，应满足：
1. 已实现接口可在详细接口文档中查到。
2. 已执行数据库变更可在数据库文档中查到。
3. 关键业务模块和核心函数可在功能函数文档中查到。
4. 已执行验证动作可在测试文档中查到。

## 8. 反例
1. 只有总控文档，没有详细接口文档。
2. 只有 SQL，没有结构化数据库说明文档。
3. 关键状态机只存在代码里，没有任何文字说明。
4. 测试只说“已验证”，但没有记录范围、方式和结论。

## 9. 使用建议
将本文件作为项目进入“真实开发”后的文档治理基线。任何长期推进、多人接手或多轮 AI 会话项目，都应强制启用本规范。

## 10. 企业级深化要求
1. 开发过程文档必须视为正式交付资产，而不是“如果有空再补”的辅助说明。
2. 文档更新应紧贴事实发生点，优先在接口冻结、数据变更、测试完成、决策落定后立即更新。
3. 对外部可见行为、关键数据口径、关键流程规则的变更，必须能在对应文档中快速定位前后差异。
4. 项目推进中若长期只更新代码不更新文档，应视为治理失效并在阶段验收中拦截。

## 11. 文档质量要求
1. 文档必须可被后来者独立阅读理解，不得只写“见代码”“同前”“已完成”这类无信息量描述。
2. 文档应尽量结构化，至少明确背景、范围、输入、输出、边界、异常、结论。
3. 关键文档需要给出文件路径、接口路径、表名、命令、截图或日志摘要等可核对信息。
4. 已失效内容应明确标注失效时间、替代入口或迁移去向，避免旧文档和新文档同时主导执行。

## 12. 交接与审计要求
1. 任意阶段中断后，接手人应能仅凭仓库文档恢复当前目标、当前状态、未决事项和下一步动作。
2. 关键决策、关键变更、关键验证必须可以从文档索引追溯到详细说明，不得只留在聊天记录中。
3. 若使用 AI 生成或维护文档，仍需保证结论可验证、引用可定位、与实际代码和配置一致。
4. 发布前应抽查文档与实现一致性，重点核对 API、DB、测试记录、上线记录和决策日志。
5. 启动期冻结稿与正式项目文档之间必须存在清晰映射关系，避免 `Step 4` 之后无法追溯正式文档来源。
