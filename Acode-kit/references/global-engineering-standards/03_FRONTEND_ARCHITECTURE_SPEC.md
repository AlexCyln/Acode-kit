# 文件名
`03_FRONTEND_ARCHITECTURE_SPEC.md`

# 文件定位
前端工程架构与实现规范。

# 适用范围
适用于所有基于已声明前端框架和 UI 组件库的前端项目。具体技术栈由 `PROJECT_OVERRIDES.md` 确定。

# 与其他文件的关系
本文件承接 `02_UI_UX_DESIGN_SPEC.md` 的页面交付结果，并依赖 `05_API_DESIGN_SPEC.md`、`08_CODE_STYLE_AND_NAMING_SPEC.md`、`16_SECURITY_SPEC.md`。

# 编写目的
统一前端工程结构、职责边界与页面实现范式，保证项目在长期迭代中仍然可维护、可扩展、可协作。

## 1. 架构目标
1. 页面层职责清晰。
2. 业务逻辑不散落在视图层。
3. 状态、请求、权限、表单、表格能力可复用。
4. 便于多人协作、长期迭代、模块拆分。

## 2. 推荐目录结构
```text
src/
  api/
    modules/
    request/
  assets/
  components/
    base/
    business/
  hooks-or-composables/
  constants/
  enums/
  layouts/
  router/
    modules/
  stores/
    modules/
  styles/
  types/
  utils/
  views/
    system/
    business/
```

说明：`hooks-or-composables/` 目录名称按已声明框架惯例选择（如 React 使用 `hooks/`，Vue 使用 `composables/`）。

## 3. 各层职责边界
1. `views/`：页面容器，负责页面编排、路由参数承接、页面级状态组合。
2. `components/base/`：纯通用组件，不包含具体业务语义。
3. `components/business/`：可跨页面复用的业务组件，如用户选择器、状态标签、审批记录面板。
4. `hooks-or-composables/`：复用逻辑，负责表单处理、列表查询、权限判断、上传、分页等。
5. `api/`：接口定义与请求封装，不处理页面展示逻辑。
6. `stores/`：跨页面共享状态，如用户信息、权限、标签页、字典缓存。
7. `utils/`：纯工具函数，不依赖页面上下文。

## 4. 页面开发范式
1. 列表页按"筛选条件 + 操作区 + 表格 + 分页"模板实现。
2. 详情页按"摘要区 + 分区详情 + 操作区 + 日志区"模板实现。
3. 新增编辑页按"表单模型 + 校验规则 + 保存提交逻辑"模板实现。
4. 审核页按"业务对象 + 审核表单 + 审核动作 + 历史记录"模板实现。
5. 若页面已有经过确认的设计稿（使用已声明设计工具），前端应以该设计稿为一比一复刻基准，不得私自增加、删除或改写模块、字段、按钮、状态、标签、标题与关键交互。

## 5. 路由设计规范
1. 路由路径使用 kebab-case，如 `/user-center/account-list`。
2. 路由名称使用 PascalCase 或语义常量统一维护。
3. 路由元信息至少包含 `title`、`permission`、`icon`、`hidden`、`keepAlive`。
4. 页面标题与菜单标题默认同步，特殊情况单独声明。

## 6. 权限路由与菜单联动
1. 菜单由后端返回时，前端根据权限和路由白名单进行过滤与挂载。
2. 权限码应与按钮、页面、接口统一命名。
3. 路由守卫负责登录校验、权限拦截、动态路由注入。
4. 无权限页面统一进入 403 页面或授权提示，不允许白屏。

## 7. 状态管理规范
1. 状态管理库只存跨页面共享状态与少量页面会话态。
2. 页面内一次性临时状态优先使用框架原生响应式 API，不进入全局 store。
3. store 需按模块拆分，禁止单个巨型 store 承载全项目状态。
4. store 中不得直接写视图渲染逻辑。

## 8. 请求封装规范
1. 统一封装 HTTP 客户端实例，处理 `baseURL`、超时、Token、错误拦截、重复请求控制。
2. API 函数应只暴露语义化方法，如 `getUserList`、`createOrder`。
3. 所有请求返回值需做类型声明。
4. 文件上传、导出下载、流式响应应单独封装处理。

## 9. 表单开发规范
1. 表单模型、校验规则、提交逻辑分离。
2. 每个表单项必须明确字段类型、默认值、必填规则。
3. 字典枚举统一从枚举中心或字典接口获取，不在页面硬编码。
4. 编辑页回填与提交转换逻辑必须独立封装。
5. 标准参考字典与业务字典应在前端状态层分开缓存与管理，不混用同一松散映射对象。

## 10. 表格开发规范
1. 表格列定义应集中管理，避免模板内散落。
2. 查询条件、分页参数、排序参数统一收口。
3. 操作列权限判断不写死在模板中，应通过权限能力函数控制。
4. 导出操作必须区分当前页导出与条件全量导出。

## 11. 错误处理规范
1. 页面级错误使用统一消息提示与错误态组件。
2. 表单提交失败应保留用户输入。
3. 登录失效统一清理本地状态并跳转登录页。
4. 不可恢复错误要记录日志并给出可读提示。

## 12. 文件上传规范
1. 限制格式、大小、数量。
2. 上传前进行前端校验，后端二次校验。
3. 上传组件需支持进度、失败重试、删除回滚。
4. 文件 URL、文件名、业务关联 ID 应明确区分。

## 13. 国际化预留规范
1. 默认项目可先中文实现，但文本不可写死到难以替换的位置。
2. 全局状态、枚举、按钮、菜单应具备可迁移到 i18n 资源文件的能力。
3. 日期、金额、数字展示格式使用统一工具。
4. 字典展示名与标准参考名称应具备中英双语切换能力，页面不直接写死中文枚举标签。

## 14. 性能与可维护性规范
1. 路由级懒加载。
2. 大表格、大表单、复杂弹窗按模块拆分。
3. 高频字典、权限、基础配置可做前端缓存。
4. 禁止超大单文件页面，单个组件文件超过 400 行必须评估拆分。
5. 前端复刻/还原应基于设计稿页面结构做组件化拆分，但拆分只影响代码组织，不得改变页面最终视觉与交互结果。

## 15. 明确禁止事项
1. 业务逻辑直接写满模板区域。
2. 页面直接拼接原始请求 URL。
3. 同一业务规则在多个页面重复实现。
4. 将菜单、权限、路由映射关系散落各处。
5. 为追求快速交付跳过类型声明与错误处理。
6. 在未回到页面审阅稿或设计稿确认的情况下，擅自增删页面字段、按钮、状态、标签或文案。
7. 以"工程方便"为理由明显偏离已确认的设计稿视觉与交互。

## 16. 页面骨架模式说明

页面骨架应遵循以下模式（伪代码，具体语法按已声明前端框架调整）：

```
页面容器组件 {
  复用逻辑钩子（查询表单、加载状态、表格数据、分页、搜索、重置、创建）

  渲染：
    筛选表单组件（绑定查询表单）
    工具栏组件（绑定创建动作）
    表格组件（绑定加载、数据、分页）
}
```

关键原则：
1. 页面组件只做编排，不包含具体业务逻辑。
2. 业务逻辑抽取到复用钩子/composable 中。
3. 子组件按职责拆分（筛选、工具栏、表格）。

## 本文件使用建议
前端项目初始化、模块扩展、页面重构时均以本文件作为目录与职责边界基准。

## AI 调用建议
AI 在生成前端页面、状态管理、路由配置、API 模块、通用复用逻辑前，必须先按本文件划分职责，不得把所有逻辑塞进单页组件。
