# 文件名
`14_CICD_SPEC.md`

# 文件定位
持续集成与持续交付流水线规范。

# 适用范围
适用于代码检查、构建、测试、镜像构建、部署、回滚等自动化流程设计。

# 与其他文件的关系
本文件依赖 `09_GIT_WORKFLOW_AND_COMMIT_SPEC.md`、`11_TESTING_AND_QA_SPEC.md`、`13_DEPLOYMENT_AND_DEVOPS_SPEC.md`。

# 编写目的
统一持续集成与持续交付的最小闭环能力，让代码检查、测试、构建、镜像、部署和回滚过程标准化。

## 1. CI/CD 的作用
1. 让代码质量检查制度化。
2. 让构建、测试、打包、发布标准化。
3. 降低人工漏项与环境差异。
4. 为回滚、审计、追溯提供依据。

## 2. 最低可行流水线
1. 代码拉取。
2. 依赖安装与缓存。
3. Lint / 静态检查。
4. 单元测试与必要接口测试。
5. 构建产物。
6. 构建镜像。
7. 推送镜像。
8. 部署到目标环境。
9. 执行部署后健康检查。

## 3. 分支与环境联动建议
1. `feature/*`：执行基础 CI，不自动部署。
2. `develop`：执行完整 CI，可自动部署到 `test`。
3. `release/*`：执行完整 CI，可部署到 `staging`，建议保留人工审批。
4. `main/master`：执行生产发布链路，必须有审批与回滚准备。

## 4. 自动发布与手工审批边界
可自动执行：
1. 代码检查。
2. 单元测试。
3. 构建。
4. 镜像打包。
5. 测试环境部署。

必须人工审批：
1. 生产环境数据库变更。
2. 生产环境部署。
3. 安全策略调整。
4. 重大版本切换与回滚。

## 5. 失败处理与回滚建议
1. 任一步骤失败应中断流水线。
2. 保留失败日志与构建产物元数据。
3. 部署后健康检查失败时，自动或人工触发回滚。
4. 回滚后需重新验证关键链路。

## 6. 工具链保持抽象
可选工具示例：
1. GitHub Actions。
2. GitLab CI。
3. Jenkins。

要求：
1. 不绑定单一平台语法。
2. 所有关键步骤都应能映射到上述平台中的等价能力。

## 7. 流水线阶段建议
### 7.1 前端
1. 安装依赖。
2. 类型检查（若使用 TypeScript 或类型系统）。
3. Lint。
4. 单元测试。
5. 使用已声明构建工具进行构建。
6. 构建部署镜像或产物。

### 7.2 后端主服务
1. 依赖缓存与安装。
2. 编译（若为编译型语言）。
3. 单元测试。
4. 打包产物。
5. 构建运行镜像。

### 7.3 辅助服务
1. 安装依赖。
2. Lint。
3. 单元测试。
4. 构建镜像。

## 8. 数据库变更策略
1. DDL 脚本需进入版本管理。
2. 数据库迁移应可追踪。
3. 高风险 DDL 必须走审批。
4. 部署流水线中需显式标记数据库变更步骤。

## 9. 流水线元数据要求
1. Commit SHA。
2. 构建号。
3. 镜像 Tag。
4. 发布环境。
5. 审批记录。
6. 回滚目标版本。

## 10. 流水线检查清单
1. 是否执行代码检查。
2. 是否执行测试。
3. 是否产出可追溯版本号。
4. 是否有审批节点。
5. 是否有回滚路径。

## 本文件使用建议
CI/CD 平台初始化、流水线重构、发布治理制度落地时应首先引用本文件确定最小闭环能力。

## AI 调用建议
AI 在生成流水线草案、发布脚本、环境联动说明时，应以本文件为抽象规范，而不是默认绑定某个具体平台。

## 后续可扩展点
可增加质量门禁阈值、代码覆盖率策略、制品仓库规范、部署审批矩阵。

## 11. 企业级流水线分层要求
1. 流水线至少区分校验、构建、制品、部署、发布后验证五类阶段，不得把所有步骤堆成一个不可治理的大 job。
2. 校验阶段应优先失败，尽量在低成本阶段拦截格式、静态检查、单元测试、契约检查失败。
3. 制品阶段必须确保同一 Commit 产出可追溯、不可变、可重复部署的构建结果。
4. 部署阶段必须区分“部署动作成功”和“业务可用成功”，健康检查与 smoke 验证不可省略。
5. 生产发布链路必须支持人工审批、灰度、回滚或紧急止损中的至少一种受控能力。

## 12. 质量门禁细化
1. 未通过 Lint、静态分析、单元测试、核心契约测试的变更不得进入后续构建阶段。
2. 涉及数据库 migration、权限、安全、计费、导出、审计的高风险变更，必须增加专项校验或人工审批节点。
3. 覆盖率阈值可按项目声明，但关键模块不得因“整体覆盖率尚可”而绕过新增测试要求。
4. 构建成功不等于可发布；若缺少配置校验、镜像扫描、部署后验证、回滚预案，则发布链路视为不完整。

## 13. 制品与环境一致性要求
1. 制品命名必须包含可追溯版本信息，如 Commit SHA、构建号、Tag 或语义版本号。
2. 同一构建产物应在测试、预发、生产环境复用，不得在每个环境重新本地构建不同产物。
3. 环境差异应通过配置注入解决，而不是通过不同代码分支、不同构建脚本或手工修改产物解决。
4. 镜像、安装包、静态制品、部署清单都应有保存周期和溯源记录。

## 14. 评审附加清单
1. 流水线是否真正覆盖了从代码到上线的最小闭环，而不是只做构建。
2. 高风险变更是否具备显式审批、验证和回滚节点。
3. 是否存在仅依赖人工发布经验、无法自动复现的关键步骤。
4. 制品、环境配置和部署记录是否能支持事故追溯。
5. 部署后验证是否覆盖关键登录、关键读路径、关键写路径或关键后台任务。
