# 文件名
`27_PROJECT_EXECUTION_FLOW_SPEC.md`

# 文件定位
具体项目在单人 + AI 模式下从零到一落地的统一执行流程范式。

# 适用范围
适用于基于本母规范体系启动的新项目、重构项目、模块化迭代项目，尤其适用于一个人借助 Codex/AI 完成项目全流程的场景。

# 与其他文件的关系
本文件以 `22_SOLO_AI_PROJECT_OPERATING_MANUAL.md` 为项目运行基线，串联 `01`、`02`、`03`、`04`、`05`、`06`、`11`、`13`、`15`、`24`、`25` 等关键文档，是“如何使用整套母规范推进具体项目”的执行总表。

# 编写目的
把母规范从“规则集合”提升为“可执行工作流”，让 AI 在具体项目中按固定步骤初始化文档、明确需求、搭建架构、编码、测试、部署并持续维护项目状态。

## 1. 总体原则
1. 母规范定义统一约束，项目级文档定义当前项目事实。
2. AI 每次只读取“常驻底座 + 当前阶段最小必要文档”。
3. 项目推进按小闭环进行，不一次性铺开全量功能。
4. 每次任务结束都必须把结果写回项目级文档。
5. 每次进入新的开发子任务前，都必须检查接口文档、数据库文档、功能函数文档、测试文档是否已存在；缺失时先补文档入口，再继续实现。
6. `PRD.md` 作为项目最高级需求边界文档，若新需求与 PRD 冲突，必须先显式提示并由人工裁决，禁止默认覆盖。
7. `TRACEABILITY_MATRIX.md` 作为项目主任务路线图与总体进度追踪文档，负责把握项目总体路线、当前主任务和下一阶段，不得退化成仅记录本轮任务的会话文档。
8. 每个主任务下一个节点都必须保留“先确认再执行”的门禁；模块启动前使用模块级正式审阅稿冻结边界，子模块 / 系统 / 页面执行前使用轻量“核心执行说明”确认后再实现。
9. 若节点需要页面承载，则页面执行前仍需先确认本次页面的核心执行说明，再完成该节点的 `UI 设计 -> 前端复刻/还原`，再继续后端接入或后续实现。
10. 项目文档必须显式区分“当前有效总控文档、当前执行中的模块级正式审阅稿、当前执行中的核心执行说明、已审阅参考资料、详细实现文档、历史归档”，避免当前执行资料与长历史混堆。

## 2. 项目启动输入
启动一个具体项目时，至少提供：
1. 项目总体需求或业务目标。
2. 本期目标或 MVP 目标。
3. 是否已有现成仓库或需从零初始化。
4. 是否已有设计稿、旧系统、接口、数据库参考。
5. 是否存在第三方集成。

## 3. AI 启动后必须先做的事
1. 读取 `README.md`、`00_GLOBAL_ENGINEERING_PRINCIPLES.md`、`15_AI_COLLABORATION_PLAYBOOK.md`、`22_SOLO_AI_PROJECT_OPERATING_MANUAL.md`。
2. 判断当前是“新项目初始化”“已有项目续建”还是“局部模块迭代”。
3. 建立或确认项目级文档目录。
4. 初始化项目级核心文档。
5. 列出待确认项与当前假设。

## 4. 项目级核心文档最小集合
建议至少维护以下文件：
1. `PROJECT_OVERVIEW.md`
2. `PROJECT_OVERRIDES.md`
3. `PRD.md`
4. `DECISION_LOG.md`
5. `TRACEABILITY_MATRIX.md`
6. `SESSION_HANDOFF.md`
7. `GO_LIVE_RECORD.md`

## 5. 标准执行阶段
### 5.1 阶段一：项目初始化
目标：
1. 明确项目目标、范围、技术边界。
2. 建立项目级文档骨架。

必须产出：
1. `PROJECT_OVERVIEW.md`
2. `PROJECT_OVERRIDES.md`
3. `SESSION_HANDOFF.md`

通过条件：
1. 项目目标明确。
2. 本期范围明确。
3. 技术栈已声明并记录于 `PROJECT_OVERRIDES.md`。

### 5.2 阶段二：需求结构化
目标：
1. 从总体需求整理出 PRD。
2. 区分 Must / Should / Could。
3. 明确本期不做与待确认假设。
4. 建立项目最高级需求边界，并明确后续冲突处理规则。

必须产出：
1. `PRD.md`
2. `DECISION_LOG.md` 初版

通过条件：
1. MVP 明确。
2. 本期不做清单明确。
3. 核心业务流程明确。
4. PRD 已可作为后续设计与实现的最高级边界输入。

### 5.3 阶段三：页面与交互结构化
目标：
1. 明确页面清单、页面层级、状态清单。
2. 如有 Pencil 设计稿，建立页面映射；如无则先产出结构稿。

必须产出：
1. 页面清单
2. 页面与需求映射
3. 状态清单

通过条件：
1. 核心页面完整。
2. 核心状态齐全。

### 5.4 阶段四：数据与接口设计
目标：
1. 明确主要表结构、状态机、接口清单、权限点、缓存点位（若使用缓存）。
2. 建立项目总体路线图和主任务推进基线。

必须产出：
1. API 清单
2. 表结构草案
3. 权限点清单
4. `TRACEABILITY_MATRIX.md` 初版

通过条件：
1. 核心接口和核心表可支撑 MVP。
2. 关键状态流转明确。
3. `TRACEABILITY_MATRIX.md` 已能表达项目主任务路线、模块阶段和当前主任务，而非仅停留在轻量映射。

### 启动期 Step 3 补充冻结要求
在进入项目框架初始化前，必须额外冻结一份“实现与目录输入包”，至少涵盖：
1. 需求边界与模块范围
2. 页面 / 交互 / 设计方向
3. 前端运行时、路由、UI 系统、样式、测试方案
4. 后端运行时、应用分层、API 风格、安全边界
5. 数据库类型、Migration 策略、数据访问方式、关键表与状态流
6. 部署方式、环境配置入口、容器化或平台交付要求
7. 第三方集成、设计资产、工具和目录约束
8. 已激活场景包与技术栈包对目录结构的专业要求

### 5.5 阶段五：项目框架初始化
目标：
1. 先完成目录结构与项目级文档正式归位。
2. 再初始化前端、后端、辅助服务（若有）、部署骨架。
3. 建立基础配置、基础依赖。
4. 先整合最终目录方案，再执行目录创建与文件归位。

必须产出：
1. 仓库目录结构
2. 已从 `.acode-kit-startup/` 直接归位的项目级控制文档
3. 前后端骨架
4. 部署基础文件（容器化配置或平台部署配置）
5. `DIRECTORY_PLAN.md`

通过条件：
1. `Step 4a` 已按 `DIRECTORY_PLAN.md` 完成目录创建与启动冻结稿直接归位。
2. `Step 4b` 已完成环境与工程骨架初始化。
3. 基础项目可启动。
4. 目录与已声明技术栈符合母规范。
5. 目录计划已经说明母规范固定目录、场景增强目录、技术栈目录片段和冲突处理结果。

### 5.6 阶段六：按小闭环实现功能
目标：
1. 以“一个需求项或一组紧密需求”为单位逐步实现。

每个闭环至少包含：
1. 页面或页面区块
2. API
3. 表结构或数据处理
4. 权限与异常处理
5. TDD 驱动的测试（红-绿-重构循环）与追踪矩阵更新

通过条件：
1. 该闭环对应需求完成。
2. 追踪矩阵更新。
3. Session Handoff 更新。

### 5.7 阶段七：Review、测试、Debug
目标：
1. 逐闭环 Review。
2. 补齐测试、联调、排障。

必须产出：
1. 测试结果
2. 缺陷清单
3. 修复记录

通过条件：
1. Must 项核心路径可用。
2. 冒烟路径通过。

### 5.8 阶段八：部署与上线
目标：
1. 准备发布。
2. 逐项执行上线检查与观察。

必须产出：
1. 环境变量清单
2. 发布记录
3. 回滚方案
4. `GO_LIVE_RECORD.md`

通过条件：
1. 上线清单全部通过。
2. 发布后观察项正常。

## 6. AI 每次任务的固定工作方式
每次 AI 接到具体任务时，按以下顺序：
1. 确认当前项目阶段。
2. 读取常驻底座文档。
3. 读取当前阶段最小必要规范。
4. 读取项目级事实文档。
5. 先对照 PRD 和项目路线图判断该任务是否属于当前主任务、是否存在冲突或越界。
6. 若当前进入新的大模块，先输出模块级正式审阅稿；若进入该模块下的子模块 / 系统 / 页面，则先输出本次轻量“核心执行说明”，而不是直接进入整包实现。
7. 若该节点需要页面承载，则在本次页面核心执行说明确认后，先向用户收集本次页面的设计风格、交互、细节要求；若与全局要求冲突，先由用户裁决。
8. 使用 Pencil 或等价设计稿工具完成页面设计，并确保页面内容严格来自模块级 UI 设计审阅稿、当前核心执行说明与真实 API / 数据库字段。
9. 在设计稿确认前，必须新增一次 Pencil 工具级页面校验核对环节，检查布局、排版、尺寸、页面边界、模块间距、内边距、文本边界、裁切、重叠、遮挡等问题，并记录发现与修复结果。
10. 设计稿经用户确认后，再进入前端一比一复刻/还原。
11. 若本轮属于新页面首次提审，则在交付用户人工复核前必须先执行真实浏览器自动预验收；默认级别为：单个页面 `L1`、关联页面组 `L2`、完整大模块 `L3`。若 AI 判断需升降级，必须先获用户同意。
12. 同一页面后续若仅为小范围静态调整，且不涉及交互、接口、前后端联动或大规模布局改动，可不重复执行真实浏览器自动预验收。
13. 只完成当前最小目标。
14. 更新 `TRACEABILITY_MATRIX.md`、`DECISION_LOG.md`、`SESSION_HANDOFF.md` 中应更新的部分。
15. 若 `SESSION_HANDOFF.md`、`DECISION_LOG.md` 或其他总控文档开始堆积长历史，应优先做收敛与归档，再继续推进实施。

## 6.1 节点合同原则
1. PRD 在 Gate 3 确认后，后续主链路必须保持固定顺序推进，任何新需求只能通过变更通道进入，不得直接插队到实现节点。
2. 每个节点只能有一个主输出：阶段一输出模块路线，阶段二输出整体 UI 架构，阶段三输出数据/API 框架，阶段四输出项目骨架，阶段五输出模块闭环，阶段六输出集成结果，阶段七输出上线结果。
3. 每个节点在退出前必须明确写回对应总控文档，尤其是 `TRACEABILITY_MATRIX.md`、`SESSION_HANDOFF.md`、`DECISION_LOG.md` 和上线相关记录。
4. 需要设计的节点只产出设计与审阅，不提前进入实现；需要实现的节点只产出当前闭环实现，不外溢到下一模块。
5. 任何节点若无法清楚说明输入、输出、约束、文档更新目标和下一个审批点，应视为未满足执行条件。

## 6.2 变更与回滚通道
1. 当变更影响范围小于等于 30% 模块时，只回到受影响的最近批准节点，并同步更新 `PRD.md`、`TRACEABILITY_MATRIX.md`、`DECISION_LOG.md` 与 `SESSION_HANDOFF.md`。
2. 当变更影响范围大于 30% 模块时，必须先做影响分析，重新输出变更骨架，再由人工确认后继续；不得沿用旧 PRD 直接硬改。
3. 回滚分为三类：实现前回滚、实现中回滚、上线后回滚；三类回滚都必须记录触发原因、回退点、恢复后的验证结果。
4. 上线后回滚必须同时写入 `GO_LIVE_RECORD.md` 和 `DECISION_LOG.md`，并暂停后续扩展直到复盘完成。

## 6.3 TDD 总线与分线执行模型
1. 总线 TDD：每个阶段输出都必须是可验证的交付物，至少包含可审阅、可判定、可回退的验证点。
2. 分线 TDD：每个 5d 实现 slice 必须先写失败测试，再实现，再重构，不得先写生产代码后补测试。
3. 需求变更、缺陷修复和回滚相关节点也要遵循测试先行思路，至少先形成失败复现、验证点或回归条件，再进入修复。
4. Stage 6 和 Stage 7 的测试侧重点分别是跨模块回归与上线前验收、回滚演练与观察期验证。

## 7.1 目录整合原则
1. 目录创建前先读取已激活场景包和技术栈包中的目录片段。
2. 母规范固定治理目录优先级最高。
3. 技术栈包决定前端、后端、数据访问、部署等专业目录骨架。
4. 场景包只能增强业务模块目录与页面形态，不得推翻基础技术目录骨架。
5. 当目录片段不足时，才允许参考 `references/project-blueprints/` 作为 fallback，而不是直接临场自创。

## 7.2 模块级正式审阅稿收敛规则
1. 正式审阅稿默认只在“大模块启动前”产出，不在每个子模块 / 系统 / 页面重复产出。
2. 正式审阅稿类型收敛为以下五类：
   - 需求变更审阅稿
   - 数据库表设计审阅稿
   - 接口设计审阅稿
   - UI 设计审阅稿
   - 测试用例审阅稿
3. 五类正式审阅稿按需产出，不强制每个模块机械凑满五份；但一旦某类成为该模块实现前置输入，就必须在模块级冻结。
4. 历史页面级 / 子模块级审阅稿默认作为历史参考保留，不要求批量重写；新规则只从生效点开始执行。

## 7.3 核心执行说明规则
1. 子模块 / 系统 / 页面在真正执行前，必须提供一份轻量“核心执行说明”并获得确认。
2. 核心执行说明属于正式开发过程文档，但不是新的正式审阅稿类型。
3. 核心执行说明建议只保留最小必要内容：
   - 本次目标
   - 输入依据
   - 实施范围
   - 明确不做
   - 执行步骤
   - 待确认点
4. 核心执行说明的作用是冻结“本次要做什么”，不是重复抄写模块级正式审阅稿。

## 9. 常见错误与纠正方式
1. 错误：直接开始写代码。
   纠正：先初始化项目级文档与范围。
2. 错误：一次要求 AI 完成整个系统。
   纠正：按阶段、按闭环拆任务。
3. 错误：做完功能不更新文档。
   纠正：每次任务结束写回追踪矩阵与 handoff。
4. 错误：发现需求变化直接修改实现。
   纠正：先更新决策与范围记录。

## 追加增强：企业级执行流深化要求

### 10. 阶段门禁强化
1. 每个阶段的输出必须可审阅、可验证、可回退，不得只输出“讨论结论”。
2. Gate / Stage / Step 的核心价值在于冻结边界和降低后续返工，不得把门禁退化成形式流程。

### 11. 文档回写强化
1. 项目级总控文档、模块级审阅稿、核心执行说明、详细实现文档和归档资料必须分层维护。
2. 若阶段推进后文档仍不能说明当前主任务、当前边界和下一步，则视为流程未完成。

### 12. 高风险节点附加要求
1. Stage 3、Step 5c、Step 5d、Stage 6、Stage 7 属于高风险节点，必须额外关注 API、数据库、权限、安全、Migration、可观测性和回滚准备。
2. 涉及多租户、外部回调、导出导入、权限体系、数据修复时，必须在进入实现前先冻结关键规则。

### 13. 评审附加清单
1. 当前节点是否只有一个主输出。
2. 当前节点是否明确了输入、输出、约束、文档更新目标和审批点。
3. 是否存在节点跳步、旧版本失效未回写、或文档与实现脱节的问题。
