# 文件名
`26_EXTERNAL_INTEGRATION_SPEC.md`

# 文件定位
第三方系统与外部服务集成治理规范。

# 适用范围
适用于统一认证、短信、邮件、文件服务、AI 服务、支付、企微、钉钉、地图、ERP/OA 等外部集成场景。

# 与其他文件的关系
本文件依赖 `05_API_DESIGN_SPEC.md`、`16_SECURITY_SPEC.md`、`13_DEPLOYMENT_AND_DEVOPS_SPEC.md`，补足跨系统接入的稳定性与安全要求。

# 编写目的
规范第三方接入时的超时、重试、签名、回调、幂等、补偿和异常处理，避免“能接通但不可控”的集成实现。

## 1. 集成原则
1. 外部系统不可信，必须做超时、重试、降级和日志记录。
2. 外部回调必须验签、幂等、防重放。
3. 第三方异常不得轻易拖垮主业务流程。
4. 所有集成都应有联调与回退方案。

## 2. 集成分类
1. 同步查询型。
2. 同步提交型。
3. 异步回调型。
4. 文件交换型。
5. 定时同步型。

## 3. 最小接入清单
1. 服务名称。
2. 用途说明。
3. 调用方向。
4. 鉴权方式。
5. 超时设置。
6. 重试策略。
7. 幂等策略。
8. 失败补偿方式。
9. 联调环境信息。

## 4. 超时与重试规范
1. 所有外部调用必须显式设置超时。
2. 重试只适用于幂等操作或明确安全的查询操作。
3. 高风险提交类接口不得无脑自动重试。
4. 重试次数必须有限，并记录失败日志。

## 5. 回调接口规范
1. 必须验签。
2. 必须校验来源。
3. 必须做幂等控制。
4. 必须保留原始报文或关键摘要用于排障。
5. 回调处理失败时需明确重试策略和人工补偿策略。

## 6. 数据映射规范
1. 第三方字段与内部字段必须有映射表。
2. 不将第三方字段名直接污染内部领域模型。
3. 枚举值、状态值必须单独转换，不直接透传。

## 7. 降级与补偿
1. 查询失败时可按业务情况降级返回缓存或默认提示。
2. 提交失败时应区分“未发送成功”“发送成功但未确认”“对方已受理”。
3. 必要时提供手工补偿、重试任务或对账机制。

## 8. 联调规范
1. 必须准备联调账号、联调环境地址、示例报文。
2. 联调前明确签名算法、字段格式、时区、金额单位。
3. 联调过程中的特殊约束必须记录在项目文档中。

## 9. 日志与监控
1. 外部调用必须记录请求耗时、结果码、traceId。
2. 对敏感报文做脱敏。
3. 高风险集成必须有成功率、失败率、超时率监控。

## 10. 禁止事项
1. 不设超时直接调用第三方。
2. 回调不验签。
3. 外部失败直接吞掉不记录。
4. 用主业务线程无限等待第三方结果。
5. 未做幂等就处理重复回调。

## 11. 最小集成模板
```md
## 外部集成项
- 名称：短信服务
- 类型：同步提交型
- 鉴权方式：Access Key + 签名
- 超时：3s
- 重试：失败后最多重试 1 次
- 幂等：按 requestId 控制
- 补偿：失败进入重试任务表
```

## 本文件使用建议
只要项目出现任何外部系统接入，都应先按本文件列清集成边界和保护措施，再开始编码。

## AI 调用建议
AI 在生成第三方接入代码、回调接口、重试任务、签名校验逻辑时，必须按本文件补齐超时、幂等、验签、日志和补偿设计。

## 追加增强：企业级外部集成治理要求

### 12. 集成边界深化
1. 第三方系统永远视为不稳定边界，集成代码不得侵入核心领域流程。
2. 对同步提交、异步回调、定时同步、文件交换四类集成应分别设计超时、重试、补偿和审计策略。

### 13. 联调与上线要求
1. 联调前必须冻结字段映射、签名算法、时区、金额单位、重试规则和错误码语义。
2. 上线前必须确认第三方回调地址、白名单、证书、密钥、重试策略和人工补偿方式。

### 14. 评审附加清单
1. 是否有幂等、验签、防重放和失败补偿。
2. 是否明确记录联调环境和生产环境差异。
3. 是否能在日志和观测中定位外部调用失败原因。
