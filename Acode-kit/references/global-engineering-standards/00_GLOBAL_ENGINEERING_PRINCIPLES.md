# 文件名
`00_GLOBAL_ENGINEERING_PRINCIPLES.md`

# 文件定位
跨项目母规范的最高优先级治理文件。

# 适用范围
适用于所有基于已声明技术栈开展的需求分析、设计、研发、测试、部署、运维与 AI 协同场景。

# 与其他文件的关系
本文件是所有专题规范的总纲。后续文档不得与本文件冲突；如需项目级覆盖，必须显式声明并经过人工审批。

# 编写目的
定义跨项目统一的工程哲学、技术栈决策框架、TDD 开发方法论、规范优先级、AI 决策边界与后续专题文档的主从关系。

## 1. 全局工程哲学
1. 规范优先于实现。未明确规范前，不允许直接进入大规模编码。
2. 设计优先于开发。页面、流程、数据模型、接口边界应先定义再实现。
3. 文档优先于口头约定。影响交付、协作、扩展的规则必须沉淀为文档。
4. 可维护性优先于短期堆功能。宁可多做抽象与分层，也不允许无边界堆逻辑。
5. 可扩展性优先于一次性脚本式实现。默认考虑模块增长、角色增长、数据量增长与部署增长。
6. 可测试性优先于隐式耦合。所有关键业务链路必须具备可验证输入、输出、状态。
7. 可部署性优先于本地可跑。实现方案必须能进入容器化或平台化、可配置、可回滚的交付链路。
8. 可观测性优先于"出了问题再查"。日志、指标、告警、审计应在设计阶段预留。

## 2. 技术栈决策框架

技术栈不再全局硬编码，而是在项目初始化时由用户确定并固化到 `PROJECT_OVERRIDES.md` 的技术栈声明区块。

### 2.1 技术栈分类

每个项目必须在以下分类中做出明确声明：

| 分类 | 说明 | 示例选项 |
|------|------|----------|
| 项目类型 | 交付形态 | Web 应用、iOS 应用、Android 应用、小程序、桌面应用、跨平台移动应用 |
| 前端框架 | 视图层核心框架 | React、Vue、Angular、SwiftUI、Jetpack Compose、Flutter、UniApp 等 |
| UI 组件库 | 界面组件体系 | shadcn/ui、Element Plus、Ant Design、Material UI、自定义组件库等 |
| 后端运行时 | 服务端语言与框架 | Spring Boot、Express/Koa、Django/FastAPI、Go Gin、.NET 等 |
| ORM/数据访问 | 数据持久化框架 | MyBatis-Plus、Prisma、SQLAlchemy、GORM、TypeORM 等 |
| 数据库 | 主存储引擎 | MySQL、PostgreSQL、MongoDB、SQLite、Supabase 等 |
| 缓存 | 缓存与会话存储 | Redis、Memcached、内存缓存、无缓存等 |
| 认证方案 | 身份验证与授权 | JWT、OAuth 2.0、Session、第三方认证服务等 |
| 部署平台 | 交付与运行环境 | Docker/Compose、Kubernetes、Vercel、云函数、App Store、Google Play 等 |
| 设计工具 | UI/UX 设计来源 | Pencil、Figma、Sketch、手绘线框等 |

### 2.2 声明规则

1. 技术栈在项目初始化阶段由用户选定，记录于 `PROJECT_OVERRIDES.md`。
2. 一旦声明，该项目全生命周期内视为"已声明技术栈"。
3. 所有后续规范中的"已声明技术栈"均指向 `PROJECT_OVERRIDES.md` 中的声明。

### 2.3 明确禁止

1. 未经项目级书面覆盖，不得擅自切换已声明技术栈中的任何组件。
2. 不得绕开统一请求层、统一异常层、统一权限层、统一日志审计层直接拼接实现。
3. 不得因 AI 输出方便而降低工程治理要求。

## 2A. TDD 宪法条款

测试驱动开发（TDD）是本规范体系下的标准开发方法论。

### 2A.1 红-绿-重构循环

所有功能实现必须遵循以下循环：

1. **红**（Red）：先编写一个描述预期行为的失败测试。
2. **绿**（Green）：编写最小量的实现代码使测试通过。
3. **重构**（Refactor）：在测试保护下优化代码结构，确保测试持续通过。

### 2A.2 TDD 基本原则

1. 测试先行于实现——未编写测试前，不编写生产代码。
2. 每个垂直切片（功能闭环）以一个失败测试作为起点。
3. 测试是活文档——测试用例即功能规格说明。
4. 重构必须在绿灯状态下进行。
5. TDD 适用于所有可自动化测试的业务逻辑、API 端点、数据访问层和工具函数。
6. UI 层 TDD 可根据已声明前端框架的生态支持程度灵活执行（组件测试或 E2E 测试）。

## 3. 文档驱动开发原则
1. 需求必须先形成结构化 PRD，再进入设计与拆解。
2. UI/UX 必须先有设计稿（使用已声明设计工具）或等价线框，再进入页面开发。
3. API 必须先有接口契约，再进入前后端联调。
4. 数据库必须先有数据模型与索引设计，再落表。
5. 部署必须先有环境配置、镜像或部署包与回滚策略，再进入上线流程。

## 4. 设计先行原则
1. 页面信息架构必须先明确层级、入口、主任务路径。
2. 核心业务流程必须先明确节点、状态、操作权限、异常分支。
3. 数据结构必须先明确主实体、关联实体、状态机、审计字段。
4. UI 实现必须尊重设计稿，但允许在"工程可实现、交互一致、复用更优"的前提下进行受控调整。

## 5. 规范优先级
1. 法规、合规、组织级安全要求。
2. 本母规范目录。
3. 项目级补充规范。
4. 模块级约定。
5. 开发者个人习惯。

发生冲突时，优先级高者覆盖低者。任何覆盖都必须留下文档与审批记录。

## 6. 默认架构立场
1. 后端采用"单体优先、模块清晰、按需拆分"原则。
2. 前端采用"页面层薄、业务组件中等、通用层稳定、请求与状态分离"原则。
3. 数据层采用"主存储优先、缓存辅助、异步增效、不提前过度微服务化"原则。
4. 认证授权采用"统一身份、统一鉴权、统一审计、统一权限扩展点"原则。

## 7. AI 在项目中的角色边界
### 7.1 AI 可以自动决定的事项
1. 在已声明技术栈范围内生成标准目录结构。
2. 按现有规范生成模板代码、重复性样板、DTO/VO/Entity、CRUD 页面骨架。
3. 按接口契约生成前后端对接代码。
4. 按建模规则生成 SQL、索引建议、缓存 key 方案草案。
5. 按测试规范生成测试用例、Mock 数据、联调清单、发布检查清单。

### 7.2 AI 必须人工确认的事项
1. 需求范围变化。
2. 核心业务规则、审批规则、计费规则、权限规则。
3. 跨系统集成方式。
4. 安全等级、脱敏策略、审计范围。
5. 技术栈变更、部署拓扑变更、数据库重大重构。
6. 对外 API 暴露策略、版本废弃策略、生产环境变更。

### 7.3 AI 明确不得擅自决定的事项
1. 更换已声明技术栈。
2. 删改关键日志、审计、权限、校验、测试环节。
3. 以"为了跑通"为由跳过异常处理、边界处理、数据校验。
4. 在需求、设计、接口、数据模型缺失时自造业务规则并直接落地。

## 8. 人工确认门禁
以下事项必须经过产品、架构、研发负责人或指定审批人确认：
1. 新增一级业务模块。
2. 调整核心领域模型。
3. 涉及金额、权限、流程审批、数据删除、数据导出。
4. 影响多环境配置、部署编排、外部网络访问、第三方依赖。

## 10. 标准交付流程
1. 需求澄清。
2. PRD 定稿。
3. 信息架构与 UI 设计。
4. 数据模型与 API 设计。
5. 前后端任务拆解。
6. TDD 驱动编码实现。
7. Code Review。
8. 测试与联调。
9. 部署与发布。
10. 观测、反馈、迭代。

## 11. 工程检查总清单
1. 是否已有项目级 PRD。
2. 是否已有设计稿（使用已声明设计工具）与页面状态说明。
3. 是否已有 API 契约与错误码。
4. 是否已有数据模型、索引、逻辑删除策略。
5. 是否已有缓存使用场景与 TTL（若项目使用缓存）。
6. 是否已有测试范围、提测门槛、发布回滚方案。
7. 是否明确 AI 可自动执行与需人工确认边界。
8. 是否已声明项目技术栈并记录于 `PROJECT_OVERRIDES.md`。

## AI 调用建议
AI 在生成任何产物前，先用本文件确定技术栈决策框架、TDD 方法论、允许动作边界与后续应读取的专题文档。

## 后续可扩展点
可增加组织级审批矩阵、合规等级矩阵、架构评审模板、质量红线清单。

## 12. 企业级深化要求
1. 工程活动必须始终围绕“需求可追踪、设计可审阅、实现可验证、发布可回滚、运行可观测”五个基本目标展开。
2. 任何为了赶进度而绕过文档、测试、Review、审批、发布链路的做法，都视为工程治理缺陷，而不是可接受的临时技巧。
3. 设计结论必须能落到明确产物：需求对应 PRD、架构对应设计文档、接口对应契约、数据对应建模与 migration、发布对应流水线与回滚方案。
4. 工程规范必须可执行而不是口号化。每一类关键规范都应能映射到脚本、模板、Lint、测试、检查清单或审批节点。
5. 项目在追求“快”的同时必须保持“稳”。默认优先选择可维护、可验证、可交接的方案，而不是一次性堆叠实现。

## 13. 质量红线
1. 不允许在需求、边界、权限、数据口径不明确的情况下直接大规模生成或提交生产代码。
2. 不允许上线未通过最小测试闭环、无回滚路径、无环境配置说明、无关键日志和告警观察点的变更。
3. 不允许以 AI 输出为理由跳过人工决策点、删除关键保护逻辑或降低审计、安全、权限要求。
4. 不允许让项目长期依赖单人记忆或单次会话上下文维持运行，必须沉淀到仓库内正式文档。

## 14. 架构与治理评审补充清单
1. 当前方案是否同时满足交付目标和后续扩展目标。
2. 关键分层、关键依赖和关键边界是否已冻结并文档化。
3. 是否已明确哪些决策由 AI 自动执行，哪些必须人工确认。
4. 是否已把测试、观测、迁移、发布、回滚作为设计输入而不是收尾动作。
5. 是否存在任何“先做出来再治理”的高风险遗留点；若存在，必须在决策日志中显式登记。
