# 文件名
`00_GLOBAL_ENGINEERING_PRINCIPLES.md`

# 文件定位
跨项目母规范的最高优先级治理文件。

# 适用范围
适用于所有基于既定技术基线开展的需求分析、设计、研发、测试、部署、运维与 AI 协同场景。

# 与其他文件的关系
本文件是所有专题规范的总纲。后续文档不得与本文件冲突；如需项目级覆盖，必须显式声明并经过人工审批。

# 编写目的
定义跨项目统一的工程哲学、默认技术基线、规范优先级、AI 决策边界与后续专题文档的主从关系。

## 1. 全局工程哲学
1. 规范优先于实现。未明确规范前，不允许直接进入大规模编码。
2. 设计优先于开发。页面、流程、数据模型、接口边界应先定义再实现。
3. 文档优先于口头约定。影响交付、协作、扩展的规则必须沉淀为文档。
4. 可维护性优先于短期堆功能。宁可多做抽象与分层，也不允许无边界堆逻辑。
5. 可扩展性优先于一次性脚本式实现。默认考虑模块增长、角色增长、数据量增长与部署增长。
6. 可测试性优先于隐式耦合。所有关键业务链路必须具备可验证输入、输出、状态。
7. 可部署性优先于本地可跑。实现方案必须能进入容器化、可配置、可回滚的交付链路。
8. 可观测性优先于“出了问题再查”。日志、指标、告警、审计应在设计阶段预留。

## 2. 默认技术决策
### 2.1 前端默认技术栈
1. Vue 3。
2. TypeScript。
3. Vite。
4. Pinia。
5. Vue Router。
6. Axios 或统一请求层封装。
7. Element Plus 作为默认 UI 组件库。
8. Pencil 作为设计稿来源。

### 2.2 后端默认技术栈
1. Java Spring Boot 作为主业务后端。
2. Maven 作为构建工具。
3. MyBatis-Plus 作为默认 ORM 辅助框架。
4. RESTful API 作为默认接口风格。
5. Python 仅作为辅助服务，用于 AI、数据处理、异步任务、脚本任务、算法能力。

### 2.3 数据与中间件
1. MySQL 作为默认数据库。
2. Redis 作为默认缓存与轻量中间件。
3. JWT + Redis 作为默认认证鉴权基线。

### 2.4 部署与交付
1. Docker 作为默认镜像封装方式。
2. docker compose 作为默认多容器编排方式。
3. 默认容器角色包含前端、Java 后端、Python 服务、MySQL、Redis。

### 2.5 明确禁止
1. 未经项目级书面覆盖，不得擅自切换到 React、Next.js、Node.js 主后端、PostgreSQL、MongoDB、Gradle、GraphQL。
2. 不得绕开统一请求层、统一异常层、统一权限层、统一日志审计层直接拼接实现。
3. 不得因 AI 输出方便而降低工程治理要求。

## 3. 文档驱动开发原则
1. 需求必须先形成结构化 PRD，再进入设计与拆解。
2. UI/UX 必须先有 Pencil 设计稿或等价线框，再进入页面开发。
3. API 必须先有接口契约，再进入前后端联调。
4. 数据库必须先有数据模型与索引设计，再落表。
5. 部署必须先有环境配置、镜像与回滚策略，再进入上线流程。

## 4. 设计先行原则
1. 页面信息架构必须先明确层级、入口、主任务路径。
2. 核心业务流程必须先明确节点、状态、操作权限、异常分支。
3. 数据结构必须先明确主实体、关联实体、状态机、审计字段。
4. UI 实现必须尊重设计稿，但允许在“工程可实现、交互一致、复用更优”的前提下进行受控调整。

## 5. 规范优先级
1. 法规、合规、组织级安全要求。
2. 本母规范目录。
3. 项目级补充规范。
4. 模块级约定。
5. 开发者个人习惯。

发生冲突时，优先级高者覆盖低者。任何覆盖都必须留下文档与审批记录。

## 6. 默认架构立场
1. 后端采用“单体优先、模块清晰、按需拆分”原则。
2. 前端采用“页面层薄、业务组件中等、通用层稳定、请求与状态分离”原则。
3. 数据层采用“关系模型优先、缓存辅助、异步增效、不提前过度微服务化”原则。
4. 认证授权采用“统一身份、统一鉴权、统一审计、统一权限扩展点”原则。

## 7. AI 在项目中的角色边界
### 7.1 AI 可以自动决定的事项
1. 在已明确技术栈范围内生成标准目录结构。
2. 按现有规范生成模板代码、重复性样板、DTO/VO/Entity、CRUD 页面骨架。
3. 按接口契约生成前后端对接代码。
4. 按建模规则生成 SQL、索引建议、Redis key 方案草案。
5. 按测试规范生成测试用例、Mock 数据、联调清单、发布检查清单。

### 7.2 AI 必须人工确认的事项
1. 需求范围变化。
2. 核心业务规则、审批规则、计费规则、权限规则。
3. 跨系统集成方式。
4. 安全等级、脱敏策略、审计范围。
5. 技术栈变更、部署拓扑变更、数据库重大重构。
6. 对外 API 暴露策略、版本废弃策略、生产环境变更。

### 7.3 AI 明确不得擅自决定的事项
1. 更换固定技术栈。
2. 删改关键日志、审计、权限、校验、测试环节。
3. 以“为了跑通”为由跳过异常处理、边界处理、数据校验。
4. 在需求、设计、接口、数据模型缺失时自造业务规则并直接落地。

## 8. 人工确认门禁
以下事项必须经过产品、架构、研发负责人或指定审批人确认：
1. 新增一级业务模块。
2. 调整核心领域模型。
3. 涉及金额、权限、流程审批、数据删除、数据导出。
4. 影响多环境配置、容器编排、外部网络访问、第三方依赖。

## 10. 标准交付流程
1. 需求澄清。
2. PRD 定稿。
3. 信息架构与 UI 设计。
4. 数据模型与 API 设计。
5. 前后端任务拆解。
6. 编码实现。
7. Code Review。
8. 测试与联调。
9. 部署与发布。
10. 观测、反馈、迭代。

## 11. 工程检查总清单
1. 是否已有项目级 PRD。
2. 是否已有 Pencil 设计稿与页面状态说明。
3. 是否已有 API 契约与错误码。
4. 是否已有数据模型、索引、逻辑删除策略。
5. 是否已有 Redis 使用场景与 TTL。
6. 是否已有测试范围、提测门槛、发布回滚方案。
7. 是否明确 AI 可自动执行与需人工确认边界。
