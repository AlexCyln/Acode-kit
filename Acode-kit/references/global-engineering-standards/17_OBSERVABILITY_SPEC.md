# 文件名
`17_OBSERVABILITY_SPEC.md`

# 文件定位
日志、指标、链路追踪、告警与运行观测规范。

# 适用范围
适用于前端、后端、辅助服务、缓存、数据库及部署平台的观测设计。

# 与其他文件的关系
本文件与 `12_DEBUG_AND_TROUBLESHOOTING_SPEC.md`、`13_DEPLOYMENT_AND_DEVOPS_SPEC.md` 共同构成运行期治理规范。

# 编写目的
建立日志、指标、链路与告警的统一观测语言，让系统问题可发现、可定位、可量化、可复盘。

## 1. 观测目标
1. 快速发现故障。
2. 快速定位问题。
3. 快速评估影响范围。
4. 为容量规划和优化提供数据。

## 2. 三大观测支柱
1. 日志：记录发生了什么。
2. 指标：量化系统状态。
3. Trace：串联请求链路。

## 3. 日志规范
1. 日志级别至少区分 `INFO`、`WARN`、`ERROR`。
2. 关键业务操作写业务日志。
3. 所有请求日志应带 `traceId`。
4. 错误日志应包含异常分类、关键上下文、资源 ID。
5. 禁止记录敏感明文。
6. 关键日志字段建议统一包含：`traceId`、`spanId`、`tenantId`、`userId`、`module`、`action`、`result`、`durationMs`。
7. 错误日志必须区分“业务失败”“系统异常”“下游异常”“安全拦截”。

## 4. 指标规范
必须具备以下基础指标：
1. 请求量、成功率、错误率、响应时间。
2. 运行时指标：CPU、内存、线程/协程、磁盘（如 JVM 应用还需关注 GC、堆内存等）。
3. 数据库连接池、慢查询数量。
4. 缓存命中率、连接数、慢命令（若使用缓存层）。
5. 异步任务积压量、失败数。
6. 多租户项目应能按租户、模块、接口维度查看关键质量指标。

## 5. Trace 规范
1. 网关、后端、辅助服务之间统一传递 `traceId`。
2. 核心跨服务调用必须可串联。
3. 数据库、缓存、外部 API 耗时应可观测。

## 6. 告警规范
1. 告警应按严重程度分级。
2. 生产不可用、错误率突增、数据库异常、缓存雪崩需立即告警。
3. 告警必须绑定负责人和值班规则。
4. 避免噪音告警，需设置聚合与抑制策略。

## 6.1 最小告警集合
1. 接口错误率突增
2. 响应时间明显劣化
3. 数据库连接池异常 / 慢查询持续升高
4. 核心异步任务积压
5. 登录失败异常升高 / 权限拦截异常升高
6. 发布后新版本异常率高于基线

## 7. 前端观测建议
1. 记录 JS 异常、接口失败、页面白屏、资源加载失败。
2. 记录关键页面加载耗时和关键交互失败率。
3. 必要时上报用户标识、浏览器信息、版本号，但不得泄露隐私数据。

## 8. 后端观测建议
1. 提供健康检查与运行指标端点。
2. 关键任务、导入导出、审批流应有业务指标。
3. 慢查询与异常接口需进入重点看板。

## 9. 仪表盘最小集合
1. 全局服务状态看板。
2. API 质量看板。
3. 数据库与缓存运行看板（若使用缓存）。
4. 任务执行看板。
5. 登录与权限异常看板。

## 10. 观测落地清单
1. 是否生成 `traceId`。
2. 是否有统一日志字段。
3. 是否暴露关键指标。
4. 是否配置核心告警。
5. 是否保留发布版本与环境维度。
6. 是否保留租户、用户、模块等排障关键维度。

## 本文件使用建议
系统上线前应至少完成本文件定义的日志、指标、告警最小闭环，避免“可运行但不可运维”的交付。

## AI 调用建议
AI 在生成日志、监控、健康检查、追踪上下文代码时，应按本文件补齐 traceId、业务指标和告警所需字段。

## 后续可扩展点
可增加 SLO/SLA 规范、值班告警路由规范、日志归档规范、链路追踪平台接入规范。

## 18. 企业级可观测性要求
1. 可观测性不是上线后的附加项，而是系统设计、排障、发布验证和容量治理的基础能力。
2. 关键业务链路必须至少具备日志、指标、健康检查和异常定位信息中的必要组合，不能只留一类信号。
3. 所有高风险动作和高价值链路都应有可追踪上下文，如 traceId、requestId、tenantId、userId、module 和 version。
4. 可观测性设计必须服务真实排障，避免只输出大量噪音日志却无法定位业务问题。

## 19. 指标与告警治理
1. 指标应覆盖系统层、应用层、业务层和依赖层，不得只有 CPU、内存、QPS 这类基础指标。
2. 告警必须可行动，至少说明影响范围、可能原因、观察入口和责任边界，避免大量无意义抖动告警。
3. 发布、迁移、租户切换、外部集成异常等关键事件应有专项观察点。
4. 日志等级、指标标签和告警阈值应定期治理，避免长期积累无价值信号。

## 20. Review 附加清单
1. 关键链路是否具备排障所需的最小证据链。
2. 是否保留了租户、用户、模块、版本和环境等关键维度。
3. 是否能通过日志、指标和告警定位发布、性能、权限和外部依赖问题。
4. 是否存在大量噪音日志或无主告警而没有治理计划。
