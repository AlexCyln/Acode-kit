# 文件名
`32_DATABASE_MIGRATION_SPEC.md`

# 文件定位
数据库结构变更、数据迁移、回填与发布链路治理规范。

# 适用范围
适用于所有关系型数据库结构变更、索引变更、约束变更、字典初始化、数据回填和版本升级场景。

# 与其他文件的关系
本文件补充 `06_DATABASE_DESIGN_SPEC.md`、`13_DEPLOYMENT_AND_DEVOPS_SPEC.md`、`14_CICD_SPEC.md`，并为技术栈包中的 Flyway / Liquibase 具体落地提供母规范基线。

# 编写目的
确保数据库变更版本化、可追溯、可回滚、可重复执行，禁止正式环境手工改库导致环境漂移。

## 1. 治理目标
1. 所有数据库结构变更必须可审阅、可追踪、可复现。
2. 开发、测试、预发、生产环境结构必须通过同一套脚本演进。
3. 数据库发布不得绕开代码仓库和发布链路。
4. 破坏性变更必须前置评估、分阶段执行并具备回退预案。

## 2. 核心原则
1. 结构变更统一通过 Migration 工具管理。
2. 脚本必须随代码入库，与应用版本一同发布。
3. 一变更一脚本，单脚本只表达一个清晰主题。
4. 禁止先手工改正式环境再补脚本。
5. 优先前向兼容，必要时采用“先加后迁再删”的渐进式策略。
6. 数据迁移和结构迁移分离评审，避免同一脚本承担过多职责。

## 3. 变更分类
1. `DDL`：表、字段、索引、约束、视图等结构变更。
2. `Reference Data`：标准字典、系统初始化数据、默认配置数据。
3. `Backfill`：历史数据回填、补齐、字段迁移。
4. `Repair`：线上事故后的受控修复脚本，仅允许通过正式审查和发布链路执行。

## 4. 版本命名规范
1. 采用有序版本号，保证工具可判断执行顺序。
2. 文件名必须体现变更意图，例如：
   - `V12__add_index_user_email.sql`
   - `V13__add_order_audit_status_column.sql`
3. 命名使用英文小写语义，不使用模糊词如 `fix.sql`、`update.sql`。
4. 若采用带日期或批次前缀的版本规则，项目级必须统一，不得混用。

## 5. 一变更一脚本规则
1. 一个脚本只处理一个明确主题，例如“新增索引”“新增字段”“初始化字典”。
2. 不得把互不相关的多个表改动混在同一脚本中。
3. 大型重构应拆为多步迁移，以便 review、回滚和灰度控制。

## 6. 环境一致性要求
1. 所有环境必须通过同一 Migration 工具执行同一脚本集合。
2. 禁止开发环境和生产环境长期存在结构漂移。
3. 若测试环境临时修复过结构，必须立即补正式脚本并回归所有环境。

## 7. 前向兼容策略
1. 优先新增字段、双写、回填、切流，再删除旧字段。
2. 删除列、改类型、改主键、改唯一约束等破坏性变更不得一步到位。
3. 对大表索引、非空约束、默认值回填要评估锁表和耗时风险。

## 8. 回滚策略
1. 每类数据库变更都必须明确回滚方式或回退策略。
2. 可逆变更应提供回滚脚本或明确回滚步骤。
3. 不可逆变更必须提前说明并准备备份、灰度、双写或止损方案。
4. 回滚不只是“回 SQL”，还包括应用版本、配置、数据状态的整体回退评估。

## 9. 发布链路约束
1. 数据库变更必须纳入正式发布链路执行。
2. 发布前必须完成迁移脚本 review、验证和回滚评估。
3. 不得跳过 CI/CD 或正式部署流程直接执行生产变更。
4. 应用启动自动执行迁移仅适用于项目已显式声明的模式；高风险环境可改为发布前专门迁移步骤。

## 10. 数据回填规则
1. 回填脚本必须说明输入范围、影响行数预估和幂等策略。
2. 大批量回填优先分批执行，避免长事务和锁冲突。
3. 回填结果必须有核对 SQL、接口验证点或业务验收点。
4. 回填失败必须能定位已执行范围和剩余范围。

## 11. Review 清单
1. 变更意图是否单一且明确。
2. 是否说明影响的表、字段、索引、数据量和高峰风险。
3. 是否说明前向兼容或破坏性策略。
4. 是否说明回滚或止损方案。
5. 是否附带迁移前后验证方式。
6. 是否同步影响 API、Mapper、实体、字典、缓存、文档和测试。

## 12. 禁止事项
1. 直接手工修改正式环境库结构。
2. 先改生产再补脚本。
3. 一个脚本混入多类无关改动。
4. 无验证点的回填脚本直接上线。
5. 无回滚评估的破坏性变更直接发布。

## 13. 示例
新增用户表邮箱索引，不应直接在正式环境执行 SQL。正确做法是新增：

```text
V12__add_index_user_email.sql
```

并通过 Migration 工具在发布时自动执行，使开发、测试、预发、生产环境保持一致。

## 本文件使用建议
数据库结构评审、发版评审、上线检查和故障修复前均应回到本文件判断是否满足版本化治理要求。

## AI 调用建议
AI 在生成 DDL、回填脚本、初始化数据脚本和数据库发布说明时，必须先遵守本文件，不得建议用户手工改正式环境数据库。

## 14. 企业级执行细则
1. Migration 不只是“放几份 SQL 文件”，而是数据库演进的唯一正式入口，必须与代码评审、发布审批、环境验证一体化治理。
2. 结构变更、索引变更、参考数据初始化、历史数据回填应区分执行策略和风险等级，不得一刀切处理。
3. 大表变更、热点表变更、租户字段回填、唯一约束调整等高风险操作，必须提前给出时间窗、锁影响、回退策略和业务告警观察点。
4. 当数据库变更影响接口、缓存、搜索、报表、导出或权限逻辑时，必须同步更新相关规范文档和测试方案。

## 15. 发布与验证要求
1. 每次 migration 发布前，至少应确认脚本顺序、影响对象、执行时长预估、备份策略、回滚策略和验证 SQL。
2. 发布后必须进行结构验证和业务验证，不能仅以“脚本执行成功”作为结束条件。
3. 对历史数据回填或批处理迁移，应记录批次范围、执行结果、失败重试策略和剩余待处理量。
4. 若项目存在多环境或多租户，验证应至少覆盖一个真实租户样本和一个关键业务链路。

## 16. Review 附加清单
1. 本次脚本是否保持单一意图且命名清晰。
2. 是否存在手工改库、环境漂移或“已在线下先执行”的隐性前提。
3. 是否明确结构变更与数据回填的边界，避免混成不可审计的大脚本。
4. 是否提供了足够的执行前检查和执行后核对依据。
5. 是否同步考虑了应用版本切换、缓存失效、搜索索引、报表口径和多租户影响。
