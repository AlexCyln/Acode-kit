# 文件名
`33_MULTI_TENANCY_ARCHITECTURE_SPEC.md`

# 文件定位
多租户架构、租户隔离、租户级配置与治理规范。

# 适用范围
适用于平台型 SaaS、多组织管理系统、租户隔离型后台和允许租户自定义配置/数据的项目。

# 与其他文件的关系
本文件补充 `04_BACKEND_ARCHITECTURE_SPEC.md`、`06_DATABASE_DESIGN_SPEC.md`、`16_SECURITY_SPEC.md`、`29_DATA_DICTIONARY_AND_REFERENCE_DATA_SPEC.md`，并指导技术栈包中的多租户具体落地。

# 编写目的
明确多租户项目中“租户边界、数据边界、权限边界、配置边界”的统一治理方式，避免实现阶段才临时补租户字段和隔离逻辑。

## 1. 设计目标
1. 让租户边界在架构阶段即显式冻结。
2. 保证不同租户之间的数据、配置、权限、日志可隔离。
3. 保证平台级数据与租户级数据边界清晰。
4. 为未来租户扩展、计费、审计和运维提供稳定基础。

## 2. 多租户模式
1. 共享库共享表 + `tenant_id`
2. 共享库分 schema
3. 独立数据库 / 独立实例

规则：
1. 项目初始化时必须明确采用哪一种模式。
2. 模式一旦确定，必须写入 `ACTIVE_STANDARDS.md` 和 `PROJECT_OVERRIDES.md`。
3. 若后续变更租户模式，视为重大架构变更，必须重新评审。

## 3. 租户边界
1. 平台级对象：平台管理员、系统配置、标准字典、平台规则。
2. 租户级对象：租户用户、租户组织、租户业务数据、租户配置。
3. 共享对象必须明确“平台只读”“租户可读不可改”“租户可扩展”哪一类边界。

## 4. 数据隔离规则
1. 租户级业务表必须显式带 `tenant_id` 或等价隔离键。
2. 查询、分页、导出、统计、缓存、异步任务都必须体现租户隔离。
3. 后端必须在最终数据访问边界保证租户过滤，不得只依赖前端传参。
4. 平台级报表若跨租户展示，必须走受控平台权限路径。

## 5. 权限与身份规则
1. 用户身份必须同时明确“属于哪个租户”和“在租户内拥有什么角色”。
2. 平台管理员权限与租户管理员权限必须分开设计。
3. 权限判断至少区分：平台级、租户级、组织级、个人级。
4. 切换租户上下文时必须有明确的鉴权、审计和缓存刷新机制。

## 6. 配置与字典规则
1. 标准参考字典默认平台级维护，不允许租户任意修改。
2. 允许租户扩展的配置、模板、字典必须显式标记作用域。
3. 配置优先级必须明确：平台默认值 < 租户覆盖值 < 临时运行态配置。

## 7. API 规则
1. API 必须明确租户上下文来源：Token、Header、域名、路径或后台会话。
2. 租户上下文不得由前端随意伪造。
3. 跨租户管理接口必须单独设计权限，不得复用租户内接口。

## 8. 数据库与 Migration 规则
1. 多租户字段和索引必须在建模阶段就进入设计稿。
2. 新增多租户能力时不得临时拼接租户过滤，必须同步结构和索引设计。
3. 涉及多租户改造的 migration 必须评估存量数据回填和索引代价。

## 9. 日志与审计规则
1. 业务日志、审计日志、操作日志必须带租户维度。
2. 平台级排障需要跨租户定位时，也必须保留租户上下文。
3. 不得输出会导致跨租户数据泄漏的日志内容。

## 10. 测试规则
1. 至少验证单租户正常路径。
2. 必须验证跨租户越权拦截。
3. 必须验证平台管理员与租户管理员权限差异。
4. 涉及缓存、导出、搜索、报表时必须验证租户隔离未被绕开。

## 11. 禁止事项
1. 先做单租户实现，后期再全局补 `tenant_id` 但无整体评审。
2. 只在前端页面层做租户隔离。
3. 平台级与租户级配置混存却无作用域字段。
4. 跨租户查询无审计留痕。

## 本文件使用建议
凡项目带有“平台、租户、组织、空间、工作区”等多主体结构，都应先判断是否进入本规范约束。

## AI 调用建议
AI 在生成多租户数据模型、权限模型、Mapper、缓存键和日志字段时，必须先遵守本文件，避免只在局部实现租户概念。

## 12. 企业级深化要求
1. 多租户不是在表里增加一个 `tenant_id` 就结束，而是覆盖数据、权限、缓存、搜索、报表、任务、日志、导出、配置的系统性约束。
2. 租户边界必须在需求和架构阶段冻结，后续所有模型、接口和实现都围绕该边界展开，而不是按功能点零散补丁式支持。
3. 平台级能力和租户级能力必须能清晰区分操作主体、数据范围、配置范围和审计范围。
4. 涉及租户扩展、自定义字典、自定义模板、自定义流程时，必须提前定义哪些可扩展、哪些不可扩展、扩展后如何治理。

## 13. 实施细则
1. 所有租户级读写路径都应在最终数据访问层强制带上租户约束，避免仅靠上层调用约定维持隔离。
2. 缓存键、消息主题、异步任务载荷、导出文件名、审计日志、搜索索引文档都必须包含租户维度或等价隔离上下文。
3. 跨租户平台能力必须单独建模和授权，不得复用租户内接口通过参数切换实现。
4. 切换租户上下文时，必须同步处理权限刷新、缓存隔离、会话隔离和审计记录，避免“身份切换成功但上下文残留”。

## 14. Review 附加清单
1. 当前方案是否已经明确采用的多租户模式和后续扩展边界。
2. 数据、权限、配置、缓存、日志、导出、报表是否全部体现租户隔离，而不是只覆盖数据库查询。
3. 平台管理员、租户管理员、普通用户的权限边界是否清楚且可验证。
4. 多租户改造是否同步评估了 migration、索引、回填、历史数据兼容和测试成本。
5. 是否存在任何跨租户能力没有审计留痕或没有单独授权。
