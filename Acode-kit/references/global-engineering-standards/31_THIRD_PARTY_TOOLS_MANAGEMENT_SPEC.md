# 文件名
`31_THIRD_PARTY_TOOLS_MANAGEMENT_SPEC.md`

# 文件定位
第三方工具（MCP 工具）的注册、安装、状态追踪与降级管理规范。

# 适用范围
适用于所有需要集成第三方 MCP 工具的项目初始化、开发与运维场景。

# 与其他文件的关系
本文件受 `00_GLOBAL_ENGINEERING_PRINCIPLES.md` 约束，为 `SKILL.md` 的工作流提供工具管理基础。`01_PRODUCT_REQUIREMENTS_STANDARD.md`、`02_UI_UX_DESIGN_SPEC.md` 中涉及设计工具的引用指向本文件。

# 编写目的
定义项目中使用的第三方 MCP 工具的注册表、安装流程、状态追踪机制与降级策略，确保工具可用性不阻塞核心工作流。

---

## 1. 工具注册表

以下为当前支持的 MCP 工具清单。每个工具包含安装方式、用途与降级方案。

### 1.1 Pencil MCP

| 属性 | 值 |
|------|-----|
| **用途** | UI/UX 设计稿创建与管理 |
| **类型** | 桌面应用 |
| **官网** | https://www.pencil.dev/ |
| **检测方式** | 扫描本地应用目录确认安装 |
| **安装方式** | 用户手动从官网下载安装桌面应用 |

### 1.2 NotebookLM MCP

| 属性 | 值 |
|------|-----|
| **用途** | 需求分析、项目骨架生成、大规模变更分析 |
| **类型** | MCP 服务 |

**安装命令**：

Claude Code：
```bash
claude mcp add notebooklm npx notebooklm-mcp@latest
```

Codex：
```bash
codex mcp add notebooklm -- npx notebooklm-mcp@latest
```

**认证**：安装后在工作区状态报告阶段提示认证。若用户输入精确的 `Log me in to NotebookLM`，Acode-kit 监听层不得干预，应直接透传给底层 agent 处理认证。
认证成功后应将 `authCompleted` 写回用户级全局缓存，以便后续会话直接复用。

**默认注入 prompt**：
```
Here's my NotebookLM: https://notebooklm.google.com/notebook/7ec4ec07-abb3-478e-99aa-f8946e103499
```

### 1.3 shadcn MCP

| 属性 | 值 |
|------|-----|
| **用途** | UI 组件库构建与管理 |
| **类型** | MCP 服务 |

**安装命令**：

Claude Code：
```bash
npx shadcn@latest mcp init --client claude
```

Codex（手动配置 `~/.codex/config.toml`）：
```toml
[mcp]

[mcp.servers.shadcn]
command = "npx"
args = ["-y", "shadcn@latest", "mcp"]
```

**调用规则**：
1. shadcn MCP 仅在项目已声明使用 `shadcn/ui` 时才有意义；它是组件检索与实现辅助工具，不是工作流启动前置条件。
2. 在 Stage 2 / Step 5b 中，如需确认组件能力、变体、组合方式、布局可行性，可调用 shadcn MCP。
3. 在 Step 5d 中，如需确认组件接入方式、组合模式、实现细节，也可调用 shadcn MCP。
4. 不得在启动阶段（Steps 1-4 / Gates 1-4）把 shadcn MCP 当作必调工具。
5. 即使 shadcn MCP 缺失，只要项目已声明 `shadcn/ui`，前端实现仍必须遵守 shadcn 组件约束，不得擅自降级为自定义基础组件。

### 1.4 Chrome DevTools MCP

| 属性 | 值 |
|------|-----|
| **用途** | 浏览器调试、前端页面实时检查 |
| **类型** | MCP 服务 |

**安装命令**：

Claude Code：
```bash
claude mcp add chrome-devtools --scope user npx chrome-devtools-mcp@latest
```

Codex：
```bash
codex mcp add chrome-devtools -- npx chrome-devtools-mcp@latest
```

**调用规则**：
1. 对于浏览器可访问的 UI 范围，`Step 5e` 在模块级交付前必须调用 Chrome DevTools MCP 完成真实浏览器验证。
2. 对于浏览器可访问的集成范围，`Stage 6` 在最终联调/交付前必须调用 Chrome DevTools MCP 完成真实浏览器验证。
3. Chrome DevTools MCP 的验证结果必须回写测试文档或交接文档，不能只在对话中口头说明。
4. 若工具缺失或不可用，必须显式记录阻塞和降级路径，不得直接跳过真实浏览器验证要求。

## 2. 工具状态定义

每个工具在项目生命周期中具有以下三种状态之一：

| 状态 | 定义 | 行为 |
|------|------|------|
| **installed** | 工具已安装且功能正常 | 正常使用 |
| **missing** | 工具未安装或未检测到 | 触发安装建议流程 |
| **degraded** | 工具已安装但部分功能不可用 | 启用降级策略，记录受影响功能 |

---

## 3. 自动安装流程

项目初始化时执行以下流程：

1. **扫描**：检测所有注册工具的安装状态。
2. **检测缺失**：标记状态为 `missing` 的工具。
3. **建议安装**：向用户展示缺失工具清单与安装命令。
4. **用户授权**：等待用户确认是否执行安装。
5. **执行安装**：按用户授权逐一安装。
6. **验证**：安装后重新检测工具状态。
7. **记录结果**：将最终工具状态写入项目记录。
8. **全局同步**：将最终工具状态同步到用户级全局缓存（例如 `~/.codex/acode-kit/.acode-kit-global.json` 或 `~/.claude/acode-kit/.acode-kit-global.json`），用于后续会话复用。
9. **认证持久化**：NotebookLM 的 `authCompleted` 一旦被确认成功，应同步到同一份全局缓存，避免新会话重复触发重认证。

---

## 4. 降级策略

当工具不可用时，采用以下替代方案：

| 工具 | 降级方案 |
|------|----------|
| **Pencil MCP** | 使用文本描述的线框图或手绘草图；用户可选择其他设计工具手动创建设计稿 |
| **NotebookLM MCP** | AI 代理直接进行需求分析，不经过 NotebookLM 增强分析；输出质量可能降低，需用户更多介入确认 |
| **shadcn MCP** | 手动安装组件库依赖；使用已声明 UI 组件库的标准安装方式 |
| **Chrome DevTools MCP** | 使用浏览器内置开发者工具进行调试；AI 代理无法直接获取运行时信息 |

## 5. 工具状态报告格式

在项目文档中记录工具状态时使用以下格式：

```markdown
## MCP 工具状态

| 工具 | 状态 | 版本 | 备注 |
|------|------|------|------|
| Pencil MCP | installed | - | 桌面应用已安装 |
| NotebookLM MCP | installed | latest | 已认证 |
| shadcn MCP | missing | - | 项目未使用 shadcn 组件库 |
| Chrome DevTools MCP | installed | latest | - |
```

### 5.1 shadcn MCP 状态说明

当项目声明使用 `shadcn/ui` 时，状态报告必须明确区分：

1. **组件约束是否生效**：是否已在 `PROJECT_OVERRIDES.md` 中声明 `shadcn/ui`
2. **shadcn MCP 是否可用**：是否已安装并可调用
3. **降级方式**：若 MCP 缺失，说明“按已声明 shadcn 组件库继续实现，MCP 辅助能力降级，不阻塞主流程”

## 6. 维护规则

1. 新增 MCP 工具必须在本文件注册后方可在工作流中引用。
2. 工具注册表的变更需更新版本号与变更日志。
3. 每个工具必须提供明确的降级策略——不允许"工具缺位则流程中断"。
4. 工具安装不得绕过用户授权。

## 7. 企业级治理要求
1. 第三方工具是工程增强能力，不是业务真相来源。任何关键结论仍必须回落到项目文档、代码、测试和正式决策。
2. 工具接入前必须明确用途、边界、权限需求、数据流向、失败影响和降级方案，不得先接入后补治理。
3. 工具状态必须可检查、可记录、可追踪，避免团队依赖“某人本地正好能用”的隐式前提。
4. 涉及外部账号、浏览器态、认证状态或收费额度的工具，必须明确认证与续期策略。

## 8. 安全与权限约束
1. 工具所需最小权限必须显式记录，避免给予不必要的文件系统、网络、浏览器或生产环境访问能力。
2. 若工具会接触源代码、设计稿、接口数据、生产日志或敏感配置，必须先评估数据分类和泄露风险。
3. 认证信息、Token、Cookie、密钥、服务地址不得直接写入仓库；如需持久化，必须走受控配置和本地安全存储。
4. 工具可访问的环境范围必须区分本地、测试、预发、生产，不得默认全环境放开。

## 9. 生命周期管理
1. 新工具引入时必须登记引入原因、替代关系、维护人或责任角色。
2. 工具版本升级、安装方式变化、弃用或替换时，必须同步更新注册表、初始化说明和降级策略。
3. 若工具长期不可用、维护中止或与当前工作流不再匹配，应显式标记为弃用状态，而不是继续保留过期说明。
4. 工具清单应定期审查，清理“已不使用但仍占据流程说明”的历史遗留工具。

## 10. Review 附加清单
1. 工具是否真的解决了当前流程痛点，而不是增加新的耦合点。
2. 是否明确了安装、认证、使用、失效、降级、替换的完整闭环。
3. 是否把工具能力误写成业务前置条件或架构前提。
4. 是否存在任何未登记却已进入正式流程依赖的外部工具。
