# 文件名
`05_API_DESIGN_SPEC.md`

# 文件定位
RESTful API 设计与接口契约标准。

# 适用范围
适用于所有前后端内部接口、前端直连接口、主服务与辅助服务协作接口，以及未来对外开放接口的设计。

# 与其他文件的关系
本文件衔接 `01_PRODUCT_REQUIREMENTS_STANDARD.md` 与 `04_BACKEND_ARCHITECTURE_SPEC.md`，并为 `03_FRONTEND_ARCHITECTURE_SPEC.md`、`16_SECURITY_SPEC.md`、`33_MULTI_TENANCY_ARCHITECTURE_SPEC.md` 提供接口契约基线。

# 编写目的
定义跨项目统一的 RESTful API 契约，约束 URI、方法、错误码、分页、版本、认证与字段传输方式。

## 1. 设计目标
1. 统一命名与协议风格。
2. 降低前后端理解成本。
3. 保证可演进、可版本化、可审计、可测试。

## 2. URI 命名规范
1. 使用名词复数描述资源，如 `/api/v1/users`。
2. 使用 kebab-case，不使用驼峰与中文。
3. 资源层级不超过三级，避免过深嵌套。
4. 动作型接口仅在资源语义无法自然表达时使用子路径，如 `/orders/{id}/approve`。

## 3. HTTP Method 使用规范
1. `GET`：查询资源，不产生副作用。
2. `POST`：创建资源、复杂查询、执行动作。
3. `PUT`：整体更新。
4. `PATCH`：部分更新。
5. `DELETE`：删除资源，若逻辑删除需明确业务语义。

## 4. 请求与响应基础规范
1. `Content-Type` 默认使用 `application/json`。
2. 上传使用 `multipart/form-data`。
3. 导出文件使用二进制流，文件名编码明确。
4. 所有接口都返回统一结构，除下载流与极少数回调接口外不得例外。

## 5. 统一返回格式
```json
{
  "code": "0",
  "message": "success",
  "data": {
    "items": [],
    "total": 0
  },
  "traceId": "a1b2c3",
  "timestamp": "2026-03-12T10:00:00Z"
}
```

## 6. 分页结构规范
分页查询请求建议：
```json
{
  "pageNum": 1,
  "pageSize": 20,
  "filters": {
    "keyword": "alex",
    "status": "ENABLED"
  },
  "sorts": [
    { "field": "createdTime", "order": "desc" }
  ]
}
```

分页响应建议：
```json
{
  "items": [],
  "pageNum": 1,
  "pageSize": 20,
  "total": 120
}
```

## 7. 列表筛选与排序规范
1. 简单查询可使用 Query String。
2. 复杂查询统一使用 `POST /resource/page` 或 `POST /resource/search`。
3. 排序字段必须白名单校验，禁止前端任意传 SQL 字段。
4. 筛选项名称必须与 PRD、页面字段、DTO 字段一致。

## 8. 批量操作规范
1. 批量删除、批量启停、批量审核需单独接口。
2. 请求体必须显式传递 ID 列表与操作参数。
3. 响应体应返回成功数、失败数、失败原因列表。

示例：
```json
{
  "ids": [1001, 1002, 1003],
  "targetStatus": "DISABLED",
  "reason": "batch maintenance"
}
```

## 9. 导入导出规范
1. 导入接口需返回任务 ID 或详细错误列表。
2. 导出需区分同步小文件导出与异步大文件导出。
3. 模板下载、导入结果、错误报告文件需单独接口。

## 10. 错误码体系
建议分层：
1. `0`：成功。
2. `4xxx`：客户端错误，如参数错误、未认证、无权限、资源不存在。
3. `5xxx`：服务端错误，如系统异常、下游异常、任务失败。
4. `8xxx`：业务错误，如状态流转非法、重复提交、余额不足。

示例：
```md
4001 参数校验失败
4010 未登录或 Token 无效
4030 无接口权限
4040 资源不存在
4090 幂等冲突或重复提交
5000 系统内部异常
8001 业务状态不允许当前操作
```

## 11. 认证头与 Token 规范
1. 使用 `Authorization: Bearer <token>`。
2. 刷新 Token 策略应项目级明确，但不得与 Access Token 混用。
3. 不得通过 Query String 传递 Token。
4. 服务间内部调用可使用独立凭证机制，不复用前端用户 Token。

## 12. 幂等性规范
1. 创建支付、提交流程、导入任务、批量任务等需具备幂等控制。
2. 前端可传 `Idempotency-Key`，后端结合缓存存储或数据库去重。
3. 幂等冲突需返回明确错误码和提示。

## 13. 版本管理规范
1. 默认 URI 级版本号，如 `/api/v1/users`。
2. 破坏性变更必须升级版本。
3. 非破坏性新增字段优先保持向后兼容。
4. 废弃接口需保留迁移窗口与公告。

## 13.1 兼容性与废弃治理
1. 字段新增优先“新增可选字段”，避免直接修改既有字段语义。
2. 字段重命名、删除、类型变更视为破坏性变更，必须升级版本或提供迁移窗口。
3. 废弃接口必须标记废弃时间、替代接口和最晚下线时间。
4. 前后端联调阶段不得临时更改字段语义却不更新文档和版本说明。

## 14. 字段传输统一规范
1. JSON 字段名统一使用 camelCase。
2. 日期时间统一使用 ISO 8601，推荐 UTC 或明确时区。
3. 金额传输默认使用最小货币单位整数，展示层再格式化。
4. 布尔值使用 `true/false`，不使用 `0/1` 冒充布尔。
5. 枚举传输优先使用稳定编码值，如 `APPROVED`，不要传中文展示名。
6. 字典字段与标准参考字段应提供统一查询接口或枚举中心能力，不允许前端自行维护私有映射。

## 15. Swagger / OpenAPI 文档规范
1. 所有对前端开放的接口必须产出 OpenAPI 文档。
2. 文档应包含接口说明、参数说明、示例、错误码、权限要求。
3. DTO 字段需写注释，避免只有代码名无业务语义。

## 15.1 权限、租户与审计信息
1. 每个接口文档必须说明权限要求、角色边界和是否涉及数据权限。
2. 多租户项目必须说明租户上下文来源和租户隔离边界。
3. 高风险接口必须说明审计要求、幂等要求和限流/二次确认要求。

## 16. 前后端字段统一规则
1. 页面字段名、DTO 字段名、VO 字段名尽量保持一致。
2. 若数据库字段与前端字段风格不同，应在 convert 层处理。
3. 不允许前端自己猜测状态值、字典值、金额单位。
4. 标准参考字典如国家码、币种、行业分类，应使用标准编码值传输，不以中文名作为业务主值。

## 17. API 设计示例
```http
GET /api/v1/users/{id}
POST /api/v1/users/page
POST /api/v1/users
PUT /api/v1/users/{id}
PATCH /api/v1/users/{id}/status
DELETE /api/v1/users/{id}
POST /api/v1/users/import
GET /api/v1/users/export-template
```

## 18. 接口评审检查清单
1. URI 是否语义清晰。
2. Method 是否匹配操作语义。
3. 是否定义统一错误码。
4. 是否说明权限要求。
5. 是否说明幂等要求。
6. 是否考虑分页、导入导出、批量场景。
7. 是否说明版本兼容策略。
8. 是否说明权限、审计、多租户上下文。

## 本文件使用建议
接口设计、联调、文档生成、SDK 生成前均应使用本文件校验契约是否统一。

## AI 调用建议
AI 在生成接口定义、前端 API 模块、OpenAPI 描述、联调 Mock 数据时必须遵循本文件的 URI、字段、错误码、版本与幂等规则。

## 15. 企业级契约治理要求
1. API 契约必须被视为前后端和外部集成的正式边界，不得以“前后端同组沟通即可”替代结构化契约。
2. 接口字段、枚举、错误码、分页结构、排序规则、幂等语义和权限语义必须稳定且可文档化。
3. 对外接口和对内接口都必须明确版本兼容策略，不得通过无通知字段漂移破坏调用方。
4. 高风险接口如导出、回调、批量操作、审批动作、资金动作，必须显式增加防重、审计和权限说明。

## 16. 读写路径设计要求
1. 读接口必须说明筛选、排序、分页、权限过滤、租户过滤和缓存策略。
2. 写接口必须说明幂等键、并发冲突、状态前置条件、审计记录和失败返回口径。
3. 批量接口必须明确部分成功、全量失败或逐项反馈的处理策略。
4. 导入导出接口必须明确文件格式、字段映射、异步处理、失败明细和重试方式。

## 17. Review 附加清单
1. 契约是否足以支持前后端并行开发和独立联调。
2. 是否显式定义了错误码、空值语义、时间与金额口径、枚举来源。
3. 是否说明了权限、租户、缓存、幂等和审计要求。
4. 是否存在“当前够用但后续不可兼容”的字段或路径设计。
