# 文件名
`11_TESTING_AND_QA_SPEC.md`

# 文件定位
测试分层、提测门槛、联调与发布验收规范。

# 适用范围
适用于功能开发、重构、Bug 修复、发版验收、回归测试与质量门禁管理。

# 与其他文件的关系
本文件与 `10_CODE_REVIEW_SPEC.md`、`13_DEPLOYMENT_AND_DEVOPS_SPEC.md`、`14_CICD_SPEC.md` 联动，控制代码从开发到上线的质量门槛。

# 编写目的
定义从自测、提测、联调、回归到发布验收的统一质量流程，确保交付物可验证、可复现、可上线。

## 1. 测试类型边界
1. 单元测试：验证函数、类、服务方法的局部逻辑。
2. 接口测试：验证 API 契约、鉴权、状态码、返回结构、边界入参。
3. 联调测试：验证前后端、主服务与辅助服务、服务与数据层/缓存层的协作。
4. 回归测试：验证旧能力未被新变更破坏。
5. UI 验收测试：验证页面与设计稿、交互、权限态、状态页一致。
6. 性能测试：验证吞吐、响应、资源占用、热点场景。
7. 安全测试：验证登录、越权、输入、Token、敏感数据保护。

## 2. 提测前置条件
1. PR 已通过或至少完成自审。
2. Must 功能开发完成。
3. 自测通过并有记录。
4. 接口文档、数据库变更、配置变更已同步。
5. 测试环境可部署，可访问依赖服务。

## 2.1 TDD 介入规则
1. 默认开发范式不是“先实现、后补验证”，而是按工程现实执行务实分层 TDD。
2. 后端业务逻辑优先执行测试先行：
   - 涉及状态机、校验规则、查询过滤、DTO/VO 映射、服务层分支、回归 bug 修复时，先写失败测试或先写可执行失败断言，再进入实现。
3. Bug 修复默认先补失败复现：
   - 能自动化时先写失败测试。
   - 暂时无法自动化时，先记录最小复现步骤、输入数据与失败表现，再进入修复，并在同轮补回回归证据。
4. 前端按能力分层接入 TDD：
   - 纯逻辑代码，如工具函数、状态映射、数据转换、store 逻辑，应优先测试先行。
   - 页面级 UI、浏览器交互、复杂样式布局任务，在测试基线不足时，至少先写测试点 / 验收点，再进入实现。
5. 数据库迁移也必须先定义验证：
   - 提交迁移脚本前，先写清迁移前状态、预期迁移后状态、核对 SQL 或接口验证点。
6. 没有测试证据的功能不算完成：
   - 测试证据可由自动化测试、接口 smoke、迁移核对、页面验收中的适当组合构成，但后端业务逻辑优先追求自动化测试。
7. 禁止空转式 TDD：
   - 不得为了追求形式而补与业务无关的样板测试、脆弱快照测试或远高于收益的过度测试代码。

## 2.2 总线与分线 TDD
1. 总线 TDD：阶段级产物必须可验证，不能只输出不可判定的讨论结论。
2. 分线 TDD：进入实现 slice 前必须有失败测试、失败复现记录或明确的验收断言，至少保证“先定义失败，再进入实现”。
3. 需求变更、回滚、重构、修复 bug 时，必须先形成可验证的失败点或回归点，再推进修复。
4. 若阶段输出无法被测试或验证，说明该节点尚未完成，不得进入下一阶段。

## 3. 测试用例结构
```md
### 用例名称
- 用例编号：
- 功能模块：
- 前置条件：
- 测试数据：
- 操作步骤：
- 预期结果：
- 结果判定：
- 备注：
```

## 4. Bug 等级定义
1. `P0`：生产阻断、数据错误、严重安全问题。
2. `P1`：核心流程不可用或大面积影响主业务。
3. `P2`：重要功能异常但有绕行方案。
4. `P3`：一般体验问题、文案问题、低风险缺陷。

## 5. 发布前验收门槛
1. P0、P1 缺陷为 0。
2. P2 缺陷需评估并明确是否带病上线。
3. Must 功能全部通过。
4. 核心回归用例通过。
5. 部署脚本、配置、回滚预案齐备。

## 5.2 数据库 Migration 验证要求
1. 每个 Migration 脚本都必须有迁移前后验证点。
2. 至少验证：结构是否生效、索引/约束是否生效、历史数据是否兼容。
3. 涉及回填时，必须验证受影响行数、抽样数据和幂等重跑结果。
4. 发布前必须确认迁移脚本、应用代码、Mapper/Entity、接口文档版本一致。

## 5.1 Must 功能验收映射
1. 每个 Must 需求必须至少映射一个验收用例。
2. 每个核心页面必须映射至少一条正常流程与一条异常流程用例。
3. 权限、数据状态、接口异常、缓存失效等跨层风险点必须有对应验证项。

模板：
```md
| 需求编号 | 页面/模块 | 接口 | 测试用例 | 验收状态 |
| --- | --- | --- | --- | --- |
| REQ-USER-001 | 用户列表 | API-USER-001 | TC-USER-001 | 通过 |
| REQ-ORDER-003 | 订单审核 | API-ORDER-004 | TC-ORDER-006 | 待验证 |
```

## 6. 前后端联调流程
1. 后端先冻结核心接口契约。
2. 前端按 Mock 或 Swagger 开发。
3. 联调前确认环境、数据、权限账号。
4. 联调问题按接口、页面、数据三类归因并记录。
5. 修复完成后执行回归。

## 7. Mock 数据开发规范
1. Mock 字段必须符合真实字段类型与枚举约束。
2. Mock 数据需覆盖空值、边界值、长文本、异常状态。
3. Mock 只作为开发辅助，不得长期替代联调。

## 8. 测试环境数据管理规范
1. 测试账号、角色、组织结构应标准化。
2. 关键演示数据需可重复初始化。
3. 不得在测试环境长期存放真实敏感数据。
4. 需提供基础字典、配置、权限初始化脚本。

## 9. 常见测试范围清单
1. 登录、退出、Token 失效。
2. 列表查询、分页、筛选、排序。
3. 新增、编辑、删除、详情。
4. 导入、导出、批量操作。
5. 权限控制、按钮显示、数据权限。
6. 异常提示、空态、错误态、禁用态。
7. Migration 执行、回填验证、回滚演练。
8. 多租户隔离、跨租户越权、平台级/租户级权限差异。
9. 日志与审计记录是否完整。

## 10. 自测清单
1. 正常流程是否跑通。
2. 异常流程是否提示正确。
3. 输入边界是否校验。
4. 权限不足时是否阻断。
5. 刷新页面、重复提交、断网重试是否可控。

## 10.4 TDD 最小执行要求
1. 新增后端业务方法前，先判断是否属于“应先写失败测试”的范围；若属于，不得直接跳进实现。
2. 每次 bug 修复前，先确认是否已有失败测试或书面复现记录；没有则先补。
3. 每次页面级任务开始前，至少先写出本轮测试点、验收点或关键失败路径。
4. 每次数据库迁移前，至少先写出迁移前后核对点。
5. Review 时必须检查“测试是否先于实现介入”，而不是只看最终是否跑过构建。

## 10.5 测试后环境清理要求
1. 每轮构建、联调、smoke、真实浏览器预验收或临时验证完成后，必须显式判断本轮为测试而启动的实例是否仍有保留必要。
2. 对以下“仅为本轮测试/验证临时启动”的实例，默认应在验证完成后及时清理：
   - 临时后端服务进程
   - 临时前端 dev server 进程
   - 临时测试脚本进程
   - 临时 Docker 容器或 compose 服务
   - 临时端口转发或代理进程
3. 若实例仍需保留，必须具备明确理由，例如“用户马上继续人工验收”或“下一步连续联调仍依赖同一实例”。
4. 若没有明确理由，不得让历史测试实例长期占用端口、CPU、内存或容器资源。
5. 测试记录或交接文档中应写明：
   - 本轮临时启动了哪些实例
   - 哪些已清理
   - 哪些因明确理由暂未清理

## 10.1 冒烟测试清单
上线前至少执行以下最小冒烟：
1. 登录、退出、Token 续期或失效处理。
2. 菜单加载与核心页面跳转。
3. 一个核心列表查询。
4. 一条核心新增或编辑。
5. 一条核心审核或状态流转。
6. 一次关键导入导出或文件上传。
7. 一条无权限访问验证。
8. 一次健康检查与日志检查。

## 10.2 上线前最小验证路径
单人 + AI 项目上线前至少验证：
1. 从登录到核心业务目标完成的完整主路径。
2. 至少一个失败路径。
3. 至少一个权限受限路径。
4. 至少一次缓存相关动作，如登录态、字典刷新或重复提交控制。

## 10.3 用户介入前的真实浏览器预验收
当且仅当本轮改动或新增内容已经进入“确有必要要求用户实际登录系统进行验证测试”的阶段时，在要求用户介入之前，AI 必须先在真实浏览器中完成一轮自动化预验收。该预验收采用显式触发与分级规则，而不是仅凭主观判断。若项目已安装 Chrome DevTools MCP 或等价浏览器自动化能力，则仅在触发场景下优先使用。对于普通编码、局部修复、纯后端改动、文档改动、数据库预埋、接口 smoke、构建验证、日志排查等不要求用户实际登录验证的任务，明确不得调用真实浏览器预验收工作流，必须继续使用构建、接口、日志和本地 smoke 等原有验证手段。

执行要求：
1. 触发规则：
   - 新页面首次实现完成后，在交付用户人工复核前，必须先执行一轮真实浏览器自动预验收。
   - 同一页面后续若仅为小范围静态调整，且不涉及交互、接口、前后端联动或大规模布局改动，默认不重复触发真实浏览器自动预验收。
   - 若页面已进入人工审核并按审核意见回修，只有当本轮修正涉及交互、接口、路由、权限、状态、上传下载、弹窗抽屉或明显布局结构变化时，才重新触发真实浏览器自动预验收。
2. 分级规则：
   - `L1`：单个页面自动化验证默认级别，至少覆盖页面打开、核心模块可见、控制台报错检查和当前页最关键读写动作。
   - `L2`：关联页面组自动化验证默认级别，适用于如“列表 + 详情 + 新建/编辑”这类闭环页面组合，至少覆盖页面间跳转、关键动作链路和核心状态回放。
   - `L3`：完整大模块自动化验证默认级别，适用于模块级闭环完成时，至少覆盖真实登录、菜单入口、关键页面组串联、权限态、异常态和主要业务主路径。
   - 默认规则为：单个页面执行 `L1`，关联页面组执行 `L2`，完整大模块执行 `L3`。
   - 若 AI 认为需要升级或降级验证级别，必须先向用户说明必要性并获得同意。
3. 使用真实测试账号登录系统，并确认登录成功；若本轮为 `L1` 且不需要跨页链路或登录态复验，可在已登录上下文中完成当前页验证。
4. 按本轮待验证范围，访问所有关键页面元素、菜单入口、主操作按钮与核心交互链路。
5. 检查页面布局、表格、表单、弹窗、导航、状态标签、分页、空态、禁用态与接口响应是否正确。
6. 捕获任何错误或异常，包括接口失败、控制台报错、视觉错位、样式异常、点击无响应、弹窗打不开、表单不可提交等。
7. 发现问题后，优先直接执行定位、修复、回归；不得在明显缺陷仍未处理时把验证责任直接转交用户。
8. 每轮预验收结束后，必须输出执行简报，并同步写入项目级测试或交接文档，至少记录：
   - 本轮验证范围
   - 本轮验证级别与触发原因
   - 成功完成的真实操作
   - 发现的问题
   - 已完成的修复与回归结果
   - 剩余风险与待人工确认项
9. 预验收完成后，若临时浏览器联调环境、前端 dev server、后端测试实例或相关容器不再需要，必须及时清理。

例外规则：
1. 若当前会话缺少 Chrome DevTools MCP 或真实浏览器自动化入口，必须明确记录“工具不可用”这一阻塞，不得口头省略。
2. 工具不可用时，仍需尽最大可能完成接口、构建、本地环境、日志与手工可复现路径检查，并在文档中标记“真实浏览器预验收待补执行”。
3. 只有在完成上述阻塞说明后，才可请求用户手工进入系统验证。

## 10.6 节点级浏览器验证硬门禁
1. 对于包含浏览器可访问页面、前端交互或用户可见 UI 的模块，`Step 5e` 在交付用户复核前必须调用 Chrome DevTools MCP 完成一轮真实浏览器验证。
2. 对于存在浏览器可访问集成范围的项目，`Stage 6` 在交付用户做最终联调或集成核查前，必须调用 Chrome DevTools MCP 完成一轮真实浏览器验证。
3. `Step 5e` 的浏览器验证重点是当前模块页面、核心交互、状态流转、控制台错误和本轮修复回归。
4. `Stage 6` 的浏览器验证重点是跨模块主路径、关键页面组联动、登录态、异常态、权限态和集成回归。
5. 若 Chrome DevTools MCP 缺失或不可用，必须显式记录阻塞、执行最强降级验证路径，并把“真实浏览器验证待补执行”写入测试或交接文档后，才允许请求用户接手验证。

## 11. 测试报告最小内容
1. 版本与构建号。
2. 测试范围。
3. 环境信息。
4. 通过率与缺陷分布。
5. 残留风险。
6. 发布建议。

## 12. 禁止事项
1. 无自测直接提测。
2. 测试环境随意手工改数据且无记录。
3. 用“本地可以”为由跳过环境验证。
4. 只测正常流程不测异常和权限。
5. 只验证应用功能，不验证数据库迁移、审计、权限和隔离链路。

## 本文件使用建议
测试经理、研发、产品在提测、验收、发布评审阶段共同引用本文件形成统一质量口径。

## AI 调用建议
AI 在生成测试用例、Mock 数据、联调清单、发布前检查表时，必须按本文件的测试分层和验收门槛输出。凡涉及页面级联调、提测、验收或要求用户实际登录验证的任务，AI 必须先执行“真实浏览器预验收”；其中 `Step 5e` 与 `Stage 6` 对浏览器可访问 UI 范围属于强制触发节点。

## 后续可扩展点
可补充自动化测试分层策略、测试数据工厂规范、性能压测模板。

## 12. 企业级质量门禁要求
1. 测试不是发布前最后补一道工序，而是需求、设计、实现、发布全流程的质量约束。
2. 每个关键需求都应有对应验证路径，至少覆盖正常路径、异常路径、权限路径和边界路径中的必要集合。
3. 高风险变更如 migration、多租户、安全、权限、外部集成、导入导出，必须有专项验证，而不是混在通用 smoke 中一笔带过。
4. 测试结论必须可复核，至少包含范围、环境、命令或步骤、结果、异常和结论。

## 13. 测试资产治理
1. 测试用例、测试数据、Mock、回归清单、验收记录都属于正式工程资产，应随项目持续维护。
2. 测试数据必须可重复构造、可清理、可说明来源，不得长期依赖手工造数和个人本地数据库状态。
3. 自动化测试失败必须优先定位根因，不得长期通过跳过、降级断言或删除用例维持绿灯。
4. 手工验收必须记录到测试文档，不得只在聊天中口头说明“已验过”。

## 14. Review 附加清单
1. 是否覆盖了关键业务主路径和关键失败路径。
2. 是否显式验证了权限、审计、migration、租户隔离和配置差异。
3. 是否记录了环境、数据、命令、结果和剩余风险。
4. 是否存在测试结论无法复现或无法审计的问题。
