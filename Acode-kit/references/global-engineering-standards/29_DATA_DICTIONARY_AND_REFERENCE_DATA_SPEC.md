# 文件名
`29_DATA_DICTIONARY_AND_REFERENCE_DATA_SPEC.md`

# 文件定位
数据字典、标准参考字典与枚举治理专项规范。

# 适用范围
适用于中小型 B/S SaaS 项目在数据建模、数据库设计、接口设计、前端接入、缓存设计阶段涉及状态、类型、来源、角色、场景、标准编码体系的所有场景。

# 与其他文件的关系
本文件补充 `06_DATABASE_DESIGN_SPEC.md`、`20_DATA_MODELING_PLAYBOOK.md`、`05_API_DESIGN_SPEC.md`、`03_FRONTEND_ARCHITECTURE_SPEC.md` 中关于枚举和字典的通用要求，并为 `07_REDIS_CACHE_SPEC.md` 提供字典缓存设计输入。

# 编写目的
统一回答以下问题：
1. 什么时候应识别为“需要数据字典”。
2. 什么时候应识别为“标准参考字典”。
3. 哪些字段不应字典化。
4. 数据库、接口、前端、缓存应如何协同落地。

## 1. 核心术语
### 1.1 平台业务字典
- 由平台定义并维护的稳定编码值集合
- 常见于状态、类型、来源、角色、优先级、场景
- 可按业务需要决定是否允许租户扩展

### 1.2 标准参考字典
- 来源于国际标准、国家标准、行业标准或组织级统一主数据
- 例如国家码、币种码、行业分类标准
- 不允许租户任意新增或改写标准项

### 1.3 自由文本字段
- 不适合做字典
- 例如备注、摘要、说明、技术路线、风险描述、协议条款摘录

## 2. 何时必须识别为字典
满足任一条件时，必须进入字典识别流程：
1. 同一字段在多个页面反复作为下拉、筛选、Tag、统计维度出现。
2. 取值集合稳定且需要前后端统一展示口径。
3. 需要中英双语名称或展示属性。
4. 需要缓存、启停、排序、分组或权限控制。
5. 未来存在租户扩展、自定义项或平台初始化项。
6. 多张业务表复用同一编码体系。

## 3. 何时必须识别为标准参考字典
满足任一条件时，应优先归为标准参考字典，而不是普通业务字典：
1. 数据来源于国际、国家、行业标准。
2. 编码体系存在层级结构、标准版本或外部发布源。
3. 业务只引用标准，不拥有自由定义权。
4. 不允许租户自行新增、修改或删除标准项。

典型示例：
- ISO 国家/地区码
- ISO 货币码
- GICS 行业分类

## 4. 何时不应做字典
以下场景不应强行字典化：
1. 自由文本说明字段。
2. 强依赖上下文、取值无限扩展的描述字段。
3. 技术追踪字段，如 `trace_id`、`hash`、文件 Key。
4. 编号、流水号、协议号等业务唯一标识。

## 5. 默认分层模型
### 5.1 平台业务字典层
推荐表：
- `sys_dict_type`
- `sys_dict_item`

关键要求：
1. 类型与字典项分表管理。
2. 字典项必须包含稳定 `code`。
3. 必须支持中英名称。
4. 必须显式标记是否允许租户扩展。
5. 前端禁止硬编码展示名。

### 5.2 标准参考字典层
推荐表：
- `std_country`
- `std_currency`
- `std_gics_industry`

关键要求：
1. 采用独立标准参考表，不与普通业务字典混表。
2. 必须保留标准编码。
3. 如存在层级结构，必须保留 `parent_code` 或等价结构字段。
4. 如存在标准版本，必须保留版本字段。
5. 如业务表有性能或历史快照需要，可冗余名称字段，但真实口径仍以标准表为准。

## 6. 建模执行路径
在数据建模阶段，必须按以下顺序执行：
1. 列出所有状态、类型、来源、角色、场景、风险等级、行业标准、地区标准字段。
2. 对每个字段判断：
   - 是否需要字典
   - 是否属于标准参考字典
   - 是否允许租户扩展
   - 是否需要缓存
3. 输出项目级字典设计稿，至少包含：
   - 字典编码
   - 中文名称
   - 来源类型
   - 是否租户扩展
   - 涉及字段
4. 再进入数据库设计、接口设计和前端接入设计。

禁止跳过字典识别直接把状态和类型散落在页面、DTO、SQL 注释中。

## 7. 数据库设计要求
1. 业务表中仍保存稳定编码值，不保存中文展示名作为主值。
2. 标准参考字典建议业务表引用标准编码，必要时允许冗余名称快照。
3. 字典项必须支持 `status`、`sort_order`、`remark` 等最小治理字段。
4. 若字典允许租户扩展，必须显式带 `tenant_id` 或等价作用域字段。
5. 若字典不允许租户扩展，应固定为平台级或标准级数据源。

## 8. API 设计要求
1. 接口传输必须使用稳定编码值，不传中文展示名作为业务值。
2. 必须提供统一字典查询接口或枚举中心能力。
3. 标准参考字典应提供独立查询接口，不与普通业务字典混用同一返回结构时至少要显式标注层级。
4. 前端禁止自行猜测状态值或行业标准值。

## 9. 前端接入要求
1. 所有下拉、Tag、筛选条件统一从字典中心或字典接口获取。
2. 字典缓存与标准参考缓存分层管理。
3. 页面模板禁止散落硬编码枚举中文值。
4. 若页面需兼容历史脏数据，应在展示层做兼容映射，但不得回退为硬编码治理。

## 10. Redis 缓存要求
1. 高频字典必须评估缓存。
2. 建议区分：
   - `dict:*` 平台业务字典
   - `ref:*` 标准参考字典
3. 字典刷新必须有明确失效路径。
4. 标准参考字典可按版本维度缓存。

## 11. 设计稿输出要求
项目级字典设计稿至少应包含：
1. 文档信息
2. 设计目标
3. 分层原则
4. 当前项目字典总览
5. 推荐实现模型
6. 优先级划分
7. API 与 Redis 建议
8. 已确认落地规则

## 12. Review 检查点
1. 是否遗漏应进入字典治理的状态/类型字段。
2. 是否误把标准参考字典混入普通业务字典。
3. 是否把自由文本误字典化。
4. 前端是否仍有硬编码枚举展示名。
5. 是否定义了字典缓存和初始化路径。

## 本文件使用建议
在做数据建模、建表、接口和前端枚举接入前，先对照本文件完成“字典识别清单”。

## AI 调用建议
AI 在识别业务字段时，遇到状态、类型、来源、角色、场景、标准编码体系，不得直接视为普通字符串字段，必须先按本文件判断其是否需要进入字典治理与何种字典层级。

## 13. 企业级深化要求
1. 数据字典不仅是“下拉选项表”，而是口径统一、跨端一致、可审计、可缓存、可扩展的主数据治理入口。
2. 字典编码一旦对外或对内稳定暴露，应视为契约字段管理，不得随意重命名或复用旧编码承载新语义。
3. 任何影响筛选、统计、报表、审批流、权限判断的字典项变更，都必须评估历史数据兼容性和统计口径影响。
4. 标准参考字典必须记录来源、版本、生效范围和同步策略，不得只保留一份“来源不明”的静态数据。

## 14. 生命周期治理
1. 字典项应支持启用、停用、排序、分组、备注和生效状态管理。
2. 已被业务数据引用的字典项，不得直接物理删除；如需退役，应采用停用、替换映射或历史保留策略。
3. 涉及标准升级时，应先评估新增项、废弃项、编码变化和历史映射关系，再执行同步。
4. 若租户允许扩展字典，必须明确平台基础项、租户扩展项、租户私有项的作用域与冲突处理规则。

## 15. 前后端协同要求
1. 前端展示名、颜色、排序、禁用状态等展示属性，应来自字典查询结果或受控配置，不得到处硬编码。
2. API 应优先返回稳定编码，并按需要附带展示信息；前端不得把展示名反向当作业务判断条件。
3. 缓存字典时必须设计失效、刷新和版本策略，避免后台字典已更新而前端长期使用旧值。
4. 涉及导入导出时，应同时定义“编码列”和“展示列”的处理口径，避免 Excel 文本值反向污染系统编码。
