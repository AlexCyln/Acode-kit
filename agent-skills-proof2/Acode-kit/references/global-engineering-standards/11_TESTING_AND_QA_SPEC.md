# 文件名
`11_TESTING_AND_QA_SPEC.md`

# 文件定位
测试分层、提测门槛、联调与发布验收规范。

# 适用范围
适用于功能开发、重构、Bug 修复、发版验收、回归测试与质量门禁管理。

# 与其他文件的关系
本文件与 `10_CODE_REVIEW_SPEC.md`、`13_DEPLOYMENT_AND_DEVOPS_SPEC.md`、`14_CICD_SPEC.md` 联动，控制代码从开发到上线的质量门槛。

# 编写目的
定义从自测、提测、联调、回归到发布验收的统一质量流程，确保交付物可验证、可复现、可上线。

## 1. 测试类型边界
1. 单元测试：验证函数、类、服务方法的局部逻辑。
2. 接口测试：验证 API 契约、鉴权、状态码、返回结构、边界入参。
3. 联调测试：验证前后端、Java 与 Python、服务与 Redis/MySQL 的协作。
4. 回归测试：验证旧能力未被新变更破坏。
5. UI 验收测试：验证页面与设计稿、交互、权限态、状态页一致。
6. 性能测试：验证吞吐、响应、资源占用、热点场景。
7. 安全测试：验证登录、越权、输入、Token、敏感数据保护。

## 2. 提测前置条件
1. PR 已通过或至少完成自审。
2. Must 功能开发完成。
3. 自测通过并有记录。
4. 接口文档、数据库变更、配置变更已同步。
5. 测试环境可部署，可访问依赖服务。

## 2.1 TDD 介入规则
1. 默认开发范式不是“先实现、后补验证”，而是按工程现实执行务实分层 TDD。
2. 后端业务逻辑优先执行测试先行：
   - 涉及状态机、校验规则、查询过滤、DTO/VO 映射、服务层分支、回归 bug 修复时，先写失败测试或先写可执行失败断言，再进入实现。
3. Bug 修复默认先补失败复现：
   - 能自动化时先写失败测试。
   - 暂时无法自动化时，先记录最小复现步骤、输入数据与失败表现，再进入修复，并在同轮补回回归证据。
4. 前端按能力分层接入 TDD：
   - 纯逻辑代码，如工具函数、状态映射、数据转换、store 逻辑，应优先测试先行。
   - 页面级 UI、浏览器交互、复杂样式布局任务，在测试基线不足时，至少先写测试点 / 验收点，再进入实现。
5. 数据库迁移也必须先定义验证：
   - 提交迁移脚本前，先写清迁移前状态、预期迁移后状态、核对 SQL 或接口验证点。
6. 没有测试证据的功能不算完成：
   - 测试证据可由自动化测试、接口 smoke、迁移核对、页面验收中的适当组合构成，但后端业务逻辑优先追求自动化测试。
7. 禁止空转式 TDD：
   - 不得为了追求形式而补与业务无关的样板测试、脆弱快照测试或远高于收益的过度测试代码。

## 3. 测试用例结构
```md
### 用例名称
- 用例编号：
- 功能模块：
- 前置条件：
- 测试数据：
- 操作步骤：
- 预期结果：
- 结果判定：
- 备注：
```

## 4. Bug 等级定义
1. `P0`：生产阻断、数据错误、严重安全问题。
2. `P1`：核心流程不可用或大面积影响主业务。
3. `P2`：重要功能异常但有绕行方案。
4. `P3`：一般体验问题、文案问题、低风险缺陷。

## 5. 发布前验收门槛
1. P0、P1 缺陷为 0。
2. P2 缺陷需评估并明确是否带病上线。
3. Must 功能全部通过。
4. 核心回归用例通过。
5. 部署脚本、配置、回滚预案齐备。

## 5.1 Must 功能验收映射
1. 每个 Must 需求必须至少映射一个验收用例。
2. 每个核心页面必须映射至少一条正常流程与一条异常流程用例。
3. 权限、数据状态、接口异常、缓存失效等跨层风险点必须有对应验证项。

模板：
```md
| 需求编号 | 页面/模块 | 接口 | 测试用例 | 验收状态 |
| --- | --- | --- | --- | --- |
| REQ-USER-001 | 用户列表 | API-USER-001 | TC-USER-001 | 通过 |
| REQ-ORDER-003 | 订单审核 | API-ORDER-004 | TC-ORDER-006 | 待验证 |
```

## 6. 前后端联调流程
1. 后端先冻结核心接口契约。
2. 前端按 Mock 或 Swagger 开发。
3. 联调前确认环境、数据、权限账号。
4. 联调问题按接口、页面、数据三类归因并记录。
5. 修复完成后执行回归。

## 7. Mock 数据开发规范
1. Mock 字段必须符合真实字段类型与枚举约束。
2. Mock 数据需覆盖空值、边界值、长文本、异常状态。
3. Mock 只作为开发辅助，不得长期替代联调。

## 8. 测试环境数据管理规范
1. 测试账号、角色、组织结构应标准化。
2. 关键演示数据需可重复初始化。
3. 不得在测试环境长期存放真实敏感数据。
4. 需提供基础字典、配置、权限初始化脚本。

## 9. 常见测试范围清单
1. 登录、退出、Token 失效。
2. 列表查询、分页、筛选、排序。
3. 新增、编辑、删除、详情。
4. 导入、导出、批量操作。
5. 权限控制、按钮显示、数据权限。
6. 异常提示、空态、错误态、禁用态。

## 10. 自测清单
1. 正常流程是否跑通。
2. 异常流程是否提示正确。
3. 输入边界是否校验。
4. 权限不足时是否阻断。
5. 刷新页面、重复提交、断网重试是否可控。

## 10.4 TDD 最小执行要求
1. 新增后端业务方法前，先判断是否属于“应先写失败测试”的范围；若属于，不得直接跳进实现。
2. 每次 bug 修复前，先确认是否已有失败测试或书面复现记录；没有则先补。
3. 每次页面级任务开始前，至少先写出本轮测试点、验收点或关键失败路径。
4. 每次数据库迁移前，至少先写出迁移前后核对点。
5. Review 时必须检查“测试是否先于实现介入”，而不是只看最终是否跑过构建。

## 10.5 测试后环境清理要求
1. 每轮构建、联调、smoke、真实浏览器预验收或临时验证完成后，必须显式判断本轮为测试而启动的实例是否仍有保留必要。
2. 对以下“仅为本轮测试/验证临时启动”的实例，默认应在验证完成后及时清理：
   - 临时 Spring Boot 进程
   - 临时前端 dev server / Vite 进程
   - 临时测试脚本进程
   - 临时 Docker 容器或 compose 服务
   - 临时端口转发或代理进程
3. 若实例仍需保留，必须具备明确理由，例如“用户马上继续人工验收”或“下一步连续联调仍依赖同一实例”。
4. 若没有明确理由，不得让历史测试实例长期占用端口、CPU、内存或容器资源。
5. 测试记录或交接文档中应写明：
   - 本轮临时启动了哪些实例
   - 哪些已清理
   - 哪些因明确理由暂未清理

## 10.1 冒烟测试清单
上线前至少执行以下最小冒烟：
1. 登录、退出、Token 续期或失效处理。
2. 菜单加载与核心页面跳转。
3. 一个核心列表查询。
4. 一条核心新增或编辑。
5. 一条核心审核或状态流转。
6. 一次关键导入导出或文件上传。
7. 一条无权限访问验证。
8. 一次健康检查与日志检查。

## 10.2 上线前最小验证路径
单人 + AI 项目上线前至少验证：
1. 从登录到核心业务目标完成的完整主路径。
2. 至少一个失败路径。
3. 至少一个权限受限路径。
4. 至少一次缓存相关动作，如登录态、字典刷新或重复提交控制。

## 10.3 用户介入前的真实浏览器预验收
当且仅当本轮改动或新增内容已经进入“确有必要要求用户实际登录系统进行验证测试”的阶段时，在要求用户介入之前，AI 必须先在真实浏览器中完成一轮自动化预验收。若项目已安装 Chrome DevTools MCP 或等价浏览器自动化能力，则仅在该场景下优先使用。对于普通编码、局部修复、纯后端改动、文档改动、数据库预埋、接口 smoke、构建验证、日志排查等不要求用户实际登录验证的任务，明确不得调用真实浏览器预验收工作流，必须继续使用构建、接口、日志和本地 smoke 等原有验证手段。

执行要求：
1. 使用真实测试账号登录系统，并确认登录成功。
2. 按本轮待验证范围，访问所有关键页面元素、菜单入口、主操作按钮与核心交互链路。
3. 检查页面布局、表格、表单、弹窗、导航、状态标签、分页、空态、禁用态与接口响应是否正确。
4. 捕获任何错误或异常，包括接口失败、控制台报错、视觉错位、样式异常、点击无响应、弹窗打不开、表单不可提交等。
5. 发现问题后，优先直接执行定位、修复、回归；不得在明显缺陷仍未处理时把验证责任直接转交用户。
6. 每轮预验收结束后，必须输出执行简报，并同步写入项目级测试或交接文档，至少记录：
   - 本轮验证范围
   - 成功完成的真实操作
   - 发现的问题
   - 已完成的修复与回归结果
   - 剩余风险与待人工确认项
7. 预验收完成后，若临时浏览器联调环境、前端 dev server、后端测试实例或相关容器不再需要，必须及时清理。

例外规则：
1. 若当前会话缺少 Chrome DevTools MCP 或真实浏览器自动化入口，必须明确记录“工具不可用”这一阻塞，不得口头省略。
2. 工具不可用时，仍需尽最大可能完成接口、构建、本地环境、日志与手工可复现路径检查，并在文档中标记“真实浏览器预验收待补执行”。
3. 只有在完成上述阻塞说明后，才可请求用户手工进入系统验证。

## 11. 测试报告最小内容
1. 版本与构建号。
2. 测试范围。
3. 环境信息。
4. 通过率与缺陷分布。
5. 残留风险。
6. 发布建议。

## 12. 禁止事项
1. 无自测直接提测。
2. 测试环境随意手工改数据且无记录。
3. 用“本地可以”为由跳过环境验证。
4. 只测正常流程不测异常和权限。

## 本文件使用建议
测试经理、研发、产品在提测、验收、发布评审阶段共同引用本文件形成统一质量口径。

## AI 调用建议
AI 在生成测试用例、Mock 数据、联调清单、发布前检查表时，必须按本文件的测试分层和验收门槛输出。凡涉及页面级联调、提测、验收或要求用户实际登录验证的任务，AI 必须先执行“真实浏览器预验收”。
