# 文件名
`02_UI_UX_DESIGN_SPEC.md`

# 文件定位
Pencil 设计稿、后台系统 UI/UX 与设计交付规范。

# 适用范围
适用于后台管理系统、业务平台、数据平台等 B/S SaaS 场景的页面设计、组件设计与前端还原。

# 与其他文件的关系
本文件依赖 `01_PRODUCT_REQUIREMENTS_STANDARD.md` 的页面需求输入，并为 `03_FRONTEND_ARCHITECTURE_SPEC.md` 提供实现基准。

# 编写目的
把 Pencil 设计稿从视觉产物提升为工程交付基准，统一后台系统页面结构、状态设计、组件规范与实现还原口径。

## 1. Pencil 在流程中的角色
1. Pencil 设计稿是页面结构、视觉规范、交互状态的主要依据。
2. 设计稿不是纯视觉图，而是工程交付物，必须可拆成页面、区块、组件、状态。
3. 设计稿缺失状态页、弹窗、校验、空态时，不允许直接进入完整开发。
4. 前端实现需尊重设计稿，不得随意替换布局、文案层级、信息优先级。
5. 当项目进入具体主任务节点时，推荐先产出更小颗粒度的子模块 UI 审阅稿；子模块审阅通过后，再进入前端复刻/还原，而不是在大而全实现草案后直接编码。
6. UI 设计优先使用 Pencil 或等价设计稿工具进行承载；若项目中已存在 Pencil MCP，则应优先通过 Pencil MCP 进行页面设计与修改。

## 2. 页面层级结构规范
1. 一级页面：承载完整业务目标，如“用户管理”“订单审核”“数据看板”。
2. 二级区块：承载筛选区、列表区、详情区、操作区、统计区。
3. 三级组件：表单项、表格、卡片、Tabs、弹窗、抽屉、步骤条、图表等。
4. 页面必须能从导航结构映射到信息架构，避免无入口页面。

## 3. 信息架构规范
1. 一个页面只允许一个主要任务。
2. 次要任务应通过按钮、弹窗、抽屉或分步流程承载，不得抢占主任务。
3. 高优先级信息置顶，如状态、主键标识、关键指标、异常提醒。
4. 列表页重点展示筛选、统计、批量操作、结果集。
5. 详情页重点展示对象概览、业务分区、时间线、操作记录。

## 4. 设计交付清单
每个页面交付必须包含：
1. 页面名称与路由意图。
2. 正常态设计。
3. 加载态。
4. 空态。
5. 错误态。
6. 禁用态。
7. 权限态。
8. 关键弹窗、抽屉、确认框。
9. 表单校验提示。
10. 交互说明与数据字段说明。
11. 字段、按钮、状态、标签、标题的来源说明：来自页面审阅稿、后端 API、数据库字段或已确认字典。

## 5. 视觉基础规则
### 5.1 栅格与留白
1. 主内容区推荐 24 栅格思维，但在实际实现中以组件布局与容器宽度优先。
2. 页面级左右内边距建议 24px。
3. 区块间距建议 16px 至 24px。
4. 同类表单项、按钮组、表格工具栏间距应统一。

### 5.2 字体与字号
1. 默认使用系统可部署的中文友好字体方案，项目级可统一指定。
2. 标题层级建议为 20 / 18 / 16 / 14。
3. 正文建议 14。
4. 辅助信息建议 12。
5. 禁止同页面无规则混用字号。

### 5.3 颜色、圆角、阴影
1. 主色、成功色、警告色、危险色、信息色必须形成系统化色板。
2. 状态色必须可被程序化映射到 Tag、Alert、Button、Badge。
3. 圆角建议统一为 4 / 6 / 8 三档。
4. 阴影仅用于层级区分，不得泛滥使用。

## 6. 组件设计规范
### 6.1 按钮
1. 每个操作区只允许一个主按钮。
2. 危险操作必须使用危险色并要求二次确认。
3. 按钮文案使用动词短语，如“新建用户”“提交审核”“导出数据”。

### 6.2 表单
1. 必填项统一标识。
2. 标签、占位、校验、帮助说明必须齐全。
3. 长表单必须分组，必要时分步。
4. 日期、金额、枚举、附件、人员选择等必须使用明确控件。

### 6.3 表格
1. 表头应体现业务含义，不使用技术字段名。
2. 常用操作固定在右侧操作列。
3. 批量选择、排序、筛选、导出需在设计阶段声明。
4. 超长文本、状态、时间、金额应使用统一展示样式。

### 6.4 弹窗与抽屉
1. 用于轻量编辑、确认、辅助信息。
2. 复杂编辑优先页面或抽屉，不强行塞入小弹窗。
3. 弹窗必须包含关闭方式、提交按钮、取消按钮、失败提示。

## 7. 页面状态规范
1. 加载态：骨架屏、Spin 或占位块，不得让页面空白闪烁。
2. 空态：说明无数据原因，提供推荐操作。
3. 错误态：说明失败原因类别，提供重试或返回。
4. 禁用态：解释禁用原因，避免只变灰不说明。
5. 权限态：无权限时展示占位提示或跳转授权页面，不得仅隐藏造成误解。

## 8. 后台系统通用页面范式
1. 列表页：标题区 + 统计区 + 查询区 + 操作区 + 表格区 + 分页区。
2. 详情页：头部摘要 + 关键状态 + 基础信息 + 业务明细 + 操作记录。
3. 新增编辑页：分区表单 + 底部操作栏。
4. 审核页：对象信息 + 审核意见 + 审批动作 + 历史记录。
5. 数据看板页：筛选区 + 指标卡片 + 图表区 + 明细钻取入口。

## 9. 设计稿到 Vue 页面还原规范
1. 前端需按设计稿拆分为页面容器、业务组件、通用组件。
2. 优先用 Element Plus 组件承载标准控件，避免重复造轮子。
3. 对设计稿中与组件库存在差异的部分，应优先通过主题、样式扩展、组合封装实现。
4. 对不可直接还原的复杂动效或布局，需与设计确认取舍，不得私自简化关键交互。

## 10. 减少设计与实现偏差的方法
1. 设计稿必须标注间距、字号、颜色、状态、交互说明。
2. 前端开发前应进行页面拆解评审。
3. 复杂页面先做低成本骨架页确认结构，再补完整视觉。
4. 交付前至少完成一次设计走查，记录偏差项与处理结论。
5. 单人 + AI 项目在每个主任务节点中，默认应按 `子模块审阅稿 -> 用户确认 -> UI 设计 -> 前端复刻/还原 -> 后续实现` 的顺序推进。
6. 页面/模块设计前，必须先向用户确认本次页面的设计风格、交互与细节要求；如与项目全局设计要求冲突，先由用户裁决。
7. 设计稿中的模块、字段、按钮、状态、标签、标题等内容，必须严格遵循已确认的页面审阅稿以及真实 API / 数据库输出，不得私自增加或减少。
8. 设计稿经用户确认通过后，才允许进入前端复刻/还原。

## 10.1 Pencil 页面校验核对环节
1. 所有基于 Pencil 的页面设计，在“设计完成”与“进入前端复刻”之间，必须新增一次工具级页面校验核对环节。
2. 必须使用 Pencil 自带能力检查至少以下问题：
   - 页面边界与主内容区边距
   - 模块之间的外边距
   - 容器内部内边距
   - 文字、按钮、图标、标签的对齐
   - 文本是否超过容器边界、被截断、未正确换行
   - 模块或子元素是否存在重叠、遮挡、裁切、越界
   - 容器高度或宽度是否足以承载真实内容
3. 若校验结果中仍存在布局问题，不得将该设计稿标记为“已确认可复刻”。
4. 校验完成后，需在项目文档中记录：
   - 使用了哪些 Pencil 校验能力
   - 发现了哪些问题
   - 最终如何修复
   - 当前是否达到 `No layout problems` 或等价状态

## 10.2 本轮沉淀的通用防回归问题
后续页面设计中，默认必须主动排查以下高频问题：
1. 顶部工具栏中的按钮、图标、文案因为容器宽度不足而被裁切。
2. 左侧导航中的分组、二级菜单因为容器高度不足而被裁切。
3. 面包屑、标题区、Hero 区等页面关键层级顺序错误，导致信息架构混乱。
4. 卡片或摘要容器高度按“预估文案”设置，真实文本进入后发生裁切。
5. 长文本未按固定宽度换行，导致内容超出卡片边界。
6. Hero 区、摘要区、详情卡等存在字段遗漏，尤其是状态、主键、关键指标、权利主体等高优先级信息。
7. 同一页面中同类卡片、按钮、字段块的内边距、间距和文字对齐不一致。

## 10.3 对应处理方式
1. 对所有关键容器进行真实内容承载校验，而不是按视觉预估定高。
2. 对长文本、编码、英文标题、说明文字等高风险字段，显式设置宽度与换行策略。
3. 对 Hero、摘要区、导航区、表格工具条等页面关键区域，逐项对照页面审阅稿核对字段完整性。
4. 每轮 UI 调整后，至少再执行一次整页工具级校验，而不是只看局部截图。
5. 页面在进入前端复刻前，必须同时满足：
   - 内容与审阅稿一致
   - 内容与 API / 数据库字段一致
   - 工具级布局校验通过
   - 用户视觉确认通过

## 11. 设计评审检查清单
1. 页面是否覆盖 PRD 中的所有 Must 功能。
2. 状态页是否齐全。
3. 字段与术语是否与需求一致。
4. 表单校验、错误提示、权限态是否明确。
5. 是否便于前端工程化拆分与复用。

## 本文件使用建议
设计师、前端与产品在页面冻结前共同使用本文件进行设计评审与实现拆解。

## AI 调用建议
AI 在生成页面结构、组件树、前端样式说明前，必须先读取本文件与 Pencil 设计稿，按本文件的页面范式和状态规则输出。
